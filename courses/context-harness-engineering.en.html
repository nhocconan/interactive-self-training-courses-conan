Chapter 01 · Part I · Foundations

One stack, three layers: Prompt · Context · Harness

Prompt engineering is what you say to the model. Context engineering is what the model sees. Harness engineering is the world the model lives in. Confusing these three layers is the most common reason AI features break when they reach production.

Lesson 1.1

One stack, three layers: Prompt · Context · Harness

L1 · Foundations
After this lesson you can
  • Cleanly tell the three layers apart — prompt, context, harness — and know which layer each failure belongs to.
  • Apply the right diagnostic order when an AI feature "works in dev, breaks in prod".
  • Ask the right three review questions for an AI-feature PR.

Start with a familiar scene: if the team doesn't share a mental model of these three layers, every AI bug turns into an argument about the prompt. An engineer rewrites the system prompt while the real problem is a tool returning 30k tokens of junk, or a retrieval pipeline pulling the wrong documents, or a loop that never terminates. The place that needs fixing is in a different layer from the place being fixed.

Definitions by scope

Prompt engineering — the message

The craft of writing a single message well: assigning a role, stating constraints, specifying the output format, in-context examples, tuning tone. This is sentence-level work. It matters, but it has a ceiling: the best prompt in the world still fails if you feed it the wrong context or run it inside a broken loop.

Context engineering — the window

The context layer curates everything the model sees in a single inference pass: system prompt, tool definitions, retrieved documents, conversation history, in-context examples, attached files, scratchpad notes. The job is to find the smallest set of tokens with the most signal that maximizes the chance of a correct result. The constraint is real: even with a 200k+ token window, the signal-to-noise ratio decides quality more than total capacity.

Harness engineering — the runtime

The whole staging around the model: tool setup, the agent loop, memory persistence, error handling, retries, output validation, spawning subagents, cost ceilings, observability. A good harness lets a mediocre prompt succeed; a bad harness makes a perfect prompt fail in production. The harness is where an AI feature becomes an AI product.

A web analogy

Prompt = HTML. Context = the entire HTTP request/response (headers, cookies, body, query params, server-rendered data). Harness = framework + infrastructure (Next.js, job queues, Redis, error boundaries, deployment). You can write perfect HTML in a broken framework — the page still won't render.

Diagnostic order

When an AI feature works in dev but breaks in prod, check in this priority order to rule out the big causes first, instead of defaulting to the prompt as the culprit.

#LayerSymptomWhat to inspect
1harnessRandom errors, cost spikes, infinite loops, state leaking between users, silent truncation, missing retries.Log the full message history. Loop counts. Tool error rate. Token-budget metrics. Per-user state isolation.
2contextFabricated facts, "ignoring" instructions, citing the wrong document, output drifting over a long session.What was actually retrieved? The message history at the moment of failure? Which tools were defined? Do the examples represent the real distribution?
3promptA genuine prompt bug: vague instructions, contradictory rules, unspecified output format, off tone.Read the prompt out loud. Would a junior engineer reading it once, with the same context, do what you meant? If not, fix the wording.
The "tweak the prompt first" trap

The "fix the prompt first" reflex exists because the prompt is visible and immediately editable. Harness and context bugs demand reading logs, instrumenting code, and reasoning about state — unpleasant work. Most teams optimize for comfort, and stay in the broken state longer.

The team-level consequence

When reviewing an AI-feature PR, the reviewer's first question should not be "is the prompt OK?" — it should be:

  1. What does the harness do when something goes wrong?

    When a tool fails / the model is overloaded / the output is malformed / the loop runs forever / cost spikes — what happens?

  2. What is actually in the context window?

    For a typical input and a worst-case input: how many tokens? In what order?

  3. Is the prompt clear?

    Does it match the examples and tools provided?

If the first two don't have satisfactory answers, the third one doesn't matter yet.

Exercise · Audit one of your existing AI features

Pick any AI feature in your system — even a simple one. Write down:

  1. The full harness: where the call originates, retry policy, error handling, cost ceilings, what happens on partial failure, where logs go.
  2. The full context for a typical call: system-prompt size, number of registered tools, history-accumulation policy, retrieved-data size, total tokens, the position of each part.
  3. The prompt: read it as if you were a new dev seeing it for the first time. Note where it is vague.

Now rank the three layers by "amount of careful engineering invested". For most teams it is inverted — prompt-heavy, harness-light. Make that visible to the whole team.

Takeaway

Prompt, context, harness are three distinct layers. When AI breaks in prod, diagnose bottom-up — harness → context → prompt; don't tweak the prompt first just because it's easy to change. The harness is where an AI feature becomes a product.

Chapter 02 · Part I · Foundations

Context theory · Attention, Context rot, Anatomy

A bigger context window gives you more room; it doesn't guarantee the model uses every part equally well. The surest approach is to measure on real tasks and keep the signal-to-noise ratio high.

Lesson 2.1

Attention budget & positional bias

L1 · Foundations
After this lesson you can
  • Explain why adding context can reduce quality (context rot & the attention budget).
  • Recognize the "lost in the middle" positional bias and place key information at the start/end.
  • Read the full anatomy of a context window: which segments are stable, which ones bloat.

You hand the AI a folder of 80 documents to "give it all the facts". The answer then skips the one appendix that holds the important exception. The problem isn't a lack of room; the model has to find the right signal among many things competing for attention.

The attention budget

The attention budget is a mental model to remember that adding tokens always has a trade-off: it adds evidence, but also adds things the model must distinguish and cost to process. Research calls the phenomenon of quality becoming less reliable as input grows longer context rot. The magnitude varies by model and task; so measure — don't apply one universal threshold to every system.

More is only better when it adds the right signal

If 5 RAG results already contain enough evidence, adding 45 near-duplicate or off-topic passages raises cost and can lead the model to pick the wrong one. Try several top-k values on the same eval set instead of choosing by feel.

Positional bias — "lost in the middle"

In the Lost in the Middle experiment, many models do better when the information they need is at the start or end of the input and worse when it's in the middle. This is a design warning, not a law for every model: keep key instructions prominent, reduce noise, and test on your actual workload.

Full anatomy — what actually fills the window

Every model call assembles a single linear sequence. Understanding how many tokens each segment costs — and how it interacts with the cache — is the foundation for every other context-engineering technique.

The 1M-token era · what changes, what doesn't

As of Jul 2026, GPT-5.6, Claude Sonnet 5 and Opus 4.8 all offer a context option around 1M tokens. That capacity is useful for long documents and tasks, but doesn't prove that any given monorepo or PDF will be understood in full. The context window is capacity; effective context is the part the model uses correctly on your eval.

The anatomy above is what you have; the next question is how to order it. A good default layout follows the two forces you've just learned: order by decreasing stability (so the top stays put in the prefix cache) and separate state from history (so you don't feed an old debate into a new turn). Paste this checklist as a frame when building a new feature.

checklist · default context layout
1. SYSTEM     · role, invariant rules, output schema       (stable · cache)
2. TOOLS      · definitions of tools actually needed        (stable · cache)
3. EXAMPLES   · representative few-shot, incl. refusal case (stable · cache)
4. KNOWLEDGE  · short project docs / map, pointing to source (stable per session)
5. STATE      · plan / scratchpad / progress — a file, not history
6. HISTORY    · compacted, keep decisions, drop the debate
7. TASK       · current request + success criteria         (dynamic · last)

When reviewing any call, ask three questions: which part that should be in the prefix cache is dynamic? which state is living in history instead of in a file? are the success criteria in the high-attention zone (the end)? Measure: the amount of cache_read tokens and the eval pass rate when you change the layout — don't trust the feeling of "that's tidier".

Takeaway

A long context gives you more capacity, not more guarantees. Keep the right evidence, place key instructions clearly, and measure on real tasks; don't treat 1M tokens as a license to stuff.

Lesson 2.2

Token economics · caching & hidden taxes

L1 · Foundations
After this lesson you can
  • Lay out the context window by stable prefix and measure how much of the cost is actually cached.
  • Name the biggest "hidden tax" in a production agent system (accumulated tool results) and how to handle it.
  • Audit the token cost of the tool registry and know when to use ToolSearch.

Lesson 2.1 gave you why stuffing context lowers quality. This lesson is the other side of the same coin: tokens are also money and latency. Understanding token economics turns context discipline from a "should do" into a measurable cost lever.

Token economics — what actually costs

Input, cached input and output have different unit prices per model. Because price lists change fast, don't memorize a ratio; read the usage of a real request and compute against the model you're running. The part usually missed is the cost that repeats through every agent loop:

  • Latency tax — long input usually raises time-to-first-token; in an agent loop, that increase repeats on every iteration.
  • Quality tax — every irrelevant token dilutes attention on the relevant ones (that's context rot).
  • Cache-miss tax — see just below.
Prompt caching — the lever you must understand

Both Anthropic and OpenAI offer prefix-based prompt caching. For Claude (as of Jul 2026), a cache read is billed at 0.1× base input, but a 5-minute cache write is 1.25× and a cache hit requires an identical prefix. Take 100k stable tokens: the first call pays a write cost equivalent to 125k input; a later call, if it hits, makes that part equivalent to 10k. Put the stable part first, the dynamic part last; a timestamp at the top can wipe out the hit for everything behind it.

Cache-friendly order (by decreasing stability)

1) Most stable first: system prompt, tool definitions, large reference documents. 2) Stable per session: project context, user persona. 3) Variable suffix: conversation history, current message. Anything dynamic (timestamp, ID) must sit in the suffix.

typescript · vercel ai sdk · cache-friendly order
const result = await generateText({
  model: anthropic('claude-sonnet-5'),

  // 1. SYSTEM — most stable. Cache aggressively.
  system: [{
    type: 'text',
    text: SYSTEM_PROMPT_STABLE,
    providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } }
  }],

  // 2. TOOLS — also stable. The registry rarely changes within a session.
  tools: TOOL_REGISTRY,

  // 3. MESSAGES — variable suffix. A cache hit stops here.
  messages: [ ...conversationHistory, { role: 'user', content: userMessage } ],
});

// Inspect cache stats — the dashboard should track cache_creation & cache_read.
console.log(result.providerMetadata?.anthropic);

Hidden tax — tool results left behind in history

Every tool result can become part of the history for later turns, until the harness compacts it, evicts it, or replaces it with a compact artifact. A single read of a 5,000-line log can be larger than the task prompt itself; repeated many times, old tool output ends up occupying most of the context. Measure tokens from the trace instead of guessing by line count.

When a long session starts re-mentioning old files

Don't conclude the model is "tired". Open the trace and check whether the history is buried under old tool output. If it is, use the tool's compaction/reset, write important state out to a structured artifact, or start a new session with a compact brief.

The tool registry — a standing fixed cost

Every tool you register injects its name + description + JSON schema into the context on every call, even when no tool is used. A quietly bloating registry steals tokens from every conversation.

Bloated tool def · ~200 tokens 
{
  name: "search_database",
  description: "This tool lets you search
   the database. Very useful when you need
   to find information. Use it when you want
   to look something up. The database holds
   many records. Remember to use correct
   SQL syntax and our system conventions...",
  parameters: { /* loose schema */ }
}
  • Vague description = wrong calls. Long description = a standing token tax. Loose schema = invalid calls slip through.
Tight tool def · ~60 tokens 
{
  name: "search_customers",
  description: "Find a customer by email,
   phone or order_id. Returns up to 10.",
  parameters: {
    type: "object",
    properties: {
      field: { enum: ["email","phone","order_id"] },
      value: { type: "string", minLength: 2 }
    },
    required: ["field","value"]
  }
}
  • Clear purpose. Concise description. Tight schema. Fewer wrong calls, less context tax.
There is no single correct MCP threshold for every system

Measure the total tokens of your tool definitions and the rate of wrongly-called tools on your eval. When the registry starts crowding out the task context or making the model choose wrong, turn off out-of-scope tools and use Tool Search / a load-on-demand registry to load schemas only when needed. The goal is to reduce excess choices, not to chase a magic 20k number.

📎 Illustrative example · an assistant across many stores

An assistant receives, all at once, brand rules, KPI definitions and the taxonomy of every store. A timestamp still sits at the top of the system block. Easily-measured symptoms: frequent cache misses and answers that pull the wrong taxonomy.

Three changes to A/B test: (1) move the current date into a tool or the variable suffix; (2) load only the taxonomy of the store being handled; (3) keep the shared instructions file as a short map, and fetch long documents on demand. Then compare cache-read tokens, latency and taxonomy-mismatch errors on the same eval set. This is an illustrative example; the improvement figures must come from a real system's traces.

The full equation

"Which context configuration is most likely to produce the desired behavior, knowing that every token competes for a finite amount of attention, position matters, and history accumulates?"

This one question reframes most day-to-day decisions: which tools to register, what goes in CLAUDE.md versus a skill, when to compact, when to spawn a subagent, when to write state out. The rest of the course is technique for answering it.

Chapter 03 · Part I · Foundations

Context Engineering patterns

Six recurring problems, six battle-tested patterns. This is Anthropic's playbook plus a few additions from production. Internalize these six and you can design context for any AI feature. And one corner that's often forgotten: context isn't just a window, it has a lifecycle.

Lesson 3.1

The six Context Engineering patterns

L2 · Working knowledge
After this lesson you can
  • Pick the right pattern for each context problem (retrieval, history bloat, long-term state, junk output).
  • Convert a "pre-load" feature to just-in-time and measure how tokens, latency and quality change.
  • Recognize & break context poisoning in long loops.

Chapter 2 gave you the physics of context. Chapter 3 gives you the technique. Six patterns, each treating one specific problem.

Pattern 1 · Just-in-time retrieval

The most important shift in agent design over the past year. Old thinking: pre-load everything the agent might need. New thinking: give the agent the tools to fetch what it needs, when it needs it.

Ahead-of-time loading 
// Load 50 customer records
// + 200 product specs
// + 30 days of orders
// into context "to be safe"

// Costs 80k tokens BEFORE
// the model reads the question.
// Most of it irrelevant.
  • Wasteful, slow, quality-dragging. The model spends its attention budget on user data it never asked about.
Just-in-time 
// Provide tools:
//   get_customer(id)
//   search_customers(q)
//   get_product(sku)
//   list_orders(filters)

// The model chooses which to call
// based on the real question, then
// calls more if the evidence isn't enough.
  • The agent explores the environment instead of receiving a pre-digested brief.
Measure with two runs

Run the same set of questions through the pre-loaded version and the fetch-on-demand version. Compare input tokens, latency, accuracy and tool-call count. Just-in-time only wins when it reduces excess data while still fetching enough evidence; if the search tool is poor, it can be cheaper but answer worse.

Pattern 2 · Compaction (recursive summarization)

When history bloats past the point of usefulness, summarize the old turns into one condensed turn and replace the original. Claude Code's /compact does exactly this; Codex has an equivalent session compaction. You should build it into your own agent system too.

Compaction rules:

  • Pin the stable part while it's still correct. Keep the system + original goal in a visible zone, keep recent turns verbatim, and summarize the middle. If the goal changes, update the source of truth instead of pinning stale instructions.
  • Summarize tool output aggressively. A 5k tool result becomes [abridged file contents · 1240 lines · src/foo.ts]. Keep the reference, drop the noise.
  • Keep decisions, drop the debate. "Chose Drizzle, rejected Prisma over bundle size" — keep. The 2,000 tokens of argument that led there — drop.
  • Compaction is itself a model call — it costs tokens and can drop detail. Trigger it from the task's eval, not from a window-percentage shared across every system.

Pattern 3 · Structured notes (working memory)

Let the agent maintain an external notes file (scratchpad, plan, todo) that lives outside the context window. The agent reads/writes this file as it works. Why it works: the file is the state of the work. The conversation can be compacted aggressively because the durable knowledge lives on disk; a new session resumes by reading the notes file.

  • Claude Code's plan.md/tasks.md pattern — the agent writes a multi-step plan to markdown, works through each step, ticks completion, leaves notes for the next step.
  • Codex CLI's working files — the same pattern, usually in a dedicated workspace directory.
  • Roll your own: a dedicated "context tool" — for a production agent, provide read_notes(key)/write_notes(key,value) tools backed by Redis or a DB.
The file is the agent's external memory

Once this clicks, you stop trying to stuff everything into the context and start treating the conversation as a resettable desktop.

Pattern 4 · Subagent isolation

Spawn a fresh agent context for a focused task. The parent only receives the final artifact/summary, so the child's detailed exploration doesn't spill into the main context. Anthropic recommends this pattern for long-running tasks; it's close to how a lead receives a report instead of reading every action of each IC.

Pattern 5 · Structured output (the safety net)

Free-form text output can drift off format; in production, just a few parse failures are enough to break a pipeline. Enforce structure at the API layer. In priority order: (1) a tool-call as output (define a "submit the answer" tool, the API validates the schema); (2) an SDK helper (generateObject + Zod, or response_format: json_schema); (3) JSON mode + validate + retry.

typescript · vercel ai sdk · structured output
import { generateObject } from 'ai';
import { z } from 'zod';

const PostAnalysis = z.object({
  sentiment: z.enum(['positive','negative','neutral','mixed']),
  topics: z.array(z.string()).max(5),
  brand_mentions: z.array(z.object({
    brand: z.string(),
    context: z.enum(['praise','complaint','neutral','comparison'])
  })),
  language: z.enum(['vi','en','mixed'])
});

const { object } = await generateObject({
  model: anthropic('claude-sonnet-5'),
  schema: PostAnalysis,
  prompt: `Analyze this post: "${postText}"`
});
// `object` is type-safe & schema-validated. No JSON.parse,
// no try/catch on junk output. Wrong field → the SDK retries with the schema error.
Non-negotiable for production

Always validate. Always type. Never JSON.parse raw output and pray. Code that does is a future incident.

Pattern 6 · Few-shot via in-context examples

Instructions tell the model what to do; examples show it an answer that meets the bar. Examples are usually helpful for format, edge cases and tone, but bad examples can drag quality down too. How to do it: pick diverse examples, including a refuse/uncertain case; run the eval when you add or remove examples; and keep the stable part in the prefix to reuse the cache.

Anti-pattern · Context poisoning

One wrong statement sitting in the history can become a premise for later turns. The agent keeps reasoning from the bad state, especially when the harness doesn't verify intermediate output. This is context poisoning: an old error reused as fact.

How to prevent it

(1) Validate intermediate output before it enters the history — if the agent says it deployed, verify before keeping that claim in the context. (2) When you find an error, rewrite or delete the bad turn — don't just append "oops, that was wrong". (3) For long loops, periodically reset to a clean state; subagents help isolate exploration from the main context.

Exercise · Replace pre-load with just-in-time
  1. Find a feature that pre-loads data into the prompt (typically: customer info, product catalog, recent orders).
  2. Define 2–3 focused tools (get_customer, search_products…) instead of loading in bulk. Remove the pre-load, update the system prompt to mention the tools.
  3. Compare on a representative query set: tokens, latency, accuracy and tool-call count. Only switch when the new version meets a pre-defined quality bar.
Takeaway

Don't stuff — give tools to fetch on demand (JIT), compact the middle, write state out to a file, isolate heavy work into a subagent, and enforce structured output. These six patterns are the context toolkit for every AI feature.

Lesson 3.2

Context lifecycle & governance

L2 · Working knowledge
After this lesson you can
  • See context as a four-stage lifecycle — ingest → use → consolidate → forget — not just a window.
  • Attach source + authority + date (provenance) to each remembered item, and know what beats what in a conflict.
  • Build a write gate and a minimal forgetting mechanism so long-term memory doesn't poison itself.

You write into CLAUDE.md, into the memory store, into the agent's long-term store. Three months later you open it: the file has tripled in size, one line still says "the project uses library X" when the team switched to Y long ago, and the agent confidently cites a "decision" that was actually reversed at the last meeting. No one did anything wrong on purpose — the problem is these memory channels only have a write direction, no way out.

Anti-pattern · the memory that never forgets

When everything written stays forever and no one re-checks freshness, you get a memory that never forgets: context poisoning (Chapter 3) made permanent. In a loop, one wrong statement poisons a few turns; in long-term memory, one expired fact poisons every future session until someone deletes it by hand. The cure isn't remembering less — it's treating context as having a lifecycle and designing the way out.

Context has a lifecycle, not just a window

Every piece of context passes through four stages. You learned patterns for the first three in Chapters 2–3; the fourth — forgetting — is the missing piece in most systems.

Provenance & authority — who may write, what beats what

Not every piece of context carries the same weight. Each item should carry its source (who created it), its authority (what it may override) and its expiry (fresh until when). Without those three, a line the model made up sits on par with a system rule.

SourceExampleAuthorityExpiry
System rulessystem prompt, CLAUDE.mdHighest — only a human edits itUntil a human changes it
Project knowledgerepo map, ADROverridable by a newer ADRBy write date; a newer ADR wins
Session notesscratchpad, plan.mdOnly within the current taskDies with the task
Tool / web resultsa web page, RAG outputLowest — it's data, not instructionsDistrust by default

A web page saying "ignore the rules above" is input to distrust, not a command. The arbitration principle: on conflict, the source with higher authority and more recent wins. That's why every long-term memory item should have a date-stamp — without a date you can't tell freshness, and "more recent" becomes a guess.

Write gate — the check before a write

A fact only enters long-term memory once it has passed verification — by a verifier (test/eval/schema) or by a human. Without a write gate, one misunderstanding by the agent means every later session inherits the error, and it reinforces itself each time it's cited again. This is exactly the durable version of context poisoning: instead of dying with the session, the error is stamped in permanently.

Forgetting is a feature, not an incident

A minimal forgetting mechanism: each item carries a date + source; periodically audit two questions — "is this still true?" and "when was it last used?"; and for a wrong item, delete it — don't append a correction line (two contradictory facts coexisting is worse). Measure: the average age of memory items, the share referenced in the last 30 days, and how often the agent cites an expired fact on the eval. Good memory shrinks over time, it doesn't bloat forever.

Exercise · Audit your long-term memory
  1. Open the CLAUDE.md / AGENTS.md / memory store you use. For each item, write down three answers: is it still true? what source did it come from? when was it last actually useful?
  2. Act on the results: delete dead items, date-stamp the ones you keep, and move "work in progress" items to a task file (the correct working-memory layer, Chapter 4) instead of leaving them in long-term memory.
  3. Measure before/after: the token size of the loaded-per-session part. If it doesn't shrink, you're hoarding, not governing.
Takeaway

Context isn't just a window — it has a lifecycle: ingest → use → consolidate → forget. Write through a write gate, remember with a source + date, and treat forgetting as a feature you design, not an incident that happens to you.

Pillar 1 · A tool is API design

The consumer of your tool is a non-deterministic agent that reads the description once and then decides what to call. Apply API design principles, but raise the stakes:

  • Few tools, each with a clear single purpose. search_customers_by_email beats universal_search. The model picks a tool by name + description; ambiguity at this layer can't be recovered downstream.
  • Fewer powerful tools beat many narrow ones. Don't mirror a REST API into 30 tools; provide 3–5 powerful "gateway" tools with rich parameters. Composition via parameters beats proliferation.
  • Idempotent when possible. The model retries. If send_email isn't idempotent, the customer gets three emails.
  • Bounded output. A list_files that returns 10,000 items just blew up the context. Always paginate/truncate/summarize. A tool that returns a 30k-token blob is a tool designed to hurt its caller.
  • Rich, actionable errors. Error: 500 is useless. Error: customer_id 'cus_abc' not found. Did you mean to search by email? teaches the model how to recover. Treat error messages as part of the tool's API contract.
typescript · production-grade tool
const searchPosts = tool({
  // CLEAR description: say when to use it, when NOT to.
  description: 'Search posts by keyword. Returns up to 20 most recent results with ' +
    'engagement metrics. Use for content/sentiment questions, do NOT use for ' +
    'total counts (use count_posts).',
  // TIGHT schema: limits + enums block bad calls.
  inputSchema: z.object({
    keyword: z.string().min(2).max(100),
    days_back: z.number().int().min(1).max(90).default(7),
  }),
  execute: async ({ keyword, days_back }, ctx) => {
    try {
      const rows = await db.searchPosts({
        keyword, days_back,
        tenantId: ctx.tenantId   // auth from the HARNESS, never from the model
      });
      if (rows.length === 0)
        return { status:'empty', message:`No posts for "${keyword}". Try a broader keyword.` };
      // BOUNDED output: max 20, key fields only, truncated text.
      return { status:'ok', count: rows.length,
        posts: rows.slice(0,20).map(p => ({ id:p.id, author:p.author, excerpt:p.text.slice(0,200) })) };
    } catch (err) {
      // RICH error: the model can act, not just fail.
      return { status:'error', message:`Search failed: ${err.message}. Try a simpler keyword.` };
    }
  }
});

Pillar 2 · The agent loop

The agent loop is conceptually very simple: call the model → if there's a tool_use, run the tool, append the result → call again → repeat until the response is text-only or the budget is hit. The complexity is in the BOUNDS.

typescript · production loop with all 4 bounds
async function runAgent(messages, tools, opts) {
  let iter = 0, totalCost = 0;
  const seen = new Set();                      // loop detection
  const MAX_ITER = opts.maxIter ?? 25;
  const MAX_COST = opts.maxCost ?? 2.0;
  const TOKEN_BUDGET = opts.tokenBudget ?? 150_000;

  while (iter++ < MAX_ITER) {
    if (estimateTokens(messages) > TOKEN_BUDGET)   // BOUND 1 · tokens → compact
      messages = await compact(messages);
    if (totalCost > MAX_COST)                        // BOUND 2 · cost ceiling → kill
      throw new CostExceededError(totalCost);

    const resp = await model({ messages, tools });
    totalCost += resp.usage.totalCost;
    messages.push(resp.message);
    if (!resp.toolCalls?.length) return { text: resp.text, totalCost, iter };  // done

    for (const tc of resp.toolCalls) {               // BOUND 3 · repeat-call guard
      const key = `${tc.toolName}:${JSON.stringify(tc.args)}`;
      if (seen.has(key)) { pushToolError(messages, tc, 'Duplicate call. Try a different approach.'); continue; }
      seen.add(key);
    }
    const results = await Promise.all(resp.toolCalls.map(tc =>
      withTimeout(executeTool(tc, tools), 30_000).catch(err => ({ id: tc.id, error: err.message }))));
    messages.push({ role:'tool', content: results });
  }
  throw new Error(`Agent exceeded ${MAX_ITER} iterations. Most likely stuck.`);  // BOUND 4 · max iter
}
Three loop types that cause incidents

(1) Infinite retry — the same tool fails identically over and over → duplicate-call detection. (2) Circular reasoning — the model flips between two states → cap the iterations. (3) Over-exploration — the model "double-checks" forever → the cost ceiling forces a stop. Set bounds and log enough messages, tool calls, errors, tokens and timing to reconstruct the agent's path.

Anti-pattern: "trust the model with permissions"

Pillar 5 · Observability — log the path you need to reconstruct

AI in production without logging is impossible to debug. The usable minimum: every model call (full input with PII masked, output, model name, latency, cache hits, tokens in/out, cost); every tool call (name, PII-masked args, result size, latency, errors); every loop (iteration count, cost, time, final state); per-user metrics (cost/user/day, p50/p95/p99 latency, success rate). Tools: Langfuse, Helicone, Braintrust, LangSmith. If you can't reconstruct exactly what happened in a failed run from the logs alone, your observability is broken.

New model, re-check the old scaffolding

The harness is code, so it too can become obsolete. After every model upgrade, run the same eval with each scaffold layer toggled on/off: long planning prompts, self-critique steps, subagents, retries, compaction. Keep any layer that improves quality or safety enough to justify its cost; drop layers that now only add latency. This is an ablation test — remove each strut of the scaffold to learn which ones still bear load.

Post-run learning · the harness learns after each run

Every run — good or bad — leaves behind knowledge, and most of it gets thrown away. A mature harness distills each run's result into a durable asset: a new eval case born from every real failure; a rule or trap written into project knowledge; a checklist or skill extracted from a process that repeatedly succeeded. This is JIT retrieval running in reverse — instead of only pulling knowledge in, the harness distills knowledge out.

The key is discipline, not volume: everything persisted must pass through the write gate (Lesson 3.2) — keep only what's verified. Selective distillation turns experience into capability; indiscriminate distillation just turns it into debt.

Anti-pattern · learning sprawl

Automatically persisting every "lesson" after each run sounds great, but it bloats memory and makes even wrong facts permanent — exactly the memory-that-never-forgets trap. Persist selectively and with a date. Measure: the rate of repeating the same class of error across weeks (should fall), the number of eval cases growing over time, and the size of the knowledge-loaded-per-session part not growing linearly with the number of runs.

Exercise · Add bounds to a real agent loop

Takeaway

The harness is the engineering you control when the model changes. Design tools like an API (clear purpose, bounded output, rich errors); the loop needs stopping bounds and a budget; authorization belongs to the harness, not the model; observability lets you reconstruct failures; and ablation tells you which scaffold is still worth keeping.

Lesson 5.1

Claude Code · The three layers of a programmable harness

L3 · Practice
After this lesson you can

  • Write a lean, cache-friendly CLAUDE.md; move domain knowledge into a Skill loaded on demand.

In Chapter 4 we built a harness by hand. Claude Code (and Codex in Chapter 6) is a pre-built harness you configure. Every feature falls into one of three layers — this is the map that keeps you from getting lost among dozens of 2026 features.

CORE · CLAUDE.md — the project's persistent context

The file Claude reads at session start. It sits at the front of the context on every call → it must be stable & cache-friendly (remember Chapter 2). The rule from real cases: keep it lean.

Do put in
  • The repo's unwritten conventions & style
  • Key build/test/run commands
  • The "traps" that caused real bugs in the last 90 days
  • Architecture in one paragraph + a pointer to the docs
Do NOT put in
  • Long domain documents (→ a Skill)
  • Anything dynamic: timestamps, current state (breaks the cache)
  • Things Claude can read from the code itself
  • The entire 2000-line style guide
CLAUDE.md heuristic

If a line is true for every project, or the agent can infer it from the code, consider cutting it. A short map pointing at the right commands, traps and docs is usually more useful than a long manual where the important rules are buried.

DELEGATION · Skills — just-in-time domain knowledge

A Skill is a directory (SKILL.md + supporting files) that describes how to do a kind of work. The description helps the agent recognize when to load the skill; the detailed content only enters the context when the task fits. This is JIT retrieval (Chapter 3) applied to procedural knowledge: keep the index small, load the deep instructions on demand.

DELEGATION · Subagents & git worktree — isolation is a feature

A subagent = a brand-new context for a focused task; the parent only receives the conclusion (exactly Pattern 4, Chapter 3). Git worktree raises this to the filesystem level: each agent works on an isolated copy of the repo → multiple agents edit files in parallel without stepping on each other, then you merge. Use a worktree when (and only when) agents genuinely write files in parallel — it costs disk + setup.

A subagent doesn't only isolate context — it can also accumulate. As of Jul 2026, a subagent can declare a memory field (scope user / project / local) to have a memory directory that persists across sessions, turning it into an "expert" that deepens over time: a reviewer that remembers the repo's traps, a debugger that remembers the failure patterns it has seen. But a subagent's persistent memory also needs a write gate + a forgetting mechanism (Lesson 3.2) — otherwise each subagent becomes its own nest of the memory that never forgets.

EXTENSION · Hooks & MCP

Hooks are programmable harness behavior: run your command on events (PreToolUse, PostToolUse, Stop…) to block dangerous commands, auto-run tests/lint, or enforce conventions. A hook's output is treated as feedback to the agent. MCP extends the tool registry with external servers (DB, API, services). Remember the registry tax from Chapter 2: every MCP enabled costs tokens on every call — enable per project scope, audit with /mcp.

The MCP paradox

The more MCPs, the "stronger" the agent looks on paper but the weaker it is in practice: every tool's schema stuffs the context, dilutes attention and eats the budget. This is where ToolSearch / load-on-demand tools shine.

EXTENSION · ToolSearch — the tool registry at scale (2026)

The escape hatch for a large registry: keep a lean tool index, then load only the JSON schema of the tools that fit the task. This is JIT retrieval applied to the tool definitions themselves. The token savings depend on the number of tools and schema length; measure on real requests.

The async mechanisms above aren't disconnected — they stack into three ascending rungs. Seeing these three rungs tells you where you are and what the next rung buys you.

A loop that repeats an unverified harness just repeats errors faster; a graph that wires up multiple agents with no verifier multiplies errors exponentially. Measure: before adding a rung, be able to point to the current rung's verifier — which test, eval or schema is gating. If you can't name it, you're not ready to climb.

Combining Claude Code + Codex

Two tools can take on different cadences: one builds, the other reviews from a clean context; or split independent branches and merge them through the same verifier. The value comes from separating roles and demanding evidence, not automatically from using two different logos.

Lesson 7.1

Coordinating two agents · builder + reviewer, scout, sync/async

L3 · Practice
After this lesson you can
  • Decide which tool to use for which work with a compact decision tree.
  • Apply a cross-tool builder+reviewer to get genuinely independent review.
  • Balance the benefit of parallelism against the reality of cost & merge risk.

Why combine instead of choose? Because the same model, run twice on the same context, produces two highly correlated answers — even the errors correlate. A different agent (different tool, different system prompt, blind to the original design rationale) is what gives you a genuinely independent view.

Coordination patterns

  1. Parallel terminals / git worktree

    Run multiple agents on isolated worktrees to do independent work in parallel, then merge. Filesystem isolation prevents stepping on each other.

  2. Builder + Reviewer (cross-tool)

    One tool builds, the other reviews with a "only inspect security/correctness" system prompt, blind to the design rationale. This breaks the self-protecting bias.

  3. "Send a scout to reconnoiter"

    Before big work, dispatch an agent to map unknown territory ("find every place the old API is used") and return a map — cheap, and it makes the main work far more accurate.

  4. Sync (Claude Code) + Async (Codex Cloud)

    You design deeply with plan mode in one tool, while delegating the batch of small, well-scoped tasks to the other running in the background.

  5. Share config across tools

    Keep one canonical source of conventions, then symlink or a thin adapter so both CLAUDE.md and AGENTS.md point back to it — avoiding drift between two copies.

The reality of cost & risk

Cost: parallel agents burn the rate limit in parallel — doubling the agents doubles the rate of consuming your quota. Risk: more parallel branches = more merges = more conflicts & integration points. Parallelization is a lever, not a free lunch; use it when the work is genuinely independent.

Discovery & Specification · Where AI creates value earliest

A paradox: AI is most useful before the first line of code is written. But this is also where errors are most expensive — an ambiguous spec, replicated by a fast agent, produces a lot of wrong code.

Lesson 8.1

Discovery & Specification with AI

L4 · Advanced
After this lesson you can
  • Avoid the two failure modes of doing specs with AI.
  • Separate the spec phase from the code phase — each with its own context and quality bar.
  • Use a scout + multi-agent perspective to make a solid PRD before coding.

In Parts I–II you got the foundations and the tools. Part III applies them to each SDLC phase. Start with the most undervalued phase — and where AI delivers the highest value.

Failure mode 1 · spec too thin
  • "Build me a dashboard" → the agent guesses, producing something that looks right but misses the implicit requirements.
  • The faster it goes, the more wrong code there is to clean up.
Failure mode 2 · spec so detailed it's paralyzing
  • Writing a 40-page spec before validating any assumption.
  • Locks in early wrong decisions; wasted effort before you've learned anything.
Root error: merging the spec phase and the code phase into one session

When spec and code share one context, the agent starts coding before the spec is mature, and the spec gets bent to fit the code already written. Separate the phases: brainstorm options → draft a PRD with multi-perspective review → only then implement against the finalized spec. Each phase its own context, its own bar.

Four patterns for doing specs with AI

  1. Brainstorm before scoping

    Use AI to widen the option space ("give me 5 approaches, pros/cons of each") before committing to a direction. It's cheap, and it avoids locking in the first choice.

  2. Send a scout to map unknown territory

    Before spec'ing, dispatch an agent to reconnoiter the codebase/docs: "what related things do we already have? what technical constraints?" The spec rests on reality, not on assumptions.

  3. The PRD is a living document, AI is a co-author

    It's not write-once. AI drafts the PRD, you push back, update it as you learn. Keep it in a file (working memory, Chapter 3), not in a chat session.

  4. Multi-agent perspective on the spec

    Have one agent play a security engineer, one an end user, one QA, all reviewing the spec. Each role finds different gaps.

Spec-driven development

The discipline: the spec is the source of truth, the code is derived. When there's a bug, first ask "is the spec wrong/missing?" before fixing the code. This keeps the agent (and you) from patching symptoms.

Takeaway

AI creates the most value before the first line of code. Don't merge spec and code into one session — separate the phases: brainstorm → a PRD with multi-perspective review → only then implement. The spec is the source of truth; scout first so the spec rests on reality.

Architecture isn't a "correct answer" but finding the best point in a constrained space (cost, team, time, operations). AI is good at widening the space (many options) and playing the expert; you're good at trade-offs against the real context.

The architecture review meeting · with AI in the room

Have each subagent-persona critique the design before the human meeting. Walk in with a list of risks & questions AI already surfaced — people focus on trade-offs, not on finding basic gaps. Remember Chapter 7: the reviewer must be a different agent from the builder.

30-second example · a 5-line condensed ADR

Context: an orders system needs multi-table transactions (orders, payments, inventory) with strong consistency constraints. Decision: use Postgres as the primary store. Rejected option: MongoDB — rejected because it lacks easy multi-document transactions and the order schema is inherently relational. Consequences: we gain ACID + JOINs + foreign-key constraints; in exchange we manage schema migrations and accept harder horizontal scaling. These five lines are an entire ADR — AI drafts it fast, you make the final trade-off.

Exercise · An expert panel for a real decision

Have each role critique the option you're leaning toward. Write a synthesized ADR. What did you find that thinking-alone had missed?

Takeaway

Architecture is constrained search. Let AI widen the options and play multiple expert personas (each role a subagent) to critique the design; you hold the decisions & trade-offs. Record them as ADRs, draw diagrams as code.

Chapter 10 · Part III · Applying it to the SDLC

Implementation · TDD, Plan Mode, Worktree

Where context & harness meet the keyboard. Implementation discipline with an agent is nothing like typing code by hand: you don't write the code, you steer a loop.

Lesson 10.1

Implementation discipline with an agent

L4 · Advanced
After this lesson you can
  • Apply the right TDD cadence for an agent: tests are an executable spec and an automatic feedback loop.
  • Use plan mode to lock the strategy before the agent touches code.
  • Choose a worktree for independent multi-file refactors, and compaction to fight degradation.

A common mistake: hand the agent a big, vague task with no plan, no test loop, then be surprised when it drifts after a while. Implementing well with an agent is discipline about the feedback loop, not about a "clever prompt".

DisciplineWhy it matters even more with an agent
TDDA test is an executable spec + an automatic feedback loop the agent runs itself after every change. Without tests, the agent has no right/wrong signal beyond "looks right".
Plan modeLock the strategy before the agent touches code → avoid drift, avoid context bloat from random trial-and-error.
WorktreeIndependent multi-file refactors run in parallel in isolated copies, merged afterwards — no stepping on each other.
Signal-driven compactionA long session can accumulate junk context (Chapter 2). Compact when near the limit or when the eval shows old history lowering quality; keep the important decisions and evidence.

The Red–Green–Refactor cadence, AI edition

  1. Red · write a failing test first

    Have the agent write (or you write) a test describing the desired behavior. A failing test = a clear definition of "not done".

  2. Green · let the agent make the test pass

    The agent runs the test after every edit (use a hook to automate). The loop has an objective signal, not a guess.

  3. Refactor · clean up with a safety net

    A green test is the net; the agent refactors and you know immediately if it breaks behavior.

A test is a repeatable success criterion

This is exactly "Goal-Driven Execution": turn "fix the bug" into "write a test that reproduces the bug, then make it pass". Strong criteria let the agent loop independently; weak criteria ("make it work") force you to keep clarifying.

30-second example · a TDD session with an agent

Sample prompt: "Bug: applyDiscount() returns a negative price when the discount code > the original price. First write a test that reproduces exactly this bug, run it to confirm it's red, then fix the function until the test is green. Don't touch the code before there's a red test." The agent writes a test expecting the price clamped at 0 → runs it, red (price came out −20) → fixes the function by adding Math.max(0, …) → re-runs, green. You have evidence it's fixed, not a promise of "it's probably done".

Exercise · One bug, one test, one loop

Next time you hit a bug: don't let the agent "fix" it. Ask it to write a test that reproduces the bug first, confirm the test fails, then fix until the test passes. Add a hook that runs the tests after every edit. Feel the difference between "the agent guesses it's fixed" and "the test proves it's fixed".

Takeaway

With an agent you don't write code — you steer a loop. Tests are an executable spec + an automatic feedback loop; plan mode locks the strategy before touching code; worktrees enable parallel refactors; compaction fights degradation.

Quick check

Chapter 11 · Part III · Applying it to the SDLC

Review & Test · A multi-perspective quality gate

An AI reviewer can run a repeatable checklist and scan diffs fast, but it can still miss things, invent issues, or be anchored by context. The value grows when the reviewer is separated from the builder and every finding is verified by a test, static analysis or documentation.

Lesson 11.1

Multi-perspective review & test generation

L4 · Advanced
After this lesson you can
  • Exploit the AI reviewer's structural advantage — and avoid the shared-context reviewer trap.
  • Build an independent, multi-perspective builder → validator chain.
  • Know what AI is good/bad at when generating tests, and use evals as regression tests for AI features.

The structural advantage: AI reads every line of the diff with equal attention, no deference to the author, no "too long to read carefully". But that advantage evaporates if the reviewer is the same agent that wrote the code — it inherits the exact same blind spots (Chapter 7).

Review patternHow & why
Built-in auto-review (Codex)A default quality gate before submitting — catches obvious errors early, cheaply.
Multi-perspective reviewSeparate lenses: correctness, security, performance, "is it reproducible" — each lens an agent. Diversity catches failure modes that repetition can't.
Builder → validator chainAn agent builds, a different agent (a review system prompt, blind to the design rationale) critiques. You can tell the validator to "try to refute" — distrust by default.

Test generation — what AI is good and bad at

AI is good at
  • Mechanical edge-case coverage (null, empty, boundaries, unicode)
  • Table-driven tests with many variants
  • Writing tests for existing code to build a net before refactoring
AI is bad at (needs you)
  • Knowing which behavior is worth testing by business value
  • Wrong-assertion tests (asserting against buggy code instead of the correct spec)
  • Tautology tests (tests that just restate the implementation)
Eval-driven development for AI features

For an LLM-powered feature, the "test" is an eval: a set of input/expected-output examples run as a regression test. Start with cases representing the happy path, the hard cases, and the expensive failures; expand by the diversity and risk you observe. Re-run every time you change the prompt, model or context.

Exercise · A minimal eval set

For an existing AI feature: collect 20 real inputs (including a few that once went wrong). Record the expected outputs. Write a simple scoring script. Now change one thing in the prompt and re-run — you'll see immediately whether it improves or regresses, instead of guessing.

Takeaway

Separate the reviewer from the builder, split review by lens when it's genuinely needed, and verify findings through a builder→validator chain. AI can suggest edge cases; the owner decides which behavior matters and which threshold is enough to ship. AI features need evals as regression tests.

Chapter 12 · Part III · Applying it to the SDLC

Deployment & Operations · AI in Production

AI is usually useful where there's a checkable signal: diffs, logs, runbooks and incidents. Production still needs cost discipline, observability and someone accountable.

Lesson 12.1

AI in CI/CD, incidents & operations

L4 · Advanced
After this lesson you can
  • Place AI at CI/CD points with a clear signal (PR review, runbook generation, incident triage).
  • Use an agent to accelerate incident response while keeping humans in the decision loop.
  • Apply cost & observability discipline to the AI features themselves.

The general rule: AI shines on work with an objective signal to anchor to — test pass/fail, error logs, diffs. It wobbles on vague work with no feedback. Production operations are full of clear signals, so it's good ground.

AI application pointWhy it fits & the boundary
PR review in CIThe diff is a clear signal. An independent AI reviewer (Chapter 11) runs automatically, labels risk. Boundary: suggest, don't auto-merge.
Drafting runbooksAI gathers scattered knowledge into a step-by-step draft; the operator confirms the order, permissions and testing.
Incident responseThe agent is an accelerator: it gathers logs, builds a timeline, proposes hypotheses. Humans hold the decisions & high-risk actions.
Production debuggingGive the agent logs + traces + code → it narrows things down fast. Remember: verify before you trust (fight context poisoning, Chapter 3).
Observability & cost discipline for the AI features themselves

Your AI feature is also a system that needs observability (Chapter 4, Pillar 5): log every call, cache hits, tokens, cost/user. And set a hard cost ceiling — a buggy loop in production can burn real money very fast. Cost runaway is failure mode #9; don't let it happen silently.

30-second example · a 4-step runbook for "queue backlog spiking"

1) Confirm the symptom: are queue depth & oldest-message age genuinely climbing steeply on the dashboard (rule out a false alarm). 2) Narrow the cause: dead/slow consumers, or a producer spike — compare in/out throughput. 3) Relieve pressure immediately: scale the number of consumers, or temporarily throttle the producer; if there's a poison message causing retry loops, move it to a dead-letter queue. 4) Verify & record: watch the queue depth fall back to baseline, then log the timeline & actions for the postmortem. AI generates this frame from the logs; the operator confirms each step.

Exercise · A runbook from a real incident

Take a recent incident. Have the agent read the logs + postmortem and generate a step-by-step runbook for next time. You (who were there) fix where it guessed wrong. The result: an operations document that, without AI, probably would never have been written.

Takeaway

AI is easier to check where there's a clear signal (CI diffs, logs, incidents). Use it to assist analysis, keep humans in the high-risk decision loop. AI features also need observability + a hard cost ceiling.

Chapter 13 · Part IV · Team & sustainability

Playbook for bringing AI to a team

Individual skill doesn't spread to a team on its own. Adoption doesn't fail because the tools are bad, but because of missing standards, missing learn-by-doing, and forgetting the question "when NOT to use AI".

Lesson 13.1

Bringing AI to a team sustainably

L5 · Expert / leading
After this lesson you can
  • Roll out adoption in 3 phases: standards → a learn-by-doing pilot → a mentored rollout.
  • Track cost the right way and know when not to use AI.
  • Fight skill atrophy and measure success at the team level, not the individual level.

This is the L5 chapter: you don't just use it well, you lead others to use it well — and know when to tell the whole team to stop. Adoption failure has a clear pattern, so it's avoidable.

Why adoption fails

Not because the tools are bad. Because: (1) no shared standard (everyone does it their own way, nothing reusable); (2) training via slides instead of learn-by-doing; (3) measuring success wrong (lines of code/day instead of outcomes); (4) forgetting "when NOT to use AI".

Three rollout phases

  1. Phase 1 · Set standards (weeks 1–2)

    Agree on a CLAUDE.md/AGENTS.md template, a shared Skill library, cross-tool review conventions, cost budgets. Standards are what make a skill reusable — the way this very course turns a style into a skill.

  2. Phase 2 · A learn-by-doing pilot team (weeks 3–6)

    A small team applies the standards to real work, recording traps & wins. Not a classroom — real work with a mentor.

  3. Phase 3 · Rollout with mentoring (week 7+)

    Pilot members mentor newcomers in pairs (driver/navigator). Knowledge spreads by working together, not through documents.

When NOT to use AI

Work you don't have the expertise to verify the output of; high-risk decisions lacking a clear signal; places that need originality/taste where the model pulls toward the average; and when cost/latency exceeds the value. Knowing this boundary is a sign of L5, not L1.

TopicDoing it right means
Cost trackingMeasure cost/outcome (per merged PR, per closed ticket), not the total bill. Hard ceiling + alerts.
Fighting skill atrophyStill review & understand AI-written code; rotate "by-hand" work to keep intuition; don't let the whole team lose the ability to read code without AI.
Measuring team-level successLead time, escaped-to-prod defect rate, review health — not "lines of code AI generated".
The closing principle

AI doesn't replace good engineers — it amplifies them. A team with good standards, context & harness discipline, and the sense to know when to stop, will pull far ahead of a team that "uses a lot of AI" without those three. The tool only amplifies; discipline decides what gets amplified.

Exercise · A minimal standard for your team

Write one page: (1) a CLAUDE.md/AGENTS.md template for the main repo; (2) 3 cross-tool review conventions; (3) one team-level success metric you'll track; (4) a "when NOT to use AI" list for your domain. Share it, gather feedback, turn it into a living standard.

Takeaway

Individual skill doesn't spread on its own. Bring AI to a team through standards → a learn-by-doing pilot → a mentored rollout; measure team-level success, not lines of code; and know when NOT to use AI. The tool only amplifies; discipline decides what gets amplified.

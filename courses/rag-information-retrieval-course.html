Chapter 00

Why retrieval at all — the hallucination floor

Large language models confidently answer questions about facts they have never seen. Retrieval is the discipline of feeding them facts before they answer.

An LLM is a probability machine over tokens. It produces fluent, plausible text whether or not the underlying claims are true. For a generic chatbot in 2024 this was tolerable; for a B2B product where the user is asking "what's the trial period for Demo AM?" it is a liability — a wrong answer in confident prose is worse than no answer at all.

Retrieval-Augmented Generation is the architectural answer: before the model writes, search a curated knowledge store, hand the model the relevant passages, and instruct it to answer only from those passages. The pipeline has three jobs:

  1. Index the knowledge into a form that supports cheap similarity search.
  2. Retrieve the right passages for an arbitrary user question.
  3. Ground the model's response in those passages, with citations.

This course is about job #2 — retrieval. Job #1 is the input; job #3 is downstream prompting. Almost every quality complaint about a RAG product traces back to a retrieval failure.

Fig 0 · LLM-only vs RAGOpen in Excalidraw →
Mental model

Think of an LLM as a brilliant intern who has read everything but remembers nothing precisely. RAG is the act of sliding the right page across the desk before they speak.

Test yourself

Why can't we "just give the LLM all our documents in the prompt"?

Try answering before flipping.

Three reasons. Cost: tokens are charged per request, and our entire knowledge base may be megabytes. Latency: long prompts inflate response time linearly. Attention dilution: even with long-context models, accuracy degrades when the relevant fact is buried in 200K tokens of noise. RAG is the cost-aware way to keep only the relevant ~3 KB in the prompt for any given question.

Chapter 01

Naive RAG — embed everything, search by cosine

The minimum viable retrieval pipeline. It works on toy datasets. It breaks the moment you have two products with similar names.

The simplest RAG you can build has four moving parts:

  1. Split each document into fixed-size chunks (say 1,000 characters).
  2. Run each chunk through an embedding model — a function that maps text to a high-dimensional vector (e.g., 1,536 floats for OpenAI's text-embedding-3-small).
  3. Store (chunk_text, vector) in a vector database.
  4. At query time, embed the user's question with the same model, find the K nearest vectors by cosine similarity, and pass those chunks to the LLM.

That's it. The whole magic of vector search is that semantically similar text ends up geometrically close in vector space — even if the words don't overlap. So "free trial" embeds near "no-cost evaluation period" because both are about the same concept.

Fig 1 · Naive RAG: index, embed, cosine-searchOpen in Excalidraw →
Worked example

Suppose your only document is a single PDF: Demo-AM-pricing.pdf. Naive RAG splits it into 3 chunks of 1,000 chars each.

1.Embed all 3 chunks → store 3 vectors of dim 1,536.
2.User asks: "trial free không?".
3.Embed query → 1 vector. Compute cosine similarity to each of the 3. Pick top 1.
4.Send the chunk + the question to the LLM with the prompt: "Answer using only this passage."

Total infrastructure: a Postgres table with a vector column.

Where this breaks

Naive RAG fails in five reliable ways. The remaining 11 chapters of this course exist to fix each one.

Test yourself

Why must the query be embedded with the same model that embedded the chunks?

Two embedding models produce vectors in different coordinate systems. Cosine similarity between vectors from different models is meaningless — geometrically nonsense. The Chatbot Center repo enforces this with a vector_dims(embedding) = ? filter on every retrieval query, and migration 0017 made document_chunks.embedding dimension-agnostic precisely so the system can hold multiple providers' vectors side by side without ever mixing them.

What does cosine similarity actually measure?

The cosine of the angle between two vectors. Range −1 to 1. It ignores magnitude — only direction matters. Two embeddings pointing the same way are "semantically aligned"; perpendicular vectors are unrelated. Most embedding APIs return unit-length vectors so cosine sim equals the dot product.

Chapter 02

Chunking strategies — don't cut the table mid-row

A chunk is the atomic unit retrieval works on. Cut badly and every layer above you is debugging your splitter.

Naive chunking splits every N characters. That's how you get a chunk that ends mid-sentence, mid-table, mid-list — and an embedding that represents nothing coherent.

The fix is natural-separator chunking: walk a hierarchy of separators and only fall back to character slicing as a last resort. The Chatbot Center chunker walks paragraph → line → sentence → word.

Fig 2 · Naive vs natural-separator chunkingOpen in Excalidraw →

Two more knobs: size and overlap

Chunk size trades context-per-chunk against retrieval precision. Bigger chunks carry more context but dilute the embedding signal across multiple ideas. Chatbot Center uses 1500/300 for files, 1000/200 for website pages — smaller for web because pages are shorter and more topical.

Overlap guards against splits that happen to land mid-thought. With 300-char overlap, a sentence near the boundary appears in both adjacent chunks; whichever chunk you retrieve, the sentence is still there.

Worked example

Document: "...the trial lasts 14 days. After that, your team is billed monthly."

ANo overlap, boundary at the period: chunk 1 ends at "14 days." Chunk 2 starts at "After that..." → the link is severed; "After that" loses its referent.
B300-char overlap: chunk 2 begins with "...lasts 14 days. After that..." → the antecedent is preserved.

Chunker code: apps/api/src/modules/rag/processors/chunker.ts. Called from RagService.processSource() with chunkSize: 1500, chunkOverlap: 300 for files, 1000/200 for website pages.

Anti-pattern

Don't chunk by token count using a model-specific tokenizer. Your chunks become coupled to one embedding provider; switching providers (which the codebase explicitly supports — OpenAI, Gemini, Qwen, Ollama) silently changes your splits. Char count is provider-independent.

Test yourself

If overlap is so helpful, why not set it to 50% of chunk size?

Two costs. Storage: every doubled chunk doubles your vector count and embedding cost. Retrieval noise: top-K now returns near-duplicate neighbors that say almost the same thing, and your final context window fills with redundancy. The Chatbot Center default of 20% overlap is the empirical sweet spot — enough to bridge boundaries, not enough to clog top-K with twins.

Chapter 03

Metadata — a chunk is not just text

A vector tells you "this chunk is semantically about X." Metadata tells you "this chunk came from the canonical pricing page, last updated Tuesday, mentions Demo AM."

The naive index stores (text, vector). That's structurally agnostic — a chunk has no idea who it came from, when, or whether it's supposed to outrank a sibling. Every modern RAG pipeline carries a typed metadata contract alongside the vector.

In Chatbot Center, every chunk carries a JSONB metadata object built by KnowledgeMetadataService:

// Per-chunk metadata stored in document_chunks.metadata
{
  sourceId: "src-am-pricing-2026",
  sourceName: "Demo AM Pricing",
  sourceType: "pdf",
  trustLevel: "canonical",        // canonical | authoritative | standard | derived
  sourceRole: "pricing",            // product | pricing | policy | promo | release_notes …
  canonicalSource: true,
  intentAllowlist: ["pricing", "capability"],
  entities: [{entityId: "demo-am", name: "Demo AM"}],
  sectionTitle: "Trial & Pricing",
  keywords: ["trial", "14-day", "price"],
  updatedAt: "2026-04-29T..."
}

Metadata serves three functions in retrieval:

  1. Filtering — only return chunks the user is allowed to see (tenant, language, version state).
  2. Boosting — a chunk from a canonical source with trustLevel: canonical outranks a same-similarity chunk from a promo page.
  3. Routing — a query classified as pricing intent can prefer chunks where sourceRole = pricing or intentAllowlist includes "pricing".
Fig 3 · Metadata is read by filter, boost, and routing logicOpen in Excalidraw →
Design rule from this codebase

Source-level metadata lives in training_sources.config and is the canonical catalog. Chunk-level metadata lives in document_chunks.metadata and stores only what's local to that chunk. Copying the full source entity catalog into every chunk turns routing pages and index sections into false positives — they'd carry the same entity boost as a real descriptive section. Keep chunk metadata narrow.

Builder: apps/api/src/modules/rag/knowledge-metadata.service.ts. Types: rag.types.ts.

Test yourself

You add a new language field to chunk metadata. Should retrieval filter by it, boost by it, or both?

Filter, not boost. Returning English chunks for a Vietnamese query produces a confidently wrong answer — boosting still admits cross-language matches when no in-language chunks exist. A hard filter is correct except when you've explicitly built cross-lingual fallback (e.g., the codebase's "expand Vietnamese trial query with English anchors like free trial" rule, which is a deliberate routing exception, not a metadata filter relaxation).

Chapter 04

Lexical hybrid — BM25 catches what vectors miss

Vectors are great at meaning. They're terrible at exact identifiers. The fix is to run a second, lexical retrieval path in parallel.

An embedding model has been trained to compress meaning. That's exactly the wrong objective when the user types "Demo AM" — a proper noun whose meaning is, well, "Demo AM." Embeddings collapse "Demo AM," "Demo Media," and "Demo Group" toward each other in vector space because they're semantically related. From the user's perspective, retrieving the wrong sibling is a hard failure.

The classic fix is BM25 — Best Matching 25 — a 30-year-old lexical ranking function that scores chunks by how often the query terms (literal words) appear, weighted by how rare those terms are in the corpus (IDF) and damped against chunk length. BM25 doesn't know meaning; it knows literal token overlap. That's exactly the property we want for proper-noun and identifier queries.

Postgres ships a near-cousin: tsvector + ts_rank_cd, with a GIN index for fast prefix and term lookups. That's what Chatbot Center uses — no separate Elasticsearch cluster needed.

Fig 4 · Two retrieval paths run in parallel; lexical anchors proper nounsOpen in Excalidraw →

The tsvector pipeline in Postgres

-- Migration 0021 added this column + GIN index
ALTER TABLE document_chunks ADD COLUMN search_vector tsvector;
CREATE INDEX ... USING GIN (search_vector);

-- Populated from content at insert time
UPDATE document_chunks
SET search_vector = to_tsvector('simple', content);

-- Query time: rank by ts_rank_cd
SELECT id, ts_rank_cd(search_vector, query) AS score
FROM document_chunks, to_tsquery('simple', 'Demo & AM & trial') query
WHERE search_vector @@ query
ORDER BY score DESC LIMIT 20;

Migration: apps/api/drizzle/0021_add_search_vector_to_document_chunks.sql. Lexical ranker: retrieval.service.ts → searchLexical().

Why 'simple' not 'english'?

The 'english' dictionary stems words ("running" → "run") and drops stopwords. Great for English prose, terrible when half your corpus is Vietnamese and the other half contains brand names. 'simple' tokenizes and lowercases without stemming, preserving proper nouns intact across languages.

Test yourself

If lexical is so good for identifiers, why not drop vector search entirely?

Lexical fails on rephrasing. The user types "how much does it cost"; the chunk says "$9 / month / seat". No literal token overlap → BM25 returns nothing. Vector still nails it because both are semantically about pricing. You need both — that's why the next chapter is about fusing them.

Chapter 05

Reciprocal rank fusion — combining two rankings without normalizing scores

You have a vector top-20 and a lexical top-20. How do you produce one merged top-20? RRF is the simplest answer that actually works.

The naive merge is to add the two scores. That fails immediately: cosine similarity ranges 0–1, but ts_rank_cd can be 0.001 to 50+ depending on the query. Adding them means whichever scale is bigger always wins. You'd need to z-normalize each list, deal with edge cases when one list is short, recalibrate when the corpus shifts… it's an endless tuning exercise.

Reciprocal Rank Fusion (RRF) sidesteps the problem by ignoring scores entirely. It uses only the rank a document achieves in each list:

RRF(d) = Σ over each ranker r:  1 / (k + rank_r(d))

The constant k dampens the influence of high-ranked items so a document that's #2 in both lists isn't dominated by a document that's #1 in one and missing from the other. The standard value, used in the Chatbot Center repo, is k = 60.

Fig 5 · RRF combines rankings without score normalizationOpen in Excalidraw →

Implementation: apps/api/src/modules/rag/rrf.ts (53 lines, k=60). Called from RetrievalService.retrieve() after vector and lexical paths complete.

Test yourself

What happens to RRF as k approaches 0?

As k → 0, the formula collapses toward 1/rank — heavy weight on rank-1 items. A document that's first in one ranker dominates a document that's second in both. As k → ∞, all positions become similar and rank no longer matters. k=60 is the empirically-tuned middle from the original RRF paper; it lets a strong consensus across rankers beat a single-ranker peak.

Chapter 06

Multi-query expansion — ask the same thing four ways

Recall lifts when you run several variants of the user's query against the index instead of just the literal text they typed.

Users type imprecisely. "trial?" by itself is not enough signal — the index has 10,000 chunks and many of them mention "trial" tangentially. The fix is to expand one query into several variants and run all of them, then fuse the results.

Chatbot Center expands deterministically (no LLM round-trip required) into up to four variants:

  1. Original — the user's literal text.
  2. Entity-anchored — query rewritten to include the planner's resolved entity. "trial" becomes "Demo AM trial" if context says the conversation is about Demo AM.
  3. Intent-anchored — for a pricing intent, the variant adds anchors like "price plan cost subscription".
  4. Paraphrase — a deterministic restatement (e.g., adding cross-lingual anchors: a Vietnamese query gets an English form appended for English website chunks).
Fig 6 · One query fans out into N variants; results re-fused via RRFOpen in Excalidraw →
Cost watch

4 variants × 2 paths = 8 retrievals per user query. Each is cheap individually, but at scale you want both (a) vector embeddings cached per variant — variants that compute to the same string don't re-embed, and (b) a hard cap on variant count. Chatbot Center caps at MULTI_QUERY_MAX_VARIANTS=4 by default.

Expander: apps/api/src/modules/rag/multi-query.service.ts.

Test yourself

Why deterministic expansion instead of asking an LLM to "rephrase this query 4 ways"?

Latency, cost, predictability. An LLM round-trip on every query adds 300–800ms and a per-query model cost. Deterministic rules give you the same recall lift for free, with output you can reason about. The codebase does reserve the option to add an LLM-based query rewriter for follow-up turns where conversation context matters — but the first-line expander is rule-based.

Chapter 07

Contextual chunking — prepend "what is this chunk about" before embedding

Anthropic's 2024 result: a one-sentence situating context, generated by an LLM and prepended to each chunk before embedding, cuts retrieval failures by 49%.

Consider this chunk, taken verbatim from the middle of a PDF:

"The price is $9 per seat per month, billed annually."

What's it about? Cannot tell. The pronoun "the" refers to a product named on a previous page that the chunker discarded. An embedding of this chunk lands in vector space near every other chunk that says "$9" or "per month" — across all your products. At retrieval time, a user asking "Demo AM price?" may not match it at all.

Contextual chunking fixes this at ingestion time, not retrieval time:

  1. For each chunk, ask a fast LLM to write a 1–2 sentence "situating context" given the chunk and a compact document summary.
  2. Prepend that context to the chunk text before embedding. The embedded vector now reflects "this is the Demo AM pricing section" + the original prose.
  3. Store the original prose unchanged. The context is for embedding; the chunk text shown to the LLM at answer time is the raw prose plus citation metadata.
Fig 7 · Context prefix is embedded with the chunk; raw prose is what's shown at answer timeOpen in Excalidraw →
Why this is different from "just put the section title on the chunk"

Section titles are static and structural. A context prefix is generated knowing the entire document and the chunk's role in it: "This excerpt is from the 'pricing' section of the Demo AM 2026 product brief, comparing the Pro and Enterprise tiers." That's information the chunker can't recover from the chunk alone.

Code: apps/api/src/modules/rag/contextual-chunker.service.ts. Failure isolation: per-chunk; if the LLM call fails, falls back to embedding the raw chunk. Concurrency capped at CONTEXTUAL_CHUNKING_MAX=6.

Test yourself

Why store the prefix in metadata.contextPrefix if we already used it for embedding?

Three reasons. Audit: an operator debugging a misranked chunk wants to see what the embedder actually saw. Re-embedding: when you switch embedding providers and reindex, you can reuse the prefix without re-calling the context LLM. Cache: if the chunk hasn't changed, the prefix doesn't need to regenerate on retrain.

Chapter 08

Cross-encoder reranking — a smaller, smarter judge for the top candidates

Bi-encoders score every chunk in milliseconds; cross-encoders score 30 chunks better than bi-encoders score one. Use both, in sequence.

Vector retrieval (and BM25) belong to a class called bi-encoders: query and document are encoded separately, then compared by a cheap function (cosine, dot). Separate encoding makes them fast — you encode every chunk once at index time, and queries run in milliseconds.

The price of separate encoding is that the encoder never sees query and chunk together. It can't reason about the specific interaction.

A cross-encoder takes (query, chunk) as a single concatenated input and outputs one relevance score. It's an order of magnitude more accurate per pair — and an order of magnitude slower. You can't run it across 100,000 chunks per query.

The standard architecture: bi-encoder retrieves the top 30, cross-encoder reranks those 30 to produce a top 5–15. Cohere Rerank, Jina Rerank, and BGE-Reranker are off-the-shelf services that do exactly this.

Fig 8 · Two-stage retrieval funnelOpen in Excalidraw →
Worked example

Query: "How long is the Demo AM trial?"

ABi-encoder top 5 includes a Demo Media trial chunk — semantically similar, wrong product. Bi-encoder didn't see the query and chunk side-by-side.
BCross-encoder reads "How long is the Demo AM trial?" + "Demo Media offers a 7-day evaluation..." together. Notices the product name mismatch. Scores it low.
CFinal top-1 is the actual Demo AM trial section.

Reranker dispatcher: apps/api/src/modules/rag/reranker.service.ts with 5 adapters under reranker/ (jina, cohere, openrouter, voyage, llm). Configured by the super-admin in the dashboard (Admin → Settings → Reranking) — see Chapter 11 § Reranking dispatcher for why and how. DISABLED by default; if disabled or unreachable, the pipeline falls back to the domain-aware ranker.

Test yourself

Why "top 30 → top 15" rather than "top 100 → top 15"?

Cross-encoder cost grows linearly with pair count, and rerank latency lands directly in your user-facing P50. 30 is the empirical sweet spot: enough that the right answer is almost always somewhere in the input set (otherwise the rerank can't help), few enough that 150–300ms total stays under the chat-streaming budget. Going to 100 doubles latency for diminishing recall gains.

Chapter 09

CRAG quality validation — knowing when retrieval failed

Sometimes retrieval simply doesn't find anything good. The honest answer is "I don't know" — but only if the system can tell.

Even with rerank, the top result can be junk. Maybe the user asked about a topic the corpus doesn't cover. Maybe the rerank scores are all flat — no clear winner. Feeding garbage to the LLM produces a confidently wrong answer; the model has no way to tell low-confidence retrievals from high-confidence ones.

Corrective RAG (CRAG) is the pattern of inspecting retrieval scores before generation and classifying retrieval into one of four states:

StateTriggerAction
confidentTop score > high threshold, clear gap to #2Generate normally with citations
moderateTop score above floor but no clear winnerGenerate but soften: "Based on what I have…"
lowTop score below floorFilter low candidates; tell the model knowledge is incomplete
emptyNo candidates above floorReturn "I don't know" or trigger clarification
Fig 9 · CRAG branches the answering layer based on retrieval confidenceOpen in Excalidraw →

Validator: apps/api/src/modules/rag/quality-validator.service.ts. The classification surfaces on RetrievalResult.quality so the chat engine can pick its prompt strategy.

In this codebase

The CRAG state pairs with the planner's clarification gate: if the planner already detected ambiguity (e.g., the query mentions "Demo" alone with two sibling entities matching), the system asks for clarification before retrieval ever runs. CRAG handles the case where retrieval ran but came up dry.

Test yourself

Why is "I don't know" a feature, not a failure?

A B2B chatbot that says "I don't have that information — would you like to talk to a human?" preserves trust. The same chatbot inventing an answer destroys it. The codebase explicitly enforces this: in chapter 3.4 of KNOWLEDGE-RETRIEVAL-ARCHITECTURE.md, the rule is "do not let general model knowledge override retrieved tenant knowledge" — when retrieval is empty, the model must not fall back on its training data.

Chapter 10

Entity-aware planning — the Chatbot Center secret sauce

Demo has dozens of products with overlapping names: Demo AM, Demo Media, Demo Social, Demo Group. Generic RAG retrieves whichever sibling vector-matches first. That's a quality bug, not a tuning issue.

This chapter is the part of the system that's least like generic RAG literature and most specific to operating a B2B platform with shared brand prefixes.

The core idea: every training source declares an entity catalog in training_sources.config.entities. Each entity has a stable entityId, a canonical name, and a list of aliases. At query time, before any retrieval runs, a planner tries to match the user's text against the catalog.

{
  entities: [
    {
      entityId: "demo-am",
      name: "Demo AM",
      aliases: ["Demo AM", "Demo AM"],
      team: "AM"
    },
    { entityId: "demo-media", name: "Demo Media", ... },
    { entityId: "demo-social", ... }
  ]
}

The planner has three jobs:

  1. Resolve the user's text to one or more candidate entities, prioritizing exact name/alias matches over single-token overlaps.
  2. Reject sibling traps: if the user typed only "Demo" — a token shared by every entity — that's not enough evidence. Multi-word aliases need either an exact phrase hit or at least two overlapping alias tokens.
  3. Clarify: if the planner finds two equally-strong candidates, return a clarification question instead of running retrieval and serving an averaged-across-siblings answer.
Fig 10 · Sibling-aware planner intercepts ambiguous queries before retrievalOpen in Excalidraw →

When the planner does resolve, the resolved entity flows through the pipeline as a hard signal:

  • Multi-query expander adds the entity to the entity-anchored variant.
  • Lexical recall boosts chunks whose metadata.entities[] contains the entityId.
  • The reranker sees the resolved entity in the query, helping cross-encoder judgment.

Planner: apps/api/src/modules/rag/retrieval-planner.service.ts (384 lines). Catalog source: training_sources.config.entities, propagated to each chunk's metadata.entities[] at ingestion but only for entities actually mentioned in that chunk — see Ch 3 callout.

Test yourself

Why doesn't the planner just embed the entity catalog and let vector similarity do the matching?

Because vector similarity is exactly the failure mode this chapter exists to fix. "Demo AM" and "Demo Media" are vector-close — that's why naive RAG confuses them. Entity resolution must be a lexical operation: exact name match, alias match, multi-token overlap rules. Vectors are great for semantic retrieval; they're the wrong tool for proper-noun disambiguation.

Chapter 11

The 2026-Q3 layer — distillation + dispatcher + surgical curation, shipped

Original proposal: docs/proposal/proposal-rag-distillation-2026.md. Everything in this chapter is live in production now — this is the educational walk-through of what landed and why.

The earlier chapters (00-10) cover the 2024-vintage RAG techniques the codebase already ran. Q3 of 2026 brought a layered upgrade in three waves, each targeting a different failure mode the older pipeline still produced. The three waves and what each one fixes:

  1. Distillation (A-C below) — reshape the corpus so retrieval has cleaner candidates to work with: semantic dedup, optional Q→A index, versionState lifecycle.
  2. Reranking dispatcher (D) — replace the single-vendor reranker stub with a 5-provider catalog the super-admin opts into.
  3. Surgical curation (E-G) — per-page metadata override, chatbot scopeMode, synonym entity merge. These cover the failure modes that hit operators in production after the distillation layer was live.

A. Semantic deduplication — shipped

Demo tenants frequently train one chatbot on three overlapping sources: a PDF, a website page, and a manually-typed FAQ — all describing the same product. After ingestion, the index holds 5–15 near-duplicate vectors per fact. Top-K retrieval distributes probability mass across them, suppressing the canonical version.

The fix runs after ingestion, as a separate BullMQ job:

  1. Cluster all chunks for the chatbot by cosine similarity ≥ 0.85.
  2. For each cluster, pick a canonical based on trustLevel, priority, sourceRole, recency.
  3. Tag non-canonicals with metadata.dedupOf = canonical_id. Don't delete — just skip them in retrieval via WHERE metadata->>'dedupOf' IS NULL.

Article benchmark: 2,042 → 1,200 chunks, +13.55% retrieval accuracy.

Fig 11a · Semantic dedup collapses competing near-duplicatesOpen in Excalidraw →

B. Optional Q→A index — shipped (opt-in per chatbot)

Queries are questions; chunks are prose. Even with contextual chunking, the shape mismatch costs retrieval precision. The proposal: for high-value sources (canonical pricing, policy, product docs), generate 3–8 question/answer pairs per chunk during ingestion. Store them as a parallel kind='qa' row alongside the prose, with a parentChunkId back-reference so citations still point at the prose.

Why parallel rows, not replacement: long-form support and explainer queries still want narrative prose. The QA index augments rather than replacing — best of both retrieval shapes, fused via the existing RRF step.

C. versionState lifecycle field — shipped

When a tenant uploads pricing-2025.pdf then later pricing-2026.pdf without deleting the old one, both index simultaneously. Today there's no signal to demote the older one. The proposal adds a versionState ∈ {draft, current, approved, deprecated} field to the metadata contract, and a one-line filter on retrieval: only return rows where versionState IN ('current', 'approved').

It pairs naturally with dedup: when a newer canonical appears, the older duplicate can auto-flip to deprecated.

Migrations: apps/api/drizzle/0030_rag_distillation_metadata.sql (partial indexes for dedupOf / kind / versionState), 0031_website_pages_metadata.sql (per-page override column). Full proposal still at docs/proposal/proposal-rag-distillation-2026.md.

D. Reranking dispatcher — 5 providers, opt-in

Chapter 8 introduced the cross-encoder rerank stage. In the original codebase that stage spoke to a single hard-coded vendor via env vars (RERANKER_URL, …). The 2026-Q3 update replaced it with a provider-agnostic dispatcher + 5 adapters. The super-admin opts in from the dashboard, picks one of the five, pastes a key, hits Test. No env vars touch the secrets path.

Why 5 providers, not 1: each one is a different cost / quality / vendor-lock-in tradeoff. Most platforms only need one at a time, but which one depends on the team.

The 2026-05-23 catalog refresh (no code-path change, only ordering + pricing) made voyage the recommended default: it has by far the most generous free tier in May 2026 and no minimum top-up. Dropdown order in Admin → Settings → Reranking and Chatbot → AI Provider → Reranking is recommended-first, served by apps/api/src/modules/rag/reranker/provider-catalog.ts. E. Per-page metadata override — migration 0031

Source-level metadata (trust, role, intentAllowlist, priority, …) only goes so far. A Website source crawling 30+ URLs inherits one config for all of them. That breaks down when one URL inside the source is noisy — e.g. /license-agreement/ is legal terms but kept winning pricing queries because "license" appears hundreds of times.

The workaround was "split it into its own website source". Migration 0031 added a per-page solution: a sparse JSONB metadata column on website_pages. Operators open a slider dialog on any URL row and override individual fields — sourceRole, intentAllowlist, versionState, trustLevel, canonicalSource. Empty = inherit source.

F. Chatbot scopeMode — single_product vs multi_product

The planner (Chapter 10) refuses to answer when two sibling entities tie on score — it asks the user to clarify. That's correct for a bot covering many products but actively harmful for a bot covering exactly one: synonyms in the entity catalog (EN + VI names of the same concept) end up looking like siblings and the bot loops "Bạn đang hỏi A hay B?".

Field on chatbots.providerConfig.scopeMode:

ModePlanner behavior on entity tieUse for
multi_product (default)Ask clarificationBots covering many distinct products / companies
single_productSuppress clarification, take top entityBots covering exactly one product (phpFox bot, single-tenant bots)

Query-time switch — no retrain. Toggle in Chatbot → Settings → AI Provider → Topical Scope.

G. Synonym entity merge — planner-side

Same root cause as F (sibling clarification fires on synonyms), different fix. When the planner loads the entity catalog, it now computes pairwise overlap coefficient on [name, ...aliases] token sets (after dropping stop-tokens like "the", "của", "với", single letters). When overlap ≥ 0.75 AND intersection ≥ 2 tokens, the entities merge — higher-ranked entity keeps its entityId + name; the other's name and aliases get appended as additional aliases.

Example: "Upgrade V4 to V5" vs "Chuyển đổi từ phpfox V4 lên V5 (MetaFox)". Shared tokens after stop-word strip = {phpfox, v4, v5} (alias adds phpfox), overlap = 1.0 → merge. Empirically conservative enough that "Demo AM Guide" vs "Demo Media Guide" (overlap 0.67) stays split.

Why overlap-coefficient not Jaccard: cross-language synonyms have a long tail of language-specific tokens. Jaccard |A∩B|/|A∪B| penalises that tail and misses legitimate synonyms; overlap |A∩B|/min(|A|,|B|) asks the better question.

Test yourself

Why does the dedup pass tag rather than delete the duplicates?

Three reasons. Reversibility: clustering decisions are heuristic; you might tune thresholds later and want to re-run without re-ingesting. Audit: operators in the Retrieval Debug panel can see which chunks were collapsed and challenge the canonical pick. Recovery: if a canonical row gets deleted (its source removed), the cluster's next-best member can be promoted automatically.

Why generate Q→A pairs only for high-value sources, not all chunks?

Cost. Each pair generation is an LLM call. Across thousands of chunks, the cost stacks up fast — and most chunks (release notes, generic policies, etc.) don't have a clean question/answer shape anyway. Gating on trustLevel ∈ {canonical, authoritative} AND sourceRole ∈ {pricing, policy, product_overview, faq} typically captures 15–25% of chunks while concentrating the cost where the lift is highest.

Why is the reranker disabled by default, even though benchmarks say it always helps?

Every rerank call costs money — either provider fees (Jina, Cohere, Voyage) or LLM tokens (the llm adapter). On-by-default would charge every chatbot the moment they deploy, before the operator decided whether to pay for that quality lift. Off-by-default + explicit Admin → Reranking toggle keeps cost in the operator's control. The dispatcher itself is zero-cost when disabled (early return in resolveConfig).

Why expose per-page sourceRole + intentAllowlist but not per-page priority?

priority lives as a column on training_sources that the retrieval SQL reads via JOIN — adding a per-page priority would mean either two scoring inputs (chunk metadata + row JOIN, with merge logic) or a much bigger SQL refactor. Empirically sourceRole + intentAllowlist are the right knobs anyway: they route on intent match (page only boosts on intents you whitelisted), not a fixed bias. If a per-page priority is ever truly needed, the existing workaround — split the URL into its own 1-URL Website source with its own priority — still works.

Why use overlap coefficient instead of Jaccard for synonym entity merge?

Cross-language synonyms have a long tail of language-specific tokens. "Upgrade V4 to V5" and "Chuyển đổi từ phpfox V4 lên V5 (MetaFox)" share {phpfox, v4, v5} but the second one carries 4 extra Vietnamese tokens. Jaccard |A∩B|/|A∪B| divides by the union, so the tail tokens dilute the score and the pair misses the merge threshold. Overlap coefficient |A∩B|/min(|A|,|B|) divides by the smaller set, asking the right question: "what fraction of the smaller side is shared?" — robust to long-tail asymmetry.

Chapter 12

Caching — cutting token cost & latency for many bots, many users

Source-of-truth: docs/CACHING-ARCHITECTURE.md. Everything here is live in production — the educational walk-through of the cache stack that landed and the measured numbers it produced.

Chapters 0–11 made retrieval good. This chapter makes it cheap and fast at scale. Chatbot Center is multi-tenant: many organizations, each with many bots, each serving many users at once. Every one of those users sends a different question — but to the same bot, whose system prompt, business context and instructions are identical for everyone. Without caching the platform re-bills those shared input tokens and re-runs the entire retrieve → generate pipeline on every single turn, of every user, of every bot.

The fix is the converged 2025–2026 industry pattern: a four-layer cache stack, checked cheapest-and-most-specific first, with every key namespaced by tenant (organization_id + chatbot_id). The layers are complementary, not competing — each catches a cost the others miss.

Fig 12a · The four-layer cache stack and what each hit savesOpen in Excalidraw →

L1 · Exact-match answer cache — Redis, the zero-risk fast path

The narrowest, cheapest layer. The key is a SHA-256 of {organizationId, chatbotId, normalized question, system prompt, provider, model, config}; a hit returns the stored answer in sub-millisecond Redis time and skips retrieval and generation. Because it is an exact hash there are zero false positives — it only fires when a user asks a byte-identical question (after whitespace/case normalization). First-turn only, so the answer never depends on conversation state. This layer pre-existed; the new layers wrap around it.

L2 · Semantic answer cache — pgvector, paraphrase-aware (opt-in)

Exact-match misses on paraphrases: "What's the refund window?" vs "How long do I have to get a refund?" are different bytes but the same question. L2 embeds the normalized question (reusing the same query-embedding cache retrieval already uses — Chapter 1 — so a repeat costs zero extra embedding calls), then finds the nearest cached answer for this bot in the response_cache pgvector table. If cosine similarity ≥ the threshold (default 0.95) and the entry is fresh and the embedding model + knowledge version match, it returns the stored answer — skipping retrieval and the LLM entirely.

Why it ships OFF by default

A semantic cache can return a wrong answer on a false-positive match — "when does the store open?" vs "when does it close?" can score ~0.85. A wrong cached answer is a worse outcome than a cache miss. So L2 is opt-in per platform, defaults to a strict 0.95 threshold (high precision), and is excluded for personalized / escalation / booking turns (those return before the cache path). Tune the threshold up for regulated domains, down only for high-volume FAQ where false positives are tolerable.

L3 · Retrieval-result cache — Redis, the follow-up-turn win

The answer caches (L1/L2) only fire on the first turn. L3 covers everything else: it caches the entire ranked retrieval result (chunks + plan + trace) per {chatbot, normalized query, knowledge version} in Redis. A hit skips multi-query expansion (Ch 6), embedding (Ch 1), the vector + lexical search (Ch 4), RRF (Ch 5) and the reranker HTTP/LLM call (Ch 8) — the LLM still generates, so this is the win on follow-up turns and on rerank/embedding cost. It uses single-flight (one request rebuilds a hot expired key while the rest wait) plus ±10% TTL jitter to prevent a cache stampede.

L0 · Provider prompt caching — the structural win on every generation

The other three layers help repeated questions. L0 helps every generation — even a unique, never-seen question — because the LLM provider can reuse the already-processed stable prefix of the prompt (the bot's persona + business context + instructions, identical for every user of that bot) at roughly 10% of the input price (Anthropic) or 50–90% off (OpenAI/Gemini). The catch: the provider only reuses a prefix that is byte-identical across requests.

The relocation trick — the single highest-ROI change

The legacy prompt interleaved stable content (persona) with per-request content (visitor name, today's date, retrieved RAG chunks). That moving content sits inside the would-be cached prefix, so the prefix changes every request and the provider can never reuse it. The fix is to relocate: put all bot-identical content first as one stable block, and all per-request content after it.

Fig 12b · The relocation trick — make the prefix byte-stable so the provider reuses itOpen in Excalidraw →

For Anthropic the adapter emits the system field as a block array and marks the stable block:

// apps/api/src/modules/ai-providers/adapters/anthropic.adapter.ts
system: [
  { type: 'text', text: stablePrefix, cache_control: { type: 'ephemeral' } }, // 📌 cached
  { type: 'text', text: dynamicSuffix }, // 🔁 RAG / visitor / date — not cached
]
// response.usage.cache_read_input_tokens  →  billed at 0.1x input
// response.usage.cache_creation_input_tokens →  billed at 1.25x (5m) once

Check yourself

Why probe the exact-match cache (L1) before the semantic cache (L2), when L2 catches strictly more questions?

Cost and safety, not coverage. L1 is a sub-millisecond Redis hash lookup with zero false positives; L2 needs an embedding call (or cache read) plus a vector scan (~11 ms) and carries a small false-positive risk. If L1 already has the answer, paying for L2 is pure waste. The ordering is "cheapest and safest first, fall through on miss" — a hit on the inner layer is strictly better, so you never reach the outer one needlessly.

A bot's system prompt is 1,800 tokens and it runs on Claude Haiku. Why does provider prompt caching save nothing here, and what changes on Sonnet?

Per-model minimum cacheable prefix. Haiku 4.5 (and Opus 4.5+) require 4,096 tokens before a prefix is cached; 1,800 < 4,096, so the prefix is silently not cached (no penalty, just no benefit). Sonnet 4.x needs only 1,024 tokens, so the same 1,800-token prefix is cached — modeled at ~−40% total input cost (−84% on the prefix portion). The lever: don't hardcode; read the model from config and only expect L0 savings above its minimum. A knowledge-rich bot (~6k-token prefix) caches on every model.

The #1 risk of a naive semantic cache is cross-tenant leakage. What two mechanisms prevent Bot A from ever receiving Bot B's cached answer?

(1) Key/scope namespacing: every L1/L3 key embeds chatbot_id, and every L2 pgvector lookup carries a WHERE chatbot_id = … predicate — the nearest-neighbour search physically can't see another bot's rows. (2) Provider isolation: Anthropic caches are workspace-scoped (never cross-org), and OpenAI's prompt_cache_key = "org:bot" isolates routing per tenant. Research shows a query-only-keyed cache leaks on up to 95% of queries — so "never key a cache on query text alone" is the rule.

Capstone

A user reports: "When I ask 'What's Demo AM trial period?', the bot answers about Demo Media." Which layers should you investigate, in what order?

Walk up the stack. (1) Planner (Ch 10): did entity resolution fire? Open the Retrieval Debug panel and check matched entities. If the planner didn't resolve "Demo AM" as the entity, the catalog is missing or the alias rules are wrong. (2) Synonym merge (Ch 11 § G): are "Demo AM" and "Demo Media" accidentally being merged because they share too many tokens? Check the overlap-coefficient threshold. (3) Lexical path (Ch 4): did "Demo AM" lexical-match outrank the sibling? If lexical signal is weak, the chunk's search_vector may be missing or section titles unhelpful. (4) Reranker (Ch 8 + Ch 11 § D): is rerank enabled? If yes, did the cross-encoder distinguish AM from Media (it usually does when both entity names appear in the query)? If rerank is off, this might be the cheapest fix. (5) Metadata (Ch 3 + Ch 11 § E): do the chunks carry the right entities[]? Is there a per-page override on a Media page accidentally boosting it? (6) Dedup (Ch 11 § A): if multiple sibling chunks are sharing rank, the canonical pick may be wrong.

Same user reports a follow-up: the bot keeps asking "Are you asking about Upgrade V4 to V5 or Chuyển đổi từ phpfox V4 lên V5?" Where do you fix this?

Two options, both in Chapter 11. Cheaper: set scopeMode = single_product on the chatbot (Ch 11 § F) — suppresses the clarification when there's an entity tie, takes the top-scoring entity directly. This is correct for any bot covering exactly one product. More structural: clean up the entity catalog so EN + VI versions of the same concept aren't separate entities — either consolidate them or let the synonym entity merge (Ch 11 § G) do it automatically. The merge fires when overlap ≥ 0.75 AND intersection ≥ 2 tokens; "phpfox v4 v5" shared in both passes the bar.

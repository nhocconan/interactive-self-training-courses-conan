MCP & Tool Layer — chuẩn kết nối agent với tool và dữ liệu

Cho kỹ sư đã xây được agent và giờ cần một tầng tool/integration chuẩn hoá thay vì N×M adapter viết tay. Đi từ kiến trúc host · client · server, sáu primitive và lifecycle JSON-RPC, tới transport, OAuth 2.1 resource server, ba đòn tấn công đã có case thật và cách phòng thủ, thiết kế tool mà agent dùng được thật (description-as-prompt, progressive disclosure, code execution) — và cả câu hỏi ngược: khi nào KHÔNG cần MCP. Bám bản spec ổn định 2025-11-25, cập nhật tháng 7/2026.

Module 01 · Phần I · Nền tảng

Nền tảng — vì sao cần một chuẩn

Trước khi đọc spec, phải thấy bài toán nó giải: N client × M tool = N×M tích hợp viết tay. Ba bài này cho bạn khung nền — vì sao cần một chuẩn mở, kiến trúc host·client·server với ranh giới bảo mật nằm ở host, và sáu primitive với «người cầm lái» riêng của từng cái. Khoá này giả định bạn đã có một agent gọi được tool (muốn học xây agent, xem «Loop Engineering»).

Bài 1.1

Bài toán N×M: vì sao mỗi tích hợp lại viết tay một lần

L1 · Nền tảng
Học xong bài này bạn có thể
  • Chỉ ra vì sao số lượng tích hợp viết tay bùng nổ theo tích N×M.
  • Giải thích MCP đổi độ phức tạp đó về N+M bằng một protocol chung.
  • Nhận ra tình huống trong công việc mình đang trả cái giá N×M.

Đội bạn có một agent chạy tốt. Tuần này nó cần đọc Slack, tra GitHub và query Postgres — bạn viết ba adapter riêng, mỗi cái một kiểu auth, một kiểu trả kết quả. Tháng sau sếp muốn thử con agent đó trên một model khác của hãng khác; SDK khác, cách gọi tool khác, thế là ba adapter viết lại gần như từ đầu. Rồi đội data muốn agent của họ cũng nối vào đúng ba nguồn ấy — và họ lại viết ba adapter nữa vì code của bạn dính chặt vào runtime của bạn. Chưa ai làm gì sai; chỉ là mỗi cặp (một bên gọi × một nguồn dữ liệu) đang đòi một mẩu code dán tay riêng.

Đây đúng là bài toán mà Anthropic nêu khi công bố MCP ngày 25/11/2024, nguyên văn: “Every new data source requires its own custom implementation, making truly connected systems difficult to scale.” — mỗi nguồn dữ liệu mới lại đòi một bản tích hợp riêng, khiến hệ thống thật sự liên thông rất khó mở rộng. Giải pháp họ đề xuất là “a new standard for connecting AI assistants to the systems where data lives” — một chuẩn chung để nối AI với nơi dữ liệu đang nằm.

Đặt tên cho nó

Bài toán N×M và tầng tích hợp (integration layer)

Gọi N là số bên gọi tool (client/model/app) và M là số tool / nguồn dữ liệu. Không có chuẩn chung, mỗi bên phải tự biết cách nói chuyện với từng nguồn → số bản tích hợp cần viết là N×M, và mỗi khi thêm một model hay một nguồn, con số lại nhân lên. MCP (Model Context Protocol) chèn vào giữa một integration layer — tầng trung gian nói một ngôn ngữ chung: mỗi nguồn dữ liệu chỉ cần một MCP server, mỗi bên gọi chỉ cần một MCP client, tổng công sức tụt về N+M.

Phân biệt nhanh với function calling — cơ chế để một model gọi hàm mà có thể bạn đã dùng: function calling là cách một model diễn đạt “tôi muốn gọi hàm X” bên trong một SDK cụ thể; MCP là chuẩn kết nối liên-quy-trình nằm bên dưới, để cái hàm X đó sống trong một server dùng lại được cho mọi client tuân spec. Function calling trả lời “model xin gọi gì”, MCP trả lời “tool nằm ở đâu và ai cũng gọi được bằng cách nào”.

Mô hình để nhớ

N×M giống như mỗi người ở một nước tự học tiếng của mọi người còn lại để nói chuyện — thêm một người, ai cũng phải học thêm một thứ tiếng. MCP là chọn một ngôn ngữ chung: mỗi người chỉ học nó một lần. Chỗ analogy gãy: ngôn ngữ chung của người có thể mơ hồ, còn MCP là spec có phiên bản — ai không nói đúng bản đã thoả thuận thì kết nối từ chối, chứ không “tạm hiểu nhau”.

Áp dụng
▶BackendBạn expose một dịch vụ nội bộ cho nhiều agent
Thay vì mỗi đội agent tự viết client HTTP + parser riêng cho service đơn hàng của bạn, bạn viết một MCP server bọc service đó. Đội A (Claude), đội B (một model khác), công cụ dev của đội C — tất cả nối vào cùng một server bằng cùng một giao thức. Bạn sửa logic một chỗ, mọi client hưởng.
▶DataBạn có 5 nguồn (warehouse, CRM, sheet, API giá, log)
Không MCP: một agent phân tích cần 5 tích hợp, agent báo cáo cần 5 tích hợp nữa — 2×5 = 10. Có MCP: 5 server nguồn + 2 client = 7, và khi thêm agent thứ ba chỉ cộng thêm 1 client chứ không phải 5 tích hợp. Tiết kiệm tăng nhanh đúng lúc hệ bạn phình to.

Khoá này giả định bạn đã có một agent gọi được tool. Muốn học xây vòng lặp agent, xem «Loop Engineering»; kỹ nghệ ngữ cảnh và harness, xem «Context & Harness Engineering»; đo độ tin cậy, xem «Agent Reliability & Evaluation».

Takeaway

MCP không “thêm tính năng” cho agent — nó đổi độ phức tạp tích hợp từ N×M adapter viết tay sang N+M qua một protocol chung, nên nó đáng giá đúng khi bạn có nhiều bên gọi phải nối nhiều nguồn dùng chung.

Kiểm tra nhanh

Đội bạn đang chạy 2 agent (2 model khác nhau) và muốn cả hai cùng nối tới 4 nguồn dữ liệu nội bộ. Một bạn đề xuất “dựng MCP cho gọn”. Lập luận nào đúng về cái MCP thực sự thay đổi ở đây?

Bài 1.2

Host · client · server — ranh giới bảo mật nằm ở host

L1 · Nền tảng
Học xong bài này bạn có thể
  • Phân vai host, client, server đúng nghĩa MCP.
  • Giải thích vì sao ranh giới bảo mật nằm ở host, không ở server.
  • Chỉ ra vì sao một server lạ không “nhìn thấy” dữ liệu của server khác.

Bạn cài ba MCP server vào cùng một agent: một server truy vấn DB nội bộ, một server đọc file local, và một server thời tiết ngoài Internet mà bạn mới thấy trên mạng. Câu hỏi làm bạn chần chừ không bấm cài: khi agent vừa query bảng khách hàng ở DB nội bộ vừa hỏi thời tiết, liệu cái server thời tiết kia có “ngó” được kết quả DB đang chạy qua cùng một cuộc hội thoại không? Nếu có, thì mỗi server lạ bạn thêm vào là một cửa rò dữ liệu.

MCP trả lời câu này bằng kiến trúc client–host–server, và câu trả lời nằm ở ai được thấy gì.

Đặt tên cho nó

Ba vai: host, client, server

  • Host — tiến trình điều phối (ví dụ app chat, IDE). Nó tạo và quản lý các client, thực thi chính sách bảo mật và consent, và là nơi duy nhất tổng hợp toàn bộ ngữ cảnh hội thoại.
  • Client — do host sinh ra, mỗi client giữ đúng một kết nối tới một server. Spec ghi rõ: “each client having a 1:1 relationship with a particular server” — mỗi client có quan hệ 1-1 với đúng một server.
  • Server — cung cấp một năng lực chuyên biệt (DB, file, API…), chạy độc lập, chỉ nhận đúng phần ngữ cảnh cần cho việc của nó.

Nguyên tắc thiết kế bảo mật số một của MCP, nguyên văn: “Servers should not be able to read the whole conversation, nor ‘see into’ other servers.” — server không được đọc toàn bộ hội thoại, cũng không được “nhìn xuyên” sang server khác. Cơ chế thực thi nó chính là kiến trúc trên: vì mỗi client chỉ nói chuyện 1-1 với một server và toàn bộ lịch sử nằm ở host, nên server thời tiết vật lý không có đường nào chạm tới kết quả DB — không phải vì nó “ngoan”, mà vì đường dây không tồn tại.

Hệ quả thực tế cho code của bạn

“1 client = 1 server” không phải chi tiết vụn: khi host của bạn tích hợp filesystem + DB + API ngoài, bạn phải mở nhiều client song song trong cùng host — không có kiểu “một client nói chuyện với N server”. Điều này quyết định luôn cách bạn quản connection pool và đặt security boundary trong code.

Hiểu lầm
  • “Server nào cũng thấy hết hội thoại nên phải tin từng server.”
  • “Một client route được tới nhiều server cho tiện.”
Đúng theo spec
  • Chỉ host thấy toàn hội thoại; server chỉ nhận phần cần cho việc của nó.
  • Mỗi client 1-1 với một server; muốn thêm nguồn thì thêm client.
Takeaway

Trong MCP, ranh giới bảo mật nằm ở host — nơi duy nhất thấy toàn hội thoại; mỗi client là một dây 1-1 tới một server, nên một server không thể đọc hội thoại hay ngó sang server khác vì đường dây đó không tồn tại, chứ không phải vì nó tự nguyện.

Kiểm tra nhanh

Bạn thêm một MCP server thời tiết (bên thứ ba, chưa rõ nguồn) vào một agent đang có sẵn server DB nội bộ. Sếp hỏi: “Nó có đọc lén được kết quả query DB không?”. Câu trả lời đúng theo kiến trúc MCP là gì?

Bài 1.3

Sáu primitive, ba người cầm lái

L1 · Nền tảng
Học xong bài này bạn có thể
  • Kể tên sáu primitive của MCP và bên nào expose chúng.
  • Gán đúng “người cầm lái” cho tool, prompt và resource.
  • Tránh lỗi UX để model tự kích hoạt thứ vốn phải do user chọn.

Đội bạn build một MCP client. Có một tính năng tiện: server cung cấp vài prompt dựng sẵn (kiểu “tóm tắt ticket theo mẫu công ty”), hiển thị như slash-command. Để cho “thông minh”, một bạn cho model tự động kích hoạt các prompt đó khi nó đoán là hợp. Kết quả: giữa chừng hội thoại model tự nhét một mẫu prompt của server vào, user không hề chọn, và mất kiểm soát cái mình tưởng mình đang lái. Lỗi không nằm ở code — nằm ở chỗ nhầm ai được quyền quyết định dùng primitive nào.

Đặt tên cho nó

Primitive và ba kiểu “người cầm lái”

Một primitive là một khối tương tác chuẩn của MCP. Có sáu cái, chia theo bên expose. Server đưa ra: tools, resources, prompts. Client đưa ra: sampling, roots, elicitation. Điểm cốt lõi không phải “có sáu cái” mà là mỗi cái có một người cầm lái khác nhau — spec nói thẳng:

  • Tools = “model-controlled” — model tự phát hiện và gọi tool theo ngữ cảnh. Nhưng kèm khuyến nghị: “there SHOULD always be a human in the loop with the ability to deny tool invocations” — luôn nên có người duyệt/chặn.
  • Prompts = “user-controlled” — server đưa ra để user chủ động chọn (đúng như slash-command), không phải để model tự bật.
  • Resources = “application-driven” — host/app quyết đưa dữ liệu nào vào ngữ cảnh theo nhu cầu của nó.

Hai primitive phía client đáng nhớ riêng. Sampling là cách server “mượn” LLM của client: server xin client chạy completion hộ, nguyên văn “with no server API keys necessary” — server không cần API key riêng, và client luôn giữ quyền duyệt prompt lẫn response trước khi trả lại. Elicitation là server xin thêm input từ user giữa chừng (ví dụ xác nhận địa chỉ), nhưng spec cấm tuyệt đối: “Servers MUST NOT use elicitation to request sensitive information.” — không được dùng để xin thông tin nhạy cảm (mật khẩu, thẻ tín dụng…).

Bẫy UX hay gặp

Để model tự kích hoạt prompt (vốn user-controlled) khiến user mất quyền chọn; ngược lại, bắt user tự tay chọn từng resource (vốn application-driven) làm trải nghiệm rối và nặng. Thiết kế client đúng là gán đúng người cầm lái cho từng primitive, không xử lý cả sáu như nhau.

Takeaway

Sáu primitive không chỉ khác chức năng mà khác người cầm lái: tool do model quyết, prompt do user chọn, resource do application đưa vào — thiết kế client là gán đúng vai đó, đừng để model bật thứ vốn thuộc quyền user.

Kiểm tra nhanh

Một MCP server đưa ra vài prompt mẫu (hiển thị như slash-command trong client của bạn). Cách để chúng vào hội thoại như thế nào là đúng vai theo MCP?

Module 02 · Phần II · Giải phẫu protocol

Giải phẫu protocol: JSON-RPC, schema, annotation

Giờ mở nắp protocol: mọi message là JSON-RPC 2.0, phiên làm việc mở đầu bằng bắt tay capability, tool khai schema cho cả đầu vào lẫn đầu ra, và annotations chỉ là hint chứ không phải hợp đồng. Ba bài này là phần «đọc spec cho đúng» — nền cho mọi quyết định thiết kế và bảo mật phía sau.

Bài 2.1

JSON-RPC và lifecycle: initialize → operate → shutdown

L2 · Vận dụng
Học xong bài này bạn có thể
  • Phân biệt ba loại message JSON-RPC 2.0: request, response, notification.
  • Kể đúng trình tự ba pha của phiên: initialize → operation → shutdown.
  • Chẩn đoán một message "rơi vào hư không" từ khung capabilities, không phải từ code gửi.

Bạn cắm một MCP server theo dõi thư mục dự án vào agent. Server phát hiện một file đổi và gửi notifications/resources/list_changed để agent biết danh sách resource vừa cập nhật. Agent… im lặng. Không lỗi, không log, không phản ứng — cái thông báo rơi vào hư không. Bạn thêm log ở server: message đã gửi đi thật. Vậy nó mất ở đâu? Câu trả lời không nằm trong đoạn code gửi, mà nằm ở bản hợp đồng hai bên ký với nhau ngay lúc bắt tay đầu phiên.

Ở Module 1 bạn đã thấy mỗi client giữ đúng một kết nối 1-1 với một server, và host là nơi tổng hợp toàn bộ ngữ cảnh. Bài này mở nắp kết nối đó ra: trên dây, hai bên nói chuyện bằng ngôn ngữ gì, theo trình tự nào, và ràng buộc gì quyết định một message có được lắng nghe hay không.

Đặt tên · Ngôn ngữ trên dây

JSON-RPC 2.0 — ba loại message, một quy tắc id

JSON-RPC 2.0 là format message nền của MCP: một chuẩn nhắn tin dạng JSON có sẵn từ trước, MCP không phát minh lại. Spec nói thẳng: "All messages between MCP clients and servers MUST follow the JSON-RPC 2.0 specification" — mọi message giữa client và server MCP bắt buộc tuân theo JSON-RPC 2.0. Có đúng ba loại:

  • Request — có id (không được null, không tái dùng trong cùng phiên), có method, có thể có params. Đây là câu hỏi chờ trả lời.
  • Response — cùng id với request, và phải có đúng một trong hai field result hoặc error ("A response MUST NOT set both" — không được đặt cả hai).
  • Notification — một thông báo một chiều: "MUST NOT include an ID" (không mang id) và bên nhận "MUST NOT send a response" (không được trả lời). Chính vì thế, một notification không được xử lý sẽ im lặng tuyệt đối — không có gì báo cho bạn biết nó bị bỏ rơi.

Message notifications/resources/list_changed ở đầu bài chính là một notification. Nó không chờ response, nên khi bên kia phớt lờ, không có tín hiệu lỗi nào phát ra. Nhưng vì sao bên kia phớt lờ? Vì pha mở đầu của phiên.

Đặt tên · Vòng đời phiên

Lifecycle ba pha và capability negotiation

Mỗi phiên MCP đi qua ba pha: 1. Initialization → 2. Operation → 3. Shutdown. Client bắt buộc mở màn bằng một initialize request mang theo protocolVersion, capabilities (những tính năng nó hỗ trợ) và clientInfo. Server đáp lại bằng protocolVersion, capabilities của mình và serverInfo. Cuối cùng client gửi notification notifications/initialized để báo "sẵn sàng".

Bước trao đổi capabilities đó gọi là capability negotiation: cả hai khai báo tính năng hỗ trợ ngay lúc initialize, và "Both parties must respect declared capabilities throughout the session" — hai bên chỉ được dùng những gì đã khai, suốt cả phiên. Spec nêu ví dụ đúng tình huống của bạn: "Emitting resource subscription notifications requires the server to declare subscription support" — muốn phát thông báo thay đổi resource, server phải khai capability tương ứng trước.

Đây là ba message bắt tay thật, chép nguyên từ ví dụ trong spec (bản 2025-06-18). Để ý khung capabilities ở cả hai chiều — đó là phần quyết định message nào được lắng nghe:

Bắt tay initialize · trích spec 2025-06-18
// 1) Client → Server
{ "jsonrpc": "2.0", "id": 1, "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "roots": { "listChanged": true },
      "sampling": {},
      "elicitation": {}
    },
    "clientInfo": { "name": "ExampleClient", "version": "1.0.0" }
  } }

// 2) Server → Client (cùng id: 1)

protocolVersion cũng phải "hợp" nhau

Cùng lúc negotiate capabilities, hai bên chốt protocolVersion: "If the server supports the requested protocol version, it MUST respond with the same version" — trùng thì server đáp lại đúng version đó; nếu không, server đề nghị một version khác nó hỗ trợ, và nếu client không nhận được thì "SHOULD disconnect". Version không khớp trả lỗi JSON-RPC -32602 ("Unsupported protocol version") kèm data: {supported: ["2024-11-05"], requested: "1.0.0"} — nhớ mã -32602 này, bài sau ta gặp lại nó.

Áp dụng
▶DebugKhi một notification "im lặng biến mất"
Đừng bắt đầu ở code gửi (message đã đi thật). Bắt đầu ở log của initialize: khung capabilities hai bên có khai đúng tính năng bạn đang phát không? Với resources/list_changed, server phải khai resources.listChanged. Thiếu khai báo ⇒ bên kia được phép phớt lờ đúng theo spec, không phải bug.
▶Nâng cấpKhi kết nối "chết ngay từ initialize"
Nếu client báo lỗi -32602 ngay khi mở phiên, khả năng cao protocolVersion hai bên không nằm trong tập hỗ trợ chung. Xem field data.supported trong lỗi để biết server chấp nhận những version nào, rồi pin client về một trong số đó — đừng đoán.
Takeaway

Mọi message MCP là JSON-RPC 2.0, và initialize là bản hợp đồng capability của cả phiên — bên phát một tính năng chưa được khai báo thì bên kia có toàn quyền phớt lờ. Debug "message rơi vào hư không" bắt đầu từ khung capabilities, không phải từ code gửi.

Kiểm tra nhanh

Server của bạn phát notifications/resources/list_changed mỗi khi thư mục đổi, nhưng client không bao giờ phản ứng. Bạn đã xác nhận server thực sự gửi message đi. Nguyên nhân khả năng cao nhất là gì?

Bài 2.2

Tool schema: từ text mù sang structured output

L2 · Vận dụng
Học xong bài này bạn có thể
  • Giải thích vì sao outputSchema + structuredContent thay thế việc parse text.
  • Đọc một CallToolResult trả song song content-text và structuredContent.
  • Trả đúng tầng lỗi: protocol error cho request sai, isError cho tool chạy-mà-hỏng.

Tool get_weather của bạn trả về một câu: "Nhiệt độ 22.5°C, trời có mây, độ ẩm 65%." Agent phải regex bới lấy con số. Tuần sau ai đó đổi format sang "22.5C / partly cloudy / humidity: 65" — regex gãy, agent đọc nhầm độ ẩm thành nhiệt độ, và cả chuỗi quyết định phía sau sai theo. Không ai đổi "API", chỉ đổi một dấu phẩy trong text mà tool nhả ra.

Nhớ ở Module 1: tools là primitive model-controlled — model tự quyết khi nào gọi. Nhưng model chọn đúng tool chưa đủ; kết quả trả về cũng phải ở dạng code hạ nguồn dùng được mà không phải đoán. Đây là chỗ tool schema vào cuộc.

Đặt tên · Hai đầu của schema

inputSchema (bắt buộc) và outputSchema (từ 2025-06-18)

Mỗi tool luôn có inputSchema: một JSON Schema mô tả tham số đầu vào — client biết phải truyền gì, kiểu gì. Từ bản 2025-06-18, tool có thể khai thêm outputSchema (tùy chọn) cho đầu ra: "Tools may also provide an output schema for validation of structured results." Khi đã khai, ràng buộc là hai chiều — "Servers MUST provide structured results that conform to this schema" (server bắt buộc trả kết quả đúng schema) và "Clients SHOULD validate structured results against this schema" (client nên kiểm chứng lại).

Kết quả có kiểu đó nằm ở field structuredContent của CallToolResult. Để không phá vỡ client cũ, spec yêu cầu trả song song: "a tool that returns structured content SHOULD also return the serialized JSON in a TextContent block" — vừa có content dạng text (JSON đã serialize thành chuỗi), vừa có structuredContent dạng object có kiểu.

Đây là response hợp lệ của get_weather_data, chép nguyên số liệu từ ví dụ trong spec 2025-06-18:

CallToolResult · structured output
{
  "content": [
    { "type": "text",
      "text": "{\"temperature\": 22.5, \"conditions\": \"Partly cloudy\", \"humidity\": 65}" }
  ],
  "structuredContent": {
    "temperature": 22.5,
    "conditions": "Partly cloudy",
    "humidity": 65
  }
}

Nửa còn lại của "agent dùng được" là lỗi. MCP tách lỗi làm hai tầng — nhầm tầng là một lý do phổ biến khiến agent kẹt vòng lặp:

Tầng lỗiKhi nào dùngVí dụ
Protocol error (JSON-RPC error)Bản thân request đã saiGọi tool không tồn tại; params malformed; cursor không hợp lệ (-32602)
Tool execution error (isError: true trong result)Tool đã chạy nhưng thất bạiAPI bên ngoài lỗi/timeout; input sai theo business logic

Nhớ mã -32602 ở bài 2.1? Đó là ví dụ điển hình của tầng protocol error: request tự nó sai. Nhưng khi tool đã chạy mà hỏng — API thời tiết timeout chẳng hạn — đó là tầng thứ hai. Bản 2025-11-25 còn làm rõ: lỗi validate input nên trả về dạng Tool Execution Error (isError) chứ không phải Protocol Error, để model đọc được và tự sửa. Message lỗi vì thế phải viết cho model đọc, không phải cho log:

Tool execution error · viết cho model đọc
{
  "content": [
    { "type": "text",
      "text": "Weather API timed out. Current state: no data returned. Recovery: retry once, or call with cached=true to use the last known value." }
  ],
  "isError": true
}
Đừng ném stack trace thô

Message lỗi tốt gồm ba thứ: trạng thái hiện tại, hành động khắc phục, và phương án thay thế — đồng thời sanitize lỗi từ upstream trước khi trả. Một stack trace thô khiến model lặp lại đúng lỗi cũ; một message có hướng recover giúp agent tự đổi hướng mà không cần người canh.

Takeaway

Khai outputSchema và trả structuredContent để code hạ nguồn nhận JSON có kiểu thay vì regex bới text; và khi tool chạy nhưng hỏng, báo bằng isError: true với message model tự-sửa-được — đừng ném stack trace, đừng nhầm sang lỗi protocol.

Kiểm tra nhanh

MCP server của bạn gọi một API thời tiết bên ngoài và API đó timeout. Tool đã chạy, chỉ là không lấy được dữ liệu. Bạn nên trả lỗi này thế nào cho đúng chuẩn MCP để agent tự xử lý tiếp?

Bài 2.3

Annotations là hint, không phải hợp đồng

L2 · Vận dụng
Học xong bài này bạn có thể
  • Kể bốn annotation và giá trị mặc định "nguy hiểm nhất" của chúng.
  • Giải thích vì sao annotation là hint để UX, không phải bảo đảm an toàn.
  • Quyết định đúng khi một server chưa tin cậy tự khai readOnlyHint: true.

Bạn khai destructiveHint: false cho tool search_docs với suy nghĩ: "client sẽ tin nó vô hại và auto-approve, khỏi phiền người dùng." Nghe hợp lý. Nhưng rồi bạn cài thêm một server lạ tải từ đâu đó — và mọi tool của nó cũng khai readOnlyHint: true. Nếu client thật sự "tin" lời khai đó, một tool thực chất xoá dữ liệu vừa được auto-approve mà không ai kịp bấm gì.

Ở Module 1 bạn đã thấy host là nơi thực thi ranh giới bảo mật, và spec khuyến nghị "there SHOULD always be a human in the loop with the ability to deny tool invocations" — luôn nên có người trong vòng lặp, có quyền từ chối lời gọi tool. Annotations tương tác trực tiếp với vòng người-đó, nên hiểu sai nó là mở toang cửa.

Đặt tên · Bốn hint và mặc định của chúng

Annotation — mô tả hành vi tool, do server tự khai

Annotation là bốn cờ mô tả hành vi một tool để client hiển thị/quyết định UX. Điểm mấu chốt trước cả định nghĩa: chúng do server tự khai. Bốn cờ và giá trị mặc định (lấy nguyên từ TypeScript schema của spec):

AnnotationNghĩa (verbatim + Việt)Mặc định
readOnlyHint"the tool does not modify its environment" — không sửa môi trườngfalse
destructiveHint"may perform destructive updates" — có thể phá huỷ (chỉ có nghĩa khi readOnly=false)true ⚠
idempotentHint"calling… repeatedly… no additional effect" — gọi lại cùng tham số không thêm tác dụngfalse
openWorldHint"may interact with an 'open world' of external entities" — chạm tới thực thể ngoài (vd web search)true ⚠

Để ý hai dòng đánh dấu ⚠: nếu bạn không khai gì, tool mặc định là destructiveHint: true và openWorldHint: true — tức mặc định ở mức nguy hiểm nhất, và một client tuân spec sẽ luôn xin xác nhận. Muốn tool đọc-only của bạn được auto-approve, bạn phải chủ động khai readOnlyHint: true.

Câu spec bạn phải thuộc

"clients MUST consider tool annotations to be untrusted unless they come from a trusted server." — client bắt buộc coi annotation là không đáng tin, trừ khi chúng đến từ một server đã được tin cậy. Nói cách khác: readOnlyHint: true từ một server lạ chỉ là lời tự khai, không phải bằng chứng.

Mô hình để nhớ

Annotation giống nhãn "hàng dễ vỡ" người gửi tự dán lên thùng: với người gửi bạn quen, cái nhãn giúp bạn xếp hàng nhẹ tay (auto-approve tool read-only) — tiện thật. Nhưng với một kiện từ người lạ, cái nhãn "vô hại" không đảm bảo bên trong không có gì nguy hiểm. Chỗ analogy gãy: thùng hàng không tự viết lại nhãn giữa đường — còn định nghĩa tool thì có thể bị đổi ngầm sau khi được duyệt (đòn "rug pull" bạn sẽ gặp ở Phần bảo mật).

Takeaway

Annotation là hint do server tự khai để cải thiện UX (auto-approve tool read-only), không phải cam kết an toàn — spec bắt client coi annotation là untrusted trừ khi tới từ server đã tin cậy, và mặc định luôn ở mức nguy hiểm nhất (destructive=true, openWorld=true).

Kiểm tra nhanh

Bạn vừa cài một MCP server mới, chưa rõ nguồn gốc. Một tool của nó khai readOnlyHint: true. Client nên làm gì cho đúng tinh thần spec?

Module 03 · Phần III · Transport & phiên bản

Transport & phiên bản

Protocol chạy trên dây nào và phiên bản nào — hai câu hỏi tưởng phụ mà quyết định kiến trúc deploy. Bài đầu so stdio với Streamable HTTP (và vì sao HTTP+SSE cũ bị thay); bài sau dạy kỹ năng ít ai dạy: đọc changelog spec như kỹ sư, pin version, và biết bản nào đang là release candidate chưa final.

Bài 3.1

stdio và Streamable HTTP — vì sao HTTP+SSE bị thay

L2 · Vận dụng
Sau bài này bạn có thể
  • Chọn đúng transport — stdio hay Streamable HTTP — theo cảnh deploy server của bạn.
  • Giải thích vì sao HTTP+SSE hai-endpoint bị thay bằng Streamable HTTP một-endpoint.
  • Nhận ra rủi ro DNS rebinding và cấu hình bắt buộc để chặn nó.

3 giờ chiều thứ Sáu. Team vừa đẩy MCP server ra sau một load balancer để nhiều agent gọi chung. SDK các bạn dùng vẫn là bản cũ, transport kiểu SSE hai-endpoint: một kết nối SSE phải giữ mở suốt phiên để server push. Vấn đề: load balancer round-robin cứ chuyển request sang instance khác, kết nối dài hạn rụng liên tục, agent mất notification giữa chừng. Không phải code sai — bạn đang dùng đúng một transport mà spec đã khai tử.

Đặt tên hai transport chuẩn

stdio và Streamable HTTP

Spec MCP định nghĩa đúng "two standard transport mechanisms" và khuyên "Clients SHOULD support stdio whenever possible". stdio transport — client spawn server làm subprocess, server đọc JSON-RPC từ stdin, ghi ra stdout (mỗi message một dòng, "MUST NOT contain embedded newlines"), dùng stderr để log — dành cho server chạy local. Streamable HTTP — một endpoint HTTP duy nhất nhận cả POST lẫn GET — dành cho server remote, và nó "replaces the HTTP+SSE transport from protocol version 2024-11-05".

Callback bài 2.1: nhớ header MCP-Protocol-Version phải đi kèm mọi request HTTP sau bước initialize? Đó chính là hợp đồng chỉ tồn tại trên transport HTTP — stdio spawn subprocess thì không có khái niệm header này.

DNS rebinding — cảnh báo lý thuyết đã thành CVE thật

Streamable HTTP kèm một lệnh MUST: "Servers MUST validate the Origin header on all incoming connections to prevent DNS rebinding attacks", và "SHOULD bind only to localhost (127.0.0.1)". Vì sao gắt vậy? Chính SDK TypeScript chính thức của MCP từng tắt sẵn bảo vệ này: CVE-2025-66414 (CVSS 7.6, High) — bản <1.24.0 để "DNS rebinding protection disabled by default", khiến một website độc hại DNS-rebind rồi gọi thẳng vào MCP server local không auth. Đã vá ở 1.24.0 (publish 2025-12-02).

Chuyển dần, không cắt phựt

Muốn đỡ client cũ trong lúc migrate, spec cho phép backwards-compat: "Continue to host both the SSE and POST endpoints of the old transport, alongside the new 'MCP endpoint'." Nghĩa là bạn dựng endpoint /mcp mới nhưng vẫn giữ cặp endpoint cũ tới khi client cập nhật xong.

Áp dụng — chọn transport theo cảnh deploy
▶Server localCông cụ chạy trên máy dev (đọc file, git, DB local)
Dùng stdio: client spawn server làm subprocess, giao tiếp qua stdin/stdout. Không cần mở cổng mạng, không cần Origin check, credential lấy từ environment. Đây là mặc định spec khuyến khích ("SHOULD support stdio whenever possible").
▶Server remoteNhiều agent gọi chung, chạy sau load balancer
Dùng Streamable HTTP một-endpoint. Bật ngay Origin validation, bind localhost nếu chỉ nội bộ, kiểm SDK ≥ 1.24.0 nếu là stack TypeScript. Dùng Mcp-Session-Id để quản session và Last-Event-ID để resume khi kết nối đứt — chính hai thứ transport SSE cũ thiếu.
Takeaway

stdio cho server chạy local, Streamable HTTP một-endpoint cho remote — và mọi server HTTP phải validate Origin header, nếu không DNS rebinding biến một website bất kỳ thành kẻ gọi tool local (đúng như CVE-2025-66414).

Quick check

Bạn expose một MCP server qua HTTP tại 127.0.0.1 cho agent nội bộ dùng, không đặt auth vì "chỉ local thôi". Cấu hình nào là bắt buộc theo spec để không lặp lại CVE-2025-66414?

Bài 3.2

Spec tiến hoá: đọc changelog như kỹ sư

L3 · Thực chiến
Sau bài này bạn có thể
  • Định vị các mốc bản spec có ngày và thay đổi phá vỡ tương thích của từng bản.
  • Pin protocol version và đọc changelog trước khi nâng SDK — thay vì học thuộc một bản.
  • Phân biệt bản stable hiện hành (2025-11-25) với một release candidate chưa final.

Server của bạn chạy êm nhiều tháng, dựa vào JSON-RPC batching để gộp nhiều lời gọi vào một request. Sáng nay CI tự bump SDK lên minor version mới, và server bắt đầu reject mọi request batch. Không có dòng release note nào nổ ra trước mặt bạn — vì thay đổi đó nằm trong changelog của bản spec 2025-06-18, bản đã bỏ batching. Học thuộc "MCP làm thế này" không cứu được bạn; đọc changelog thì có.

Đặt tên cơ chế

Protocol version negotiation

MCP không có "một phiên bản" — nó có nhiều bản gắn ngày, và hai bên thoả thuận bản dùng chung lúc bắt tay. Quy tắc protocol version negotiation: "If the server supports the requested protocol version, it MUST respond with the same version. Otherwise, the server MUST respond with another protocol version it supports... If the client does not support the version in the server's response, it SHOULD disconnect." Không khớp thì có lỗi rõ ràng — mã -32602 "Unsupported protocol version", kèm data liệt kê bản server hỗ trợ. Vì thế pin version (khai đúng bản mình build cho) là kỹ năng nền, không phải tuỳ chọn.

Callback bài 2.2: structuredContent/outputSchema mà bạn học để agent hết phải regex-parse text — nó ra đời đúng ở mốc 2025-06-18, cùng bản đã bỏ batching. Cùng một changelog vừa cho bạn một thứ vừa lấy đi một thứ; đọc cả hai chiều mới an toàn nâng cấp.

Module 04 · Phần IV · Bảo mật

Bảo mật MCP: OAuth, tấn công, phòng thủ

MCP mở ra một bề mặt tấn công đặc trưng: phần model đọc được mà user không thấy. Ba bài này đi từ luồng OAuth 2.1 đúng vai (MCP server là resource server), qua ba đòn tấn công đã xảy ra thật — tool poisoning, rug pull có CVE, confused deputy — tới ba lớp phòng thủ cấp MUST của spec.

Bài 4.1

OAuth 2.1: MCP server là resource server, không phải authorization server

L3 · Thực chiến
Sau bài này bạn có thể
  • Phân biệt vai trò resource server và authorization server trong một hệ MCP có auth.
  • Đọc được luồng 401 → metadata → PKCE → Bearer token và chỉ ra chỗ user thật sự bấm "đồng ý".
  • Quyết định khi nào KHÔNG dùng OAuth (transport stdio) và lấy credential từ đâu.

Một dev dựng MCP server tra cứu đơn hàng cho công ty. Anh làm luôn màn hình đăng nhập, tự sinh và tự lưu token ngay trong server — chạy được trong demo. Ba tháng sau, phòng bảo mật yêu cầu mọi service phải đăng nhập qua SSO công ty (Okta), và audit hỏi "token này ai cấp, hết hạn theo policy nào". Cả hai câu đều không trả lời được: server đang tự đóng vai lò phát token, một vai mà spec MCP nói rõ nó không nên nhận.

Đặt tên

Resource server ≠ authorization server

Spec MCP (bản 2025-06-18) tách bạch hai vai. MCP server = resource server: nó chỉ nhận request kèm token và validate token đó. Authorization server là một thực thể riêng lo việc nói chuyện với user, hiện màn hình consent, và phát token. Nguyên văn spec: "A protected MCP server acts as an OAuth 2.1 resource server… The authorization server is responsible for interacting with the user (if necessary) and issuing access tokens… It may be hosted with the resource server or a separate entity." — tức MCP server có thể ở chung máy với authorization server, nhưng vai trò thì tách rời và bạn không tự chế cơ chế phát token bên trong MCP server.

Điểm này quan trọng vì nó cho phép cắm thẳng vào SSO có sẵn (Auth0, Okta, Keycloak). MCP server chỉ cần biết cách kiểm token, còn ai đăng nhập / cấp token là chuyện của authorization server — đúng mô hình mọi hệ enterprise đang chạy.

Ba MUST của client giữ cho luồng này an toàn, tất cả trích spec 2025-06-18:

  • PKCE bắt buộc — "MCP clients MUST implement PKCE". PKCE (Proof Key for Code Exchange) là cặp bí mật verifier–challenge client tự sinh, đảm bảo chỉ đúng client ban đầu mới đổi được authorization code lấy token — chặn kẻ chặn code giữa đường.
  • Protected Resource Metadata (RFC 9728) — MCP server "MUST include the authorization_servers field" và "MUST use the HTTP header WWW-Authenticate when returning a 401". Đây là cách client tự khám phá "phải đi hỏi authorization server nào".
  • Resource Indicator (RFC 8707) — client "MUST identify the MCP server that the client intends to use the token with" qua tham số resource là URI chuẩn của server (vd https://mcp.example.com/mcp). Đây là audience binding: token được đóng dấu "chỉ dùng cho server này" — nền tảng cho phòng thủ ở bài 4.3.
stdio thì đừng OAuth

Với transport stdio (server local, đã học ở bài 3.1), spec nói ngược lại: "Implementations using an STDIO transport SHOULD NOT follow this specification, and instead retrieve credentials from the environment." — server chạy như subprocess cục bộ thì lấy credential từ biến môi trường là đúng chuẩn, không cần dựng cả bộ OAuth. Nhưng khi bạn expose server đó qua HTTP, toàn bộ luồng trên áp dụng ngay.

Áp dụng
▶Backend · dựng serverBạn được giao viết MCP server có auth cho công ty
Đừng viết màn hình login hay code phát token. Trỏ server tới authorization server thật (Okta/Keycloak/Auth0). Việc của bạn: (1) trả 401 + WWW-Authenticate khi thiếu token; (2) publish Protected Resource Metadata tại .well-known/oauth-protected-resource chứa authorization_servers; (3) validate audience mọi token trước khi xử lý.
▶Review · duyệt PRĐồng nghiệp mở PR thêm MCP server mới
Câu hỏi review đầu tiên: "Server này tự phát token hay validate token của authorization server ngoài?" Nếu thấy code sinh JWT / lưu password / có bảng user ngay trong MCP server → dừng lại: nó đang lấn vai authorization server, sẽ không tích hợp nổi SSO và không audit được về sau.
Takeaway

MCP server là resource server — chỉ validate token có đúng audience; việc login, consent và phát token giao cho một authorization server tách biệt. Tự chế lò phát token trong MCP server là tự khoá mình khỏi SSO và khỏi audit.

Kiểm tra nhanh

Team bạn cần MCP server (chạy qua HTTP) tra cứu đơn hàng, và công ty đã có Okta làm SSO. Kiến trúc auth nào đúng chuẩn spec?

Bài 4.2

Ba đòn tấn công đã xảy ra thật: tool poisoning, rug pull, confused deputy

L4 · Nâng cao
Sau bài này bạn có thể
  • Chỉ ra bề mặt tấn công đặc trưng của MCP: phần model đọc mà user không thấy.
  • Phân biệt tool poisoning, rug pull và confused deputy qua cơ chế, không chỉ qua tên.
  • Gọi tên đúng ca thật (WhatsApp MCP, CVE-2025-54136) khi mô tả rủi ro cho đồng nghiệp.

Một người cài thêm một MCP server nhỏ khai báo tool get_fact_of_the_day() — "lấy một sự thật vui mỗi ngày", nghe vô hại. Vài hôm sau, toàn bộ lịch sử chat WhatsApp của họ bị gửi ra một số điện thoại lạ. Không có lỗ hổng code nào bị khai thác: cái server "fact of the day" đó giấu trong mô tả tool một đoạn chỉ thị mà chỉ AI đọc được — bảo agent, khi gọi send_message của server WhatsApp, hãy đổi người nhận thành số của kẻ tấn công rồi kèm theo toàn bộ lịch sử chat. Đây là ca công khai của Invariant Labs, đăng 2025-04-07.

Đặt tên

Bề mặt tấn công = khoảng cách visibility giữa AI và user

Định nghĩa gốc từ Invariant Labs: "A Tool Poisoning Attack occurs when malicious instructions are embedded within MCP tool descriptions that are invisible to users but visible to AI models." — tool poisoning là nhét chỉ thị độc vào mô tả tool, thứ user không đọc nhưng model đọc và làm theo. Nhớ lại bài 2.3: spec đã cảnh báo "clients MUST consider tool annotations to be untrusted". Ở đây ta thấy đúng lý do của lệnh đó — không chỉ annotation, mà cả description đều là nội dung do server tự khai, tuyệt đối không được tin mặc định.

Ba đòn dùng chung khe hở đó nhưng khác thời điểm và khác cơ chế:

Vì sao đây không phải rủi ro lý thuyết

Benchmark MCPTox (arXiv, 2025-08) chạy trên 45 MCP server thật, sinh 1.312 test case độc hại. Kết quả: mô hình o1-mini bị tool poisoning thành công với attack success rate 72,8%; và tỉ lệ agent từ chối hành vi độc rất thấp — mô hình tốt nhất trong benchmark (Claude-3.7-Sonnet) có refusal rate dưới 3%. Nhận xét chung của nhóm: "more capable models are often more susceptible" — model càng mạnh càng dễ dính, vì nó càng làm theo chỉ thị tốt.

Takeaway

Bề mặt tấn công đặc trưng của MCP là khoảng cách visibility: model đọc toàn bộ mô tả tool, user chỉ thấy tên. Chỉ thị ẩn ở đó là tool poisoning; đổi định nghĩa sau khi được approve là rug pull; và consent cookie dùng chung biến proxy thành confused deputy.

Kiểm tra nhanh

Một MCP server bạn đã approve tuần trước, description tool vẫn y nguyên trên UI, nhưng hôm nay agent bỗng chạy một lệnh shell lạ mỗi lần khởi động. Nhiều khả năng đây là đòn nào, và vì sao spec để hở?

Bài 4.3

Phòng thủ theo tầng: audience check, cấm passthrough, hash-pin tool list

L4 · Nâng cao
Sau bài này bạn có thể
  • Áp ba lệnh MUST của spec như một checklist phòng thủ theo tầng.
  • Vẽ đúng luồng token đúng (đổi token riêng cho downstream) thay vì forward token của client.
  • Giải thích vì sao hash-pin tools/list chặn được rug pull đã học ở bài 4.2.

Một MCP server được dựng theo kiểu "pure proxy": nhận token của client rồi forward y nguyên sang API nội bộ để gọi hộ. Chạy tốt — cho tới khi phòng bảo mật mở audit log của API downstream và thấy mọi request từ 200 nhân viên đều mang cùng một danh tính. Không truy được ai đã sửa đơn hàng nào. Cái tiện lúc dựng (forward thẳng token) đã xoá sạch dấu vết kiểm toán.

Đặt tên

Token passthrough — anti-pattern spec cấm tuyệt đối

Spec định nghĩa: "'Token passthrough' is an anti-pattern where an MCP server accepts tokens from an MCP client without validating that the tokens were properly issued to the MCP server and passes them through to the downstream API." Và ra hai lệnh cứng: "MCP servers MUST NOT accept any tokens that were not explicitly issued for the MCP server", cùng "The MCP server MUST NOT pass through the token it received from the MCP client." — token passthrough là chuyển thẳng token của client xuống dưới; spec cấm, không phải khuyến nghị.

Ba tầng phòng thủ, mỗi tầng đóng một bề mặt tấn công ở bài 4.2 — tất cả là MUST của spec:

  1. Audience validation — chặn token lạ

    Spec: "MCP servers MUST reject tokens that do not include them in the audience claim." Validate mọi token trước khi xử lý; token hợp lệ về chữ ký nhưng phát cho server khác vẫn phải reject. Đây là mặt trái của resource indicator (bài 4.1): đã đóng dấu audience thì phải kiểm audience.

  2. Cấm token passthrough — giữ audit trail

    Không bao giờ forward token của client xuống downstream. Cần gọi upstream thì đóng vai OAuth client riêng, lấy token khác (Token Exchange, RFC 8693). Nếu server là proxy gọi hộ nhiều client: bắt buộc per-client consent registry và redirect_uri khớp exact string match — đây là mitigation chính thức cho confused deputy.

  3. Hash-pin tools/list — chặn rug pull

    Hash-pin = băm toàn bộ tools/list (name + description + inputSchema) ở lần kết nối đầu, so lại ở mọi lần sau; bất kỳ delta nào — kể cả thêm một khoảng trắng — phải chặn và bắt re-approve. Chính là cách MCPoison (CVE-2025-54136) được vá: sau bản Cursor 1.3, "any change… including something as small as adding a space… triggers a mandatory approval prompt".

Cả server tham chiếu chính chủ cũng từng quên validate input

CVE-2025-68145 (GHSA-j22h-9j4x-23w5) trên mcp-server-git chính thức: thiếu path validation khi dùng flag --repository, cho phép thao tác trên repo ngoài phạm vi cấu hình (CWE-22 Path Traversal, CVSS 6.4, vá ở bản 2025.12.18). Bài học: "server MUST validate tool inputs" là yêu cầu chính thức, nhưng ngay cả reference server cũng có lúc lỡ — đừng giả định input đã sạch.

Checklist audit · copy vào PR review
[ ] Server reject token sai audience? (không chỉ kiểm chữ ký/hạn dùng)
[ ] Server KHÔNG forward token client xuống downstream? (dùng token riêng)
[ ] Nếu là proxy: có per-client consent registry, redirect_uri exact match?
[ ] Có hash-pin tools/list, chặn mọi delta (kể cả khoảng trắng) → re-approve?
[ ] Tool inputs được validate (path/args) trước khi thực thi?
Takeaway

Ba MUST của spec chặn phần lớn bề mặt: validate audience mọi token, không forward token client xuống downstream, hash-pin tools/list. Mỗi lệnh đóng đúng một đòn đã học — token lạ, audit-trail vỡ, và rug pull.

Module 05 · Phần V · Thiết kế server tốt

Thiết kế server tốt

Server «chạy được» khác server «agent dùng được». Ba bài này là phần thiết kế: tool description là prompt, granularity theo workflow thay vì wrap REST 1:1, ngân sách context với progressive disclosure và code execution — và câu hỏi trung thực nhất khoá: khi nào bạn KHÔNG cần MCP.

Bài 5.1

Tool description là prompt; granularity theo workflow

L4 · Nâng cao
Sau bài này bạn có thể
  • Viết tool description như đang onboard một người mới, không phải copy tên field từ API doc.
  • Đặt tên tool có namespace theo service + resource.
  • Gộp tool theo workflow agent hay đi chung, thay vì wrap 1:1 mỗi REST endpoint.

Team bạn có sẵn một REST API 30 endpoint, nên dựng MCP server bằng cách wrap thẳng: 30 endpoint thành 30 tool, description copy nguyên từ tài liệu API. Kết quả: để đặt một cuộc họp, agent phải chain list_users → list_events → create_event; và nó chọn nhầm tool vì description chỉ ghi "returns user object" — thứ viết cho lập trình viên đọc, không phải cho một tác nhân phi tất định đang quyết định gọi gì.

Khái niệm

Description-as-prompt · namespacing · tool granularity

Anthropic gọi tool là "a fundamentally new software paradigm: contracts between deterministic systems and non-deterministic agents" (hợp đồng giữa hệ tất định và tác nhân phi tất định). Vì phía đọc là model, "Tool descriptions are prompts. Every word in your tool's name, description, and parameter documentation shapes how agents understand and use it" — mỗi chữ trong tên, mô tả, tài liệu tham số đều định hình cách agent hiểu và dùng tool. Cách viết đúng: "think of how you would describe your tool to a new hire" — mô tả như cho người mới vào team (nói rõ context ngầm, thuật ngữ nội bộ, quan hệ giữa các resource), và đặt tên tham số tường minh (user_id thay vì user). Namespacing: đặt tiền tố theo service + resource (asana_search, jira_search, asana_projects_search). Tool granularity: "avoid tools that merely wrap existing software functionality" — gộp các bước agent hay đi chung thành một tool có mục đích rõ.

Granularity sai
  • 1 tool = 1 REST endpoint (list_users, list_events, create_event)
  • Agent phải chain 3 call cho một việc
  • Nhiều điểm rơi để chọn nhầm hoặc quên bước
Granularity theo workflow
  • 1 tool schedule_event gộp check-availability + tạo event
  • Khớp bước agent hay đi chung với nhau
  • Ít lượt quyết định hơn → ít chỗ sai hơn
Bằng chứng thực nghiệm

Anthropic báo cáo Claude Sonnet 3.5 đạt kết quả tốt nhất trên SWE-bench Verified tại thời điểm đó sau khi họ tinh chỉnh chính xác các tool description — "even small refinements... can yield dramatic improvements." Đây là khẳng định định tính, bài không nêu con số phần trăm; điều rút ra là: viết tool description thuộc quy trình prompt-engineering, không phải việc làm cho có sau khi code xong. (Anthropic Engineering, "Writing effective tools for AI agents", 11/09/2025)

Nhắc lại bài 4.2

Ở bài 4.2 bạn thấy tool poisoning lợi dụng đúng kênh này: model đọc trọn description mà người dùng chỉ thấy tóm tắt. "Description là prompt" là mặt sáng của cùng một sự thật — kênh mạnh nhất để hướng dẫn agent cũng là kênh tấn công tiêm chỉ thị ẩn. Thiết kế tốt và phòng thủ dùng chung một hiểu biết.

Takeaway

Tool description là prompt — mỗi chữ trong tên/param định hình cách agent dùng tool; viết như onboarding người mới, đặt tên có namespace, và gộp tool theo workflow thay vì wrap 1:1 endpoint REST cũ.

Bài 5.2

Ngân sách context: progressive disclosure và code execution

L4 · Nâng cao
Sau bài này bạn có thể
  • Nhận ra vì sao nhồi hết tool definitions vào context không scale.
  • Lộ tool dần bằng progressive disclosure (search_tools + cursor pagination).
  • Dùng code execution để giữ kết quả trung gian ngoài context window.

Agent của bạn nối 5 server. Mỗi request "cõng" toàn bộ tool definitions của cả 5 vào system prompt trước cả khi đọc câu hỏi của user. Rồi nó tải một transcript cuộc họp từ Google Drive để đính vào một Salesforce lead — cả nội dung transcript chạy qua context window hai lần (đọc ra rồi ghi vào). Một spreadsheet 10.000 dòng cũng đi trọn qua model dù bạn chỉ cần 5 dòng. Context không phải tài nguyên miễn phí, và bạn đang đốt nó vào những thứ model không cần đọc.

Khái niệm

Progressive disclosure · code execution với MCP

Anthropic mô tả vấn đề gốc: "In cases where agents are connected to thousands of tools, they'll need to process hundreds of thousands of tokens before reading a request" — hàng trăm nghìn token tiêu trước khi đọc request. Hai lối thoát. Progressive disclosure: lộ tool dần theo nhu cầu — một tool search_tools(query, detail_level) trả 3 mức chi tiết tăng dần (chỉ tên → tên + description → full schema), thay vì nhồi hết định nghĩa vào system prompt ngay từ đầu. Code execution với MCP: để agent viết code gọi các server như một filesystem-of-tools (servers/google-drive/getDocument.ts, servers/salesforce/updateRecord.ts) — chỉ load định nghĩa cần dùng, và giữ kết quả trung gian ở execution environment; model chỉ nhận confirmation cuối. Trong ví dụ Google Drive → Salesforce, cách này giảm từ 150.000 token xuống 2.000 token — tiết kiệm 98,7%. (Anthropic Engineering, "Code execution with MCP", 04/11/2025)

Áp dụng: ba cần gạt giảm token
▶search_toolsLộ tool dần theo 3 mức chi tiết
Thay vì đưa full schema của mọi tool vào system prompt, cho agent gọi search_tools(query, detail_level): mức 1 trả về chỉ tên, mức 2 thêm description, mức 3 mới full schema. Agent tự tăng độ chi tiết khi thật cần — token định nghĩa tool trở thành chi phí theo nhu cầu, không phải chi phí cố định mỗi request.
▶response_formatGiới hạn kích thước response tool
Claude Code mặc định giới hạn response tool ở 25.000 token. Thêm tham số kiểu response_format: "concise" | "detailed" để agent tự chọn mức chi tiết: chế độ concise dùng khoảng ⅓ số token so với detailed. Đừng trả cả một bảng khổng lồ khi model chỉ cần vài dòng.
▶paginationCursor-based cho list lớn
Với tools/list, resources/list, prompts/list… MCP dùng cursor-based pagination: "The cursor is an opaque string token." Page size do server quyết, client MUST NOT giả định cỡ trang cố định và MUST coi cursor là opaque (không parse, không lưu qua session). Cursor không hợp lệ → trả đúng lỗi JSON-RPC -32602 (Invalid params) thay vì crash.
Nhắc lại bài 2.1 & 2.2

Mã lỗi -32602 ở trên chính là JSON-RPC 2.0 của bài 2.1 — cùng bộ error code bạn đã gặp khi bắt tay initialize. Và việc "giữ kết quả trung gian ngoài context, chỉ trả cái model cần" nối thẳng với structured output của bài 2.2: downstream code nhận JSON có kiểu để lọc, thay vì đẩy cả transcript text qua model để nó tự parse.

Takeaway

Đừng nhồi hết tool definitions vào context — lộ tool dần (progressive disclosure) và để code giữ kết quả trung gian ngoài context: một ví dụ Anthropic giảm 150.000 → 2.000 token (98,7%).

Đây là quan điểm phản biện công khai, không phải khẳng định của Anthropic hay MCP maintainers — nhưng là bộ lọc thực tế trước khi bỏ công dựng server. Takeaway

Một server đạt chuẩn không phải là «gắn thêm nhiều tính năng». Nó là một tập quyết định, mỗi quyết định trỏ về một bài đã dạy vì sao phải làm vậy. Distill · mỗi bài một dòng mang đi
BàiMang đi
1.1MCP không thêm tính năng — nó đổi độ phức tạp tích hợp từ N×M adapter viết tay sang N+M qua một protocol chung; đáng giá đúng khi có nhiều bên gọi phải nối nhiều nguồn dùng chung.
1.2Ranh giới bảo mật nằm ở host — nơi duy nhất thấy toàn hội thoại; mỗi client là một dây 1-1 tới một server, nên server không đọc được hội thoại hay ngó sang server khác vì đường dây đó không tồn tại, chứ không phải vì tự nguyện.
1.3Sáu primitive khác nhau ở người cầm lái: tool do model quyết, prompt do user chọn, resource do application đưa vào — thiết kế client là gán đúng vai, đừng để model bật thứ vốn thuộc quyền user.
2.1Mọi message là JSON-RPC 2.0; initialize là bản hợp đồng capability của cả phiên — phát một tính năng chưa khai báo thì bên kia có quyền phớt lờ. Debug "message rơi vào hư không" bắt đầu từ khung capabilities, không từ code gửi.
2.2Khai outputSchema và trả structuredContent để code hạ nguồn nhận JSON có kiểu thay vì regex bới text; tool hỏng thì báo bằng isError với message model tự-sửa-được — đừng ném stack trace, đừng nhầm sang lỗi protocol.
2.3Annotation là hint server tự khai để cải thiện UX, không phải cam kết an toàn — spec bắt client coi annotation là untrusted trừ khi tới từ server đã tin cậy; mặc định luôn ở mức nguy hiểm nhất (destructive=true, openWorld=true).
3.1stdio cho server local, Streamable HTTP một-endpoint cho remote — và mọi server HTTP phải validate Origin, nếu không DNS rebinding biến một website bất kỳ thành kẻ gọi tool local (đúng như CVE-2025-66414).
3.2Đừng học thuộc một bản spec — pin protocol version và đọc changelog trước khi nâng SDK; bản stable hiện hành là 2025-11-25, còn 2026-07-28 vẫn là release candidate chưa final (tính đến 07/2026).
4.1MCP server là resource server — chỉ validate token có đúng audience; login, consent và phát token giao cho một authorization server tách biệt. Tự chế lò phát token trong MCP server là tự khoá mình khỏi SSO và khỏi audit.
4.2Bề mặt tấn công đặc trưng của MCP là khoảng cách visibility: model đọc toàn bộ mô tả tool, user chỉ thấy tên. Chỉ thị ẩn ở đó là tool poisoning; đổi định nghĩa sau approve là rug pull; consent cookie dùng chung biến proxy thành confused deputy.
4.3Ba MUST của spec chặn phần lớn bề mặt: validate audience mọi token, không forward token client xuống downstream, hash-pin tools/list — mỗi lệnh đóng đúng một đòn đã học: token lạ, audit-trail vỡ, và rug pull.
5.1Tool description là prompt — mỗi chữ trong tên/param định hình cách agent dùng tool; viết như onboarding người mới, đặt tên có namespace, và gộp tool theo workflow thay vì wrap 1:1 endpoint REST cũ.
5.2Đừng nhồi hết tool definitions vào context — lộ tool dần (progressive disclosure) và để code giữ kết quả trung gian ngoài context: một ví dụ Anthropic giảm 150.000 → 2.000 token (98,7%).
5.3MCP đáng giá nhất khi cần chuẩn hoá cho nhiều client/model/server dùng chung; nếu nội bộ, một model, ưu tiên tốc độ dev thì CLI/function calling trực tiếp có thể rẻ hơn — chọn bằng câu hỏi, không bằng hype.
6.1«Ai đứng sau MCP» trả lời bằng mốc, không cảm tính: Anthropic khởi xướng 2024-11-25, nay Agentic AI Foundation quản trị từ 2025-12-09 với hơn 10.000 server công khai và 97M+ lượt tải SDK/tháng — governance giữ nguyên, adoption đa nhà.
6.2Server đạt chuẩn không phải phép cộng tính năng — mà là mỗi quyết định (granularity, structured output, transport, audience check, hash-pin, progressive disclosure) đều truy được về một bài đã học; checklist là bản đồ để tự kiểm tra, không phải để học thuộc.
Checklist · khi dựng một MCP server mới

  1. Có cần MCP không?

    Nhiều client/model/server dùng chung → có; nếu chỉ một script nội bộ, một model, ưu tiên tốc độ dev thì cân nhắc CLI/function calling trực tiếp — chọn bằng câu hỏi, không bằng hype (5.3).

  2. Gán primitive đúng vai

    tool cho việc model tự quyết, prompt cho việc user chọn, resource cho việc application đưa vào — đừng để model bật thứ vốn thuộc quyền user (1.3).

  3. Schema & structured output

    Khai inputSchema/outputSchema; trả structuredContent (JSON có kiểu) song song với text; lỗi nghiệp vụ báo bằng isError với message model tự-sửa-được, tách khỏi lỗi protocol (2.2).

  4. Annotation trung thực

    Khai readOnlyHint/destructiveHint đúng bản chất tool để client auto-approve hợp lý — và nhớ client vẫn coi annotation của server lạ là untrusted (2.3).

  5. Transport theo cảnh deploy

    stdio cho local, Streamable HTTP một-endpoint cho remote; server HTTP bắt buộc validate Origin để chặn DNS rebinding (3.1). Pin protocol version, đọc changelog trước khi nâng SDK (3.2).

  6. OAuth: là resource server

    Chỉ validate token đúng audience, publish Protected Resource Metadata; giao login/consent/phát token cho authorization server tách biệt — đừng tự chế lò phát token (4.1). stdio thì "SHOULD NOT" OAuth, lấy credential từ environment.

  7. Ba lớp phòng thủ

    KHÔNG passthrough token của client xuống downstream (đổi token riêng đúng audience) · hash-pin tools/list để mọi delta buộc re-approve (chặn rug pull) · per-client consent + exact-match redirect URI (chặn confused deputy) (4.2 · 4.3).

  8. Description & granularity

    Description là prompt — viết như onboarding người mới, đặt tên có namespace, gộp tool theo workflow agent hay đi chung thay vì wrap 1:1 REST endpoint (5.1).

  9. Ngân sách context

    Nếu server phình nhiều tool: lộ tool dần (progressive disclosure) và để agent viết code giữ kết quả trung gian ngoài context trước khi tối ưu prompt (5.2).

Takeaway

Một trang mang đi không phải để đọc lại — để hành động: 16 dòng distill, một checklist chạy tay theo thứ tự, một bản đồ host·client·server ba lớp. Dán nó cạnh màn hình và đi từ trong ra ngoài — primitive đúng vai, schema, bảo mật, thiết kế, rồi hệ sinh thái — không bỏ lớp nào.

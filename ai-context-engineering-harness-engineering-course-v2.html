Three Disciplines, One Stack

Prompt engineering is what you tell the model. Context engineering is what the model can see. Harness engineering is the world the model lives in. Conflating them is the most common reason AI features fail in production.

Why this distinction is load-bearing

If your team does not have a shared mental model for these three layers, every AI bug becomes a debate about prompts. Engineers will rewrite system prompts when the actual problem is a tool returning 30k tokens of garbage, or a retrieval pipeline grabbing the wrong documents, or a loop that never terminates. The fix lives at a different layer than the fix being attempted.

stack · bottom-up dependency
Prompt Engineering message-level · phrasing, examples, output format
Context Engineering window-level · what enters, in what order, at what cost
Harness Engineering runtime-level · tools, loops, memory, recovery, observability

Definition by scope

Prompt engineering — the message

The craft of writing a single message well. Includes role framing, constraint articulation, output format specification, in-context examples, and tone calibration. This is sentence-level work. It matters, but it has a ceiling — the best prompt in the world will still fail if you hand it the wrong context or run it inside a broken loop.

Context engineering — the window

The discipline of curating everything the model sees on a single forward pass. System prompt, tool definitions, retrieved documents, conversation history, in-context examples, file attachments, scratchpad notes. The job is to find the smallest set of high-signal tokens that maximize the likelihood of the desired outcome. The constraint is real: even with 200k+ context windows, signal-to-noise governs quality more than total volume does.

Harness engineering — the runtime

The orchestration around the model: tool implementations, the agentic loop, memory persistence, error handling, retry logic, output validation, subagent spawning, cost ceilings, observability. A good harness lets a mediocre prompt succeed; a bad harness makes a perfect prompt fail in production. The harness is where AI features become AI products.

Web analogy: Prompt = HTML. Context = the entire HTTP request/response (headers, cookies, body, query params, server-rendered data). Harness = the framework + infrastructure (Next.js, BullMQ, Redis, error boundaries, deployment). You can write perfect HTML in a broken framework and the page still won't render.

The diagnosis hierarchy

When an AI feature works in dev but fails in prod, root-cause is overwhelmingly in this priority order. Walk top-down, not bottom-up.

#LayerSymptomsWhat to inspect
1 harness Random failures, runaway costs, infinite loops, leaked state across users, silent truncation, missing retries. Logs of full message history. Loop iteration counts. Tool error rates. Token-budget metrics. Per-user state isolation.
2 context Hallucinated facts, "ignored" instructions, wrong document cited, output drift over long sessions. What was actually retrieved? What's the message history at failure time? What tools were defined? Are examples representative of real distribution?
3 prompt Genuine prompt issues: ambiguity in instruction, conflicting rules, output format never specified, tone mismatch. Read the prompt aloud. Would a junior engineer reading it once, with the same context, do what you want? If not, fix the wording.

The reflex to "tweak the prompt first" exists because prompts are visible and editable. Harness and context bugs require reading logs, instrumenting code, and thinking about state — uncomfortable work. Most teams optimize for comfort and stay broken longer.

The team-level implication

For your colleagues: when reviewing an AI-feature PR, the reviewer's first question should not be "is the prompt good?" — it should be:

  1. What does the harness do when [tool fails / model overloaded / output malformed / loop runs forever / cost spikes]?
  2. What's actually in the context window for typical and worst-case inputs? How many tokens? In what order?
  3. Is the prompt clear? Does it match the examples and tools provided?

If the first two have no satisfying answers, the third doesn't matter yet.

Exercise · Audit one of your existing AI features

Pick any AI feature in your stack — even a simple one. Spend 20 minutes documenting:

  1. The full harness: where the call originates, retry policy, error handling, cost ceiling, what happens on partial failure, where logs go.
  2. The full context for one representative call: system prompt size, tool registry size, history accumulation policy, retrieved-data size, total tokens, position of each.
  3. The prompt: read it as if you were a new dev seeing it for the first time. Note ambiguities.

Now rank the three by "amount of careful engineering invested." For most teams, it's inverted — heavy on prompt, light on harness. Make this visible to your team.

A teammate's RAG chatbot returns hallucinated product specs in production but works perfectly with their test queries. Where do you look first, and why?
Correct: C. "Works in dev, breaks in prod" with RAG is almost always a retrieval distribution mismatch. Test queries hit cleanly chunked seed data; production queries hit messy real data with off-topic neighbors retrieved or stale chunks dominating top-k. Context Theory · Attention, Rot, Anatomy

Bigger context windows are not a free lunch. The math that governs why is non-negotiable, and your engineering must account for it.

The "attention budget" — Anthropic's framing

Anthropic's engineering team frames context as a finite resource with diminishing marginal returns. The mechanism: a transformer attends every token to every other token. For n tokens that is n² relationships. As n grows, attention dilutes — the model has the same total "attention budget" stretched thinner across more material.

The empirical consequence is now well-documented and goes by the term context rot: as token count rises, the model's ability to accurately recall and use specific information decreases. Even when the relevant fact is technically in context, the model may miss it, drift from it, or confuse it with noise nearby.

Read this carefully because it is counterintuitive: more context can lower quality. Stuffing 50 RAG results when 5 would do is not "being safe" — it is actively degrading the answer. This is the opposite of how dev teams instinctively reach for "give the model more information so it has what it needs."

Position bias — "lost in the middle"

Position within the window is not neutral. Models attend disproportionately to the start (the system prompt area) and the end (the most recent turn). Tokens deep in the middle of a long conversation are statistically the most likely to be functionally ignored.

Practical consequence: a 50k retrieved document inserted into the middle of a long agent conversation is where information goes to die. The model can be holding the answer in context and still fail to use it.

The full anatomy — what actually fills 200k tokens

Every model call assembles a single linear sequence. Understanding what each segment costs in tokens — and how it interacts with caching — is the foundation of every other context engineering technique you will learn.

typical agent call · 200k window
System Prompt ~500–4,000 tk · role, rules, output schema · stable, cacheable
Tool Definitions ~200–10,000 tk · names, descriptions, JSON schemas · cacheable
In-context Examples (few-shot) ~500–10,000 tk · demonstrations · stable, cacheable
Working Memory · Scratchpad / Plan / Notes ~0–20,000 tk · agent's externalized thinking
Retrieved Documents (RAG, file attachments) ~0–50,000 tk · varies wildly · the danger zone
Conversation History ~0–100,000 tk · prior turns + tool results · grows monotonically without compaction
Current User Message ~10–5,000 tk · the actual query · highest attention weight
[Assistant turn — being generated] budget: max_tokens setting · output is also part of next turn's history

Token economics — what actually costs what

Input vs output pricing

For Claude Sonnet 4.x and GPT-5 family, input tokens are roughly 4–10x cheaper than output. So stuffing context looks cheap — until you account for the second-order costs:

  • Latency tax — more input = longer time-to-first-token. A 100k context can add several seconds before the model starts responding. For agent loops, this compounds across iterations.
  • Quality tax — every irrelevant token degrades attention on relevant ones (the context rot effect).
  • Cache invalidation tax — more on this below.

Prompt caching — the lever you must understand

Both Anthropic and OpenAI offer prompt caching: if a prefix of your context is identical to a prefix you sent recently (typically within minutes), the model skips re-processing that prefix and charges roughly 10% of normal input cost for those cached tokens. Latency drops dramatically too.

The catch: caching is prefix-based. The cache hits up to the first byte that differs. If you reorder your system prompt, the entire downstream is re-billed at full rate.

Cache-friendly ordering, in priority of stability:

  1. Most stable first: system prompt, tool definitions, large reference documents (almost never change).
  2. Stable per-session: project context, user persona, working memory pointers.
  3. Variable suffix: conversation history, current user message.

If you have a 3,000-token system prompt that changes every request because you templated in a timestamp, you have zero cache benefit. Move dynamic data to the suffix.

typescript · vercel ai sdk · cache-friendly ordering 
import { generateText } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';

const result = await generateText({
  model: anthropic('claude-sonnet-4-6'),
  
  // 1. SYSTEM — most stable. Cache aggressively.
  // Note the ephemeral cacheControl marker — Anthropic's prompt caching API.
  system: [
    {
      type: 'text',
      text: SYSTEM_PROMPT_STABLE,
      providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } }
    }
  ],
  
  // 2. TOOLS — also stable. The registry rarely changes within a session.
  tools: TOOL_REGISTRY,
  
  // 3. MESSAGES — the variable suffix.
  // Recent history + current user message. Cache hit will land here.
  messages: [
    ...conversationHistory,
    { role: 'user', content: userMessage }
  ],
});

// Inspect cache stats — your dashboard should track cache_creation
// and cache_read tokens to verify caching is actually working.
console.log(result.providerMetadata?.anthropic);

The hidden tax — tool results in history

The single most underestimated cost in production agent systems. Every tool call's result becomes part of the conversation history for subsequent turns. Forever. Until you compact or evict.

A view on a 5,000-line log file just dropped 30k tokens into your conversation. A list_files over a large repo dropped another 5k. A web_search for context dropped 8k. After ten such tool calls, the conversation has 100k+ tokens of stale tool output competing for attention with the actual task.

This is why long Claude Code or Codex sessions feel like they degrade. They literally do. The model is searching through layers of stale tool outputs to find current relevance. The fix is not "model is tired" — the fix is /compact, /clear, structured note-taking (covered in Module 3), or starting a fresh session with a brief.

Tool registry as permanent overhead

Every tool you register has its name, description, and full JSON schema injected into context on every call. The model sees this even when not using any tools. A bloated registry quietly steals tokens from every conversation.

▸ Bloated tool def · ~200 tokens
{
  name: "search_database",
  description: "This tool allows you to search the database. It is very helpful when you need to find information. Use it when you want to look something up. The database contains many records. Each record has fields. You should use this tool whenever appropriate. Be careful to construct your query correctly. Make sure to use proper SQL syntax and follow the conventions of our system...",
  parameters: { /* loose schema */ }
}

Vague description = wrong calls. Long description = permanent token tax. Loose schema = invalid calls slip through.

▸ Tight tool def · ~60 tokens
{
  name: "search_customers",
  description: "Find customers by email, phone, or order_id. Returns up to 10 matches.",
  parameters: {
    type: "object",
    properties: {
      field: { enum: ["email","phone","order_id"] },
      value: { type: "string", minLength: 2 }
    },
    required: ["field","value"]
  }
}

Sharp purpose. Tight description. Strict schema. Fewer wrong calls, less context tax.

The 20k MCP rule (community heuristic): if the total token cost of your registered tools exceeds 20k, you have crippled your agent. With 200k context you've now got only ~180k for everything else, and remember every other system component is also hungry. Audit with /mcp in Claude Code or equivalent in Codex.

The full equation

For any AI feature, the engineering question is not "how do I write a good prompt?" It is:

What configuration of context is most likely to produce the desired behavior, given that every token competes for finite attention, position matters, and history accumulates?

This single question reframes most of your daily decisions: which tools to register, what goes in CLAUDE.md vs a skill, when to compact, when to spawn a subagent, when to externalize state. The rest of Part I is techniques for answering it.

Context Engineering Patterns

Six recurring problems, six battle-tested patterns. This is Anthropic's playbook plus production hard-earned additions. Internalize these and you can architect context for any AI feature.

Pattern 1 · Just-in-time retrieval

The most important shift in agent design over the last year. Old thinking: pre-load everything the agent might need into context. New thinking: give the agent tools to fetch what it needs, when it needs it.

▸ Ahead-of-time loading
// Load 50 customer records
// + 200 product specs  
// + 30 days of orders
// into context "just in case"

// 80k tokens consumed before
// the model reads the question.

// Most of it irrelevant to
// any single query. All of it
// competing for attention.

Wasteful. Slow. Quality-degrading. The model spends attention budget on data the user did not ask about.

▸ Just-in-time
// Tools provided:
//   get_customer(id)
//   search_customers(q)
//   get_product(sku)
//   list_orders(filters)

// Model decides which to call
// based on the actual query.

// 5–15k tokens used per task.
// Tools can be re-called for
// verification or new branches.

The agent explores its environment instead of receiving a pre-digested briefing.

This pattern requires you to invest in tool design (Module 4) but the payoff is dramatic: smaller, sharper, faster, cheaper, more accurate.

Concrete win: replacing pre-loaded RAG dumps with on-demand search tools typically cuts per-query input cost by 60–80% while improving accuracy. The agent retrieves what's relevant; the agent ignores what's not. The agent can re-query when it realizes it needs more.

Pattern 2 · Compaction (recursive summarization)

When conversation history grows past useful, summarize older turns into a single condensed turn and replace the originals. Claude Code's /compact implements this. Codex has equivalent session compaction. You should also implement it in your own agentic systems.

Before · 95k tokens

System (pinned)2k

Critical compaction rules:

  • Never evict head or tail. System prompt + the original user goal stay pinned forever. Recent turns stay verbatim. Only middle gets summarized.
  • Summarize tool outputs aggressively. A 5k tool result becomes [file content elided · 1240 lines · path: src/foo.ts]. The reference is preserved; the noise is gone.
  • Preserve decisions, not deliberation. "Decided to use Drizzle, rejected Prisma due to bundle size" — keep. The 2,000 tokens of debate that led there — discard.
  • Compaction is itself a model call. It costs tokens. Triggering it too often is wasteful; triggering too late means you are running rotted context. Many teams trigger at 50–60% of the window.

Pattern 3 · Structured note-taking (working memory)

Anthropic's research highlights this as a long-horizon task technique. The idea: have the agent maintain an external notes file (a scratchpad, a plan file, a todo list) that persists outside the context window. The agent reads from and writes to this file during work.

Why this works: the file is the state of the work. The conversation can be compacted aggressively because the durable knowledge is on disk. A fresh session can resume by reading the notes file.

Concrete implementations:

  • Claude Code's plan.md / tasks.md pattern — the agent writes a multi-step plan to a markdown file, executes step-by-step, ticks off completed items, leaves notes for future steps.
  • Codex CLI working files — similar pattern, often in .codex/scratch/ or a dedicated workspace dir.
  • Custom: dedicated "context tool" — for production agents, expose read_notes(key) / write_notes(key, value) tools backed by Redis or a DB.

The file is the agent's externalized memory. Once you internalize this, you stop trying to cram everything into context and start treating the conversation as a working surface that can be reset.

Pattern 4 · Sub-agent isolation

Spawn a fresh agent context for a focused task. The parent gets back only the final summary; the parent's context is never polluted by the subagent's exploration.

This is Anthropic's recommended approach for long-horizon tasks. It maps cleanly onto how human teams work: the lead doesn't watch every keystroke of every IC, they get briefed.

parent agent
→
spawn subagent w/ brief
→
[subagent: 50 tool calls, 80k tokens internal]
→
return summary (200 tokens)
→
parent continues clean

Use cases that map perfectly to subagents:

  • Research / exploration — "find every place we use the deprecated API." Subagent does 30 searches, returns a list.
  • Risky or uncertain work — "try a refactor approach; if it fails, the parent context isn't polluted by the failed attempt."
  • Specialist tasks — DB migrations, security review, performance profiling. Each spawns a subagent with its own system prompt and minimal tool set.
  • Parallel work — fan out three subagents to investigate three competing hypotheses, gather their reports, synthesize.

Pattern 5 · Structured outputs (the safety net)

Free-form text outputs are unparseable garbage some percentage of the time. That percentage breaks production. Force structure at the API layer.

In order of preference:

  1. Tool-call as output — define a tool whose only job is "submit the answer." The model must call it; the API validates the schema.
  2. SDK helpers — Vercel AI SDK's generateObject with Zod, OpenAI SDK's response_format: { type: "json_schema" }. Same idea, abstracted.
  3. JSON mode + validate + retry — fallback when the above aren't available. Catch the parse failure and re-prompt with the specific error.
typescript · vercel ai sdk · structured output 
import { generateObject } from 'ai';
import { z } from 'zod';

const PostAnalysis = z.object({
  sentiment: z.enum(['positive', 'negative', 'neutral', 'mixed']),
  topics: z.array(z.string()).max(5),
  brand_mentions: z.array(z.object({
    brand: z.string(),
    context: z.enum(['praise', 'complaint', 'neutral', 'comparison'])
  })),
  is_kol_post: z.boolean(),
  language: z.enum(['vi', 'en', 'mixed'])
});

const { object } = await generateObject({
  model: anthropic('claude-sonnet-4-6'),
  schema: PostAnalysis,
  prompt: `Analyze this TikTok post: "${postText}"`
});

// `object` is type-safe and schema-validated.
// No JSON.parse, no try/catch on malformed output, no regex hacks.
// If the model hallucinates a field, the SDK auto-retries with
// the schema error. If it can't, you get a typed exception.

Always validate. Always type. Never JSON.parse raw model output and pray. This is non-negotiable for production. Code that does so is a future incident.

Pattern 6 · Few-shot via in-context examples

Instructions tell the model what to do. Examples show it. For output formatting, edge case handling, and tone, examples beat instructions every time. The catch: bad examples poison harder than no examples.

few-shot via fake assistant turns 
// Inject examples as fake user/assistant pairs. The model
// pattern-matches the demonstration and replicates the form.
messages: [
  { role: 'user', content: 'Classify intent: "Where can I track my order?"' },
  { role: 'assistant', content: '{"intent":"order_tracking","confidence":0.95}' },
  
  { role: 'user', content: 'Classify intent: "lol nice product"' },
  { role: 'assistant', content: '{"intent":"feedback_positive","confidence":0.7}' },
  
  { role: 'user', content: 'Classify intent: "đơn hàng của tôi đâu rồi?"' }
  // real query — model will follow the demonstrated format
]

Rules for examples:

  • Cover edge cases the model would otherwise mishandle (the rare cases, the multilingual ones, the malformed ones).
  • Demonstrate refusal/uncertainty as well — "if unclear, output X" should have an example showing X.
  • Audit examples regularly. If real distribution drifts, your examples become misleading.
  • Cache them. Examples are stable; they belong in the cacheable prefix.

Anti-pattern · Context poisoning

Once the model says something wrong and that wrong statement is in history, it tends to defend it on subsequent turns. Models are consistent to a fault. This is a real failure mode in long agent loops.

Mitigations:

  • Validate intermediate outputs before they enter history. If the agent claims it deployed something, verify it actually deployed before letting that claim live in context.
  • On detected error, rewrite or remove the bad turn. Do not just append "no actually, that was wrong" — the false claim is still in context.
  • For long loops, periodically reset to a clean state with a fresh summary. Sub-agents are excellent for this — the parent never sees the polluted middle.
Exercise · Replace pre-loading with just-in-time

In your codebase, find one feature that pre-loads data into a prompt (often: customer info, product catalog, recent orders). Convert it:

  1. Define 2–3 sharp tools (get_customer, search_products, etc.) instead of bulk-loading.
  2. Remove the pre-load. Update the system prompt to mention the tools.
  3. Compare on a sample of 20 real queries: token usage, latency, accuracy.

Most teams see 60%+ token reduction with equal or better quality. The agent only fetches what it actually needs.

You're building an AI assistant for a social listening dashboard with months of data. Users ask everything from "how many mentions of brand X last week" to "what was the general sentiment about the new product launch". Which context strategy?
Correct: C. Most dashboard questions are aggregation queries that semantic search actively hurts (cosine similarity won't tell you a count). Some are genuinely semantic. Route intent first, then choose the retriever appropriate to that intent. Use just-in-time tools for both paths so the context stays lean. This is the architecture you want behind any "AI dashboard" feature, and it generalizes — same pattern works for support tickets, product analytics, document search.

Harness Engineering · The Runtime

Five pillars that turn a model into a product. The harness is your IP. The model is interchangeable.

The five pillars

harness · pillars and concerns
1 · TOOLS the model's hands · read and write the world
2 · LOOP decide when to call tools, when to stop, when to give up
3 · MEMORY what persists across calls, sessions, users
4 · RECOVERY retry, fallback, validation, error containment
5 · OBSERVABILITY log everything, measure quality, evals as regression tests

Pillar 1 · Tools as API design

Designing tools is API design. The consumer is a non-deterministic agent reading your description once and deciding what to call. Apply API design principles, raised in stakes.

Rule: small surface, sharp purpose

Each tool does one thing. search_customers_by_email beats universal_search. The model picks tools by name + description; ambiguity at this layer is unrecoverable downstream.

Rule: prefer few powerful tools over many narrow ones

This is the "gateway pattern" community wisdom: rather than mirroring your REST API as 30 tools, expose 3–5 powerful gateways with rich parameters. download_data(filters...), execute_in_sandbox(code), take_gated_action(args...). Composability via parameters beats proliferation.

Rule: idempotent where possible

Models retry. If send_email isn't idempotent, the customer gets three emails. Either make tools idempotent or have the harness deduplicate.

Rule: bounded outputs

A list_files that returns 10,000 entries just exploded your context. Always paginate, truncate, or summarize tool results before returning. A tool returning a 30k-token blob is a tool designed to ruin its caller.

Rule: rich, actionable error messages

When a tool fails, the error goes into the model's context. Error: 500 is useless. Error: customer_id 'cus_abc' not found. Did you mean to search by email instead? teaches the model how to recover. Treat error messages as part of your tool's API contract.

typescript · production-grade tool 
import { tool } from 'ai';
import { z } from 'zod';

const searchPosts = tool({
  // SHARP description. Tells the model when to use, when not to.
  description:
    'Search social posts by keyword. Returns up to 20 most recent matches with engagement metrics. Use for content/sentiment questions, NOT for aggregate counts (use count_posts instead).',
  
  // STRICT schema. Bounds + enums prevent invalid calls.
  inputSchema: z.object({
    keyword: z.string().min(2).max(100),
    days_back: z.number().int().min(1).max(90).default(7),
    min_views: z.number().int().nonnegative().default(0)
  }),
  
  execute: async ({ keyword, days_back, min_views }, ctx) => {
    try {
      const rows = await db.searchPosts({
        keyword, days_back, min_views,
        tenantId: ctx.tenantId  // auth from harness, NEVER from model
      });
      
      if (rows.length === 0) {
        // EMPTY result with guidance — model can adapt
        return {
          status: 'empty',
          message: `No posts for "${keyword}" in last ${days_back}d. Try broader keyword or longer window.`
        };
      }
      
      // BOUNDED output — never return more than 20, only key fields,
      // truncate text. Full text fetched on demand via separate tool.
      return {
        status: 'ok',
        count: rows.length,
        truncated: rows.length === 20,
        posts: rows.slice(0, 20).map(p => ({
          id: p.id,
          author: p.author,
          views: p.views,
          excerpt: p.text.slice(0, 200)
        }))
      };
    } catch (err) {
      // RICH error — model can act on it, not just fail
      return {
        status: 'error',
        message: `Search failed: ${err.message}. Retry with simpler keyword or check db status via /health.`
      };
    }
  }
});

Pillar 2 · The agentic loop

An agentic loop is conceptually simple: call model → if response has tool_use, execute tool, append result → call model again → repeat until response is text-only or budget hit. The complexity is in the bounds.

typescript · production loop with all bounds 
async function runAgent(initialMessages, tools, options) {
  let messages = [...initialMessages];
  let iter = 0;
  let totalCost = 0;
  const seenToolCalls = new Set();  // detect repeats
  
  const MAX_ITER = options.maxIter ?? 25;
  const MAX_COST_USD = options.maxCost ?? 2.0;
  const MAX_TOKENS = options.tokenBudget ?? 150000;
  
  while (iter++ < MAX_ITER) {
    // BOUND 1 · token budget — compact if approaching limit
    if (estimateTokens(messages) > MAX_TOKENS) {
      messages = await compact(messages);
    }
    
    // BOUND 2 · cost ceiling — kill switch
    if (totalCost > MAX_COST_USD) {
      throw new CostExceededError(totalCost);
    }
    
    const resp = await model({ messages, tools });
    totalCost += resp.usage.totalCost;
    messages.push(resp.message);
    
    // Termination: no tool calls = done
    if (!resp.toolCalls?.length) return { text: resp.text, totalCost, iter };
    
    // BOUND 3 · detect circular calls (same tool, same args twice)
    for (const tc of resp.toolCalls) {
      const key = `${tc.toolName}:${JSON.stringify(tc.args)}`;
      if (seenToolCalls.has(key)) {
        messages.push({
          role: 'tool',
          content: [{ toolCallId: tc.id, error: 'Duplicate call detected. Try a different approach or ask user for guidance.' }]
        });
        continue;
      }
      seenToolCalls.add(key);
    }
    
    // Execute (parallel where independent, with per-tool timeout)
    const results = await Promise.all(
      resp.toolCalls.map(tc => 
        withTimeout(executeTool(tc, tools), 30_000)
          .catch(err => ({ toolCallId: tc.id, error: err.message }))
      )
    );
    messages.push({ role: 'tool', content: results });
  }
  
  // BOUND 4 · max iterations — don't loop forever
  throw new Error(`Agent exceeded ${MAX_ITER} iterations. Likely stuck.`);
}

Three loops kill agents in production:

  1. Infinite tool retry — same tool fails the same way repeatedly. Detect duplicate calls.
  2. Circular reasoning — model alternates between two states ("research → reconsider → research → reconsider"). Add iteration cap.
  3. Over-exploration — model "double-checking" forever. Cost ceiling forces termination.

Always set bounds. Always log full message history for debugging. Agents that fail silently waste days; agents with good logs fail in minutes.

Pillar 3 · Memory architecture

Three tiers, often confused. Each has a job; using the wrong one causes specific bugs.

TierWhat it isLifetimeUse forAnti-use

Pillar 4 · Recovery — failure modes ranked

#Failure modeRemediation

The "trust the LLM with auth" anti-pattern: a tool exposes delete_user(user_id), the model decides who to delete based on the conversation. There is no real authorization layer — the model's interpretation of the user's intent is the auth check.

The harness, not the model, owns authorization. Tools take an actor identity from session/JWT, not from the model's argument. The model can request actions; the harness validates against permissions before executing.

Pillar 5 · Observability — log everything

Production AI without logging is impossible to debug. The minimum viable observability for an agent system:

  • Every model call: full input messages (with PII redaction), full output, model name, latency, prompt cache hit rate, input/output token counts, cost.
  • Every tool call: tool name, args (redacted), result size, latency, error if any.
  • Every loop: total iterations, total cost, total time, terminal state (success/timeout/cost-exceeded/error).
  • Per-user metrics: cost per user per day, latency p50/p95/p99, success rate.

Tools that exist for this: Langfuse, Helicone, Braintrust, LangSmith. Even structured JSON to stdout with trace_id beats nothing. If you can't reconstruct exactly what happened in a failed agent run from logs alone, your observability is broken.

DELEGATION · Skills — just-in-time domain knowledge

A Skill is a folder with SKILL.md + optional supporting files. Claude reads SKILL.md only when the task triggers it (description matches). This is just-in-time context done right: zero token cost until needed.

The trigger description is the most important line

DELEGATION · Subagents — context isolation as a feature

A subagent is a fresh Claude session spawned with its own context, system prompt, and tool subset. Parent agent gets back only the final result. This is THE pattern for tasks that would otherwise pollute main context.

When to spawn:

  • Search and research — main says "find every usage of X" — subagent does 50 grep/view calls and returns a 200-token summary
  • Risky or sandboxed work — let a subagent attempt a destructive refactor; if it fails, main context is unpolluted
  • Specialist tasks — DB tools only, security-review-only, performance-only
  • Parallelizable independent work — fan out subagents for competing hypotheses

EXTENSION · Hooks — programmable harness behavior

Hooks are scripts that fire on Claude Code lifecycle events. They run on your machine; their output (or block decision) gets fed back into the harness. This is how you enforce policies the model can't be trusted to follow.

Real-world hooks worth writing:

  • PreToolUse on Bash — block any command containing rm -rf /, DROP TABLE, --force against prod, git push --force on protected branches
  • PostToolUse on Edit — auto-run biome format on edited files; auto-run pnpm typecheck on .ts changes and feed errors back to Claude
  • Stop hook — when Claude says "done," run typecheck + tests; if errors, append result to context so Claude sees and fixes
  • UserPromptSubmit — inject current git branch, current task ID from Linear/Jira, current sprint context automatically
  • SessionStart — print or inject the most recent compact summary so resumed sessions have continuity

// .claude/hooks/safety-check.sh — blocks dangerous commands
#!/usr/bin/env bash
set -e
INPUT=$(cat)  # JSON of the proposed tool call
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# Patterns that should never run, even with approval
BLOCKED=(
  "rm -rf /"
  "rm -rf ~"
  "DROP DATABASE"
  "DROP TABLE"
  "git push --force origin main"
  "git push --force origin master"
  "--force-with-lease origin main"
)

for pattern in "${BLOCKED[@]}"; do
  if [[ "$COMMAND" == *"$pattern"* ]]; then
    echo '{"decision":"block","reason":"Command matches dangerous pattern: '"$pattern"'. Refusing for safety."}'
    exit 0
  fi
done

# Pass through
echo '{"decision":"allow"}'

Hooks turn Claude Code from "AI that asks permission" to "AI that operates within deterministic safety rails." The model can hallucinate; the hook cannot. Use hooks for anything where you would lose sleep if the model got it wrong.

EXTENSION · MCP — extending the tool registry

Model Context Protocol lets you plug external tools into Claude Code. For your team this means: a project-specific MCP server exposing tools like get_tenant_data, query_metrics, list_campaigns.

Same discipline as in-process tools (Module 4): tight descriptions, bounded outputs, idempotent, rich errors. The only difference is the transport (stdio or HTTP).

The 20k-token MCP rule: if your registered MCP tools' total token cost exceeds 20k, you have crippled your agent. Run /mcp to audit. Disconnect MCPs not relevant to the current project. Project-scope your MCP enablement aggressively.

CORE · Plan mode and /compact discipline

Two of the most underused features:

Plan mode

Before executing complex changes, Claude generates a plan and waits for your approval. Critically, plan mode runs with read-only tools by default — Claude can investigate but not modify. You see the plan, approve or refine, then execute.

Use plan mode for: any change touching 3+ files, any architectural decision, any work where "I'm not sure exactly what we want yet." Skip it for: small obvious tweaks where planning is overhead.

/compact discipline

Claude Code's /compact summarizes the conversation and resets to a much smaller context. The community heuristic: trigger at 60k tokens or when you notice quality degrading.

Better: /compact with a specific instruction. /compact focus on the API design decisions and the bug we identified, drop the file exploration. The summary is biased toward what you tell it to keep.

Agent Teams (experimental, v2.1.32+)

Multiple Claude instances coordinate on a shared task list. One acts as team lead, others as specialists. Best for: parallel exploration where teammates can operate independently. Token costs scale linearly with team size; coordination overhead grows non-linearly.

Practical sizing: 3–5 teammates. Beyond that, diminishing returns set in fast and you spend more on coordination than work.

Enable via CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. Requires Opus 4.6+ and v2.1.32+.

Sandbox modes · safe-by-default execution

Codex runs commands inside OS-level sandboxes by default. macOS uses seatbelt; Linux uses bubblewrap; Windows uses native sandbox or proxy-only egress rules. This is a meaningful safety advantage over running an agent with full shell access.

Sandbox modes:

  • --sandbox read-only — agent can read your files, can't write or execute
  • --sandbox workspace-write — agent can read anywhere, write only inside the current workspace. The default for most work.
  • --full-auto — shortcut: workspace-write sandbox + on-request approval. The "low-friction local work" mode.
  • --yolo / --dangerously-bypass-approvals-and-sandbox — no sandbox, no approvals. Only inside externally hardened environments (containers, VMs).

The --yolo flag is exactly as dangerous as it sounds. Practitioners who use it routinely run Codex inside Docker containers so the blast radius is contained. If you don't have container isolation, don't yolo.

The auto-reviewer · a built-in quality gate

Codex can route eligible approval prompts through an automatic reviewer agent before the request runs. The reviewer assesses risk and either approves, denies, or stops the action — and the outcome shows up in the Codex App as an automatic review item with status and risk level.

This is a structural advantage: the same configuration that lets Codex execute also gates execution through a second agent's judgment. You can configure the reviewer to be more or less strict based on environment.

The cost reality · parallel agents burn rate limits in parallel

Three agents running in parallel burn through your rate limits roughly 3x as fast. Two practical implications:

  • For Claude Code: Sonnet 4.6+ on the standard plan can hit limits faster than you expect with parallel use. Either upgrade or stagger work.
  • For Codex: ChatGPT plan limits apply across surfaces. Cloud uses your quota too.
  • Cost discipline: cheap models for scout work and subagents (codex-mini, Haiku); expensive models for the work that needs depth (Opus, GPT-5.5).

The risk reality · integration / merge / safety

Parallel work creates integration risk. Each agent's output may be locally correct but globally incompatible.

Mitigations:

  • Validation step — after parallel agents complete, run an explicit integration test or have a "validator" agent review combined output before merge
  • Scope agents narrowly — each agent owns disjoint files. Check the spec; if two agents would touch the same file, sequence them, don't parallelize
  • Builder-validator chain — quality check step runs after builders finish, catches interface mismatches before they hit production
  • Run risky work in containers — Codex Cloud already does this; for local YOLO mode, run inside Docker

For your team, frame it as a workflow choice menu, not a tool war:

  1. Default: pick whichever tool you're more fluent in. Both are excellent.

  3. Specialize by task: Codex Cloud for clear-scope async, Claude Code for sync deep work, the other tool for second-opinion review

Discovery & Spec · Where AI Earns Its Keep First

The phase where bad work becomes 10x more expensive downstream. Also the phase where AI's compounding leverage is most underexploited.

The two failure modes of spec work

Specs fail in two characteristic ways:

  1. Too vague — engineers fill in details with assumptions, half of which are wrong. Discovered at code review or worse, in production.
  2. Too prescriptive too early — locks in an approach before alternatives are explored. The team builds the second-best solution because the spec said so.

AI helps with both, but only if you use it before writing the spec, not after.

Pattern 1 · Brainstorm before scope

Use a Claude Code session in plan mode (or a Codex CLI exploration session) to brainstorm approaches before committing to one. The model is excellent at generating options space, then helping you compare tradeoffs.

prompt · brainstorm before spec 
// Open Claude Code in plan mode (Shift+Tab to toggle)

I need to add real-time notifications to our dashboard. Before we
write a spec, brainstorm 3-4 architectural approaches with very
different tradeoffs. For each, list:

1. Implementation summary (3-5 sentences)
2. Cost profile (infra + dev time)
3. Failure modes
4. When this approach is wrong
5. What it locks us out of

Then recommend which to spec out first based on our stack
(read CLAUDE.md), team size, and the fact that we need this
shipped in 3 weeks.

Don't make the recommendation until you've laid out all options.

The discipline here is options first, recommendation last. If you ask the model "design this" without options exploration, you get the first plausible answer — and miss the better one. Forcing options-space generation is one of the highest-ROI prompting moves in spec work.

Pattern 2 · Send a scout to map unknowns

Before scoping a change in unfamiliar code (yours or a teammate's), send a scout. The scout's deliverable is not code — it's a map.

scout prompt · for unfamiliar codebase work 
Investigate this question — do not implement anything:

If we want to add per-tenant rate limiting to the API, where would
that change need to live? Specifically:

1. List every file that would need to be modified or added
2. Identify any existing middleware/guard patterns to follow
3. Find any tests that would need to update
4. Note any "gotchas" — places where assumptions about
   global rate limiting are baked in
5. Estimate the change in lines of code and risk level

Output a markdown report. Do not modify any files.

Codex Cloud is excellent for scouts because they often involve broad codebase exploration that bloats local context. Fire from your phone while in a meeting; come back to a written report you can use to scope the spec.

Pattern 3 · PRD as living document, AI as co-author

For substantial features, draft the PRD as a markdown file in the repo (often docs/specs/feature-name.md) and use AI to iterate on it. Both Claude Code and Codex can do this; Claude Code has the edge for long-form structured writing.

Key sections every PRD should have, AI helps with each:

  • Problem statement — AI surfaces unstated assumptions ("does this also need to handle X case?")
  • User flows — AI walks through and finds gaps
  • Success criteria — AI suggests measurable definitions, not vague ones
  • Non-goals — explicitly out of scope. Ask AI to suggest things that look in scope but should be out.
  • Risks & open questions — AI is excellent at surfacing these from related codebase context
  • Implementation outline — high level only, with explicit "decisions deferred to ADR" markers

Treat the PRD itself as code. It lives in the repo. It evolves through PRs. AI reads CLAUDE.md/AGENTS.md and the PRD together when you start implementation, so the PRD's quality directly affects code quality. Sloppy specs produce sloppy code, no matter the model.

Spec-driven development · the discipline

"Spec-driven" means: the spec is the source of truth. Code derives from it. When requirements change, you update the spec, then re-derive (or re-prompt) the code. The spec and code stay in sync because the spec is upstream.

For AI work this matters extra: the spec is also your prompt material. A clean PRD becomes high-signal context for both implementation and review prompts. A messy PRD pollutes everything downstream.

Anti-patterns to call out for the team

  • "Let me just have AI write the spec for me" — without input, AI generates plausible-sounding but generic specs. Specs require domain context the AI doesn't have.
  • Skipping the brainstorm — going straight from problem to "design this for me" closes the option space.
  • Asking for spec + code in the same session — context bloats, decisions get half-baked, you can't tell what's spec vs implementation detail.
  • Treating PRD as one-shot — good PRDs evolve. Plan to revise after first implementation pass surfaces hidden complexity.

Architecture & Design

System design with AI is best done as a structured dialogue with multiple specialist personas, anchored by Architecture Decision Records. The model is your design partner, not your designer.

Architecture as constrained search

System design is fundamentally a constrained search problem: given functional requirements + non-functional requirements (performance, cost, security, ops) + existing constraints (stack, team, deadlines), find an architecture that satisfies all of them well enough.

AI is excellent at the search part. Humans are essential at the "well enough" judgment. The workflow needs to play to both.

Pattern 1 · ADRs (Architecture Decision Records)

Every meaningful architectural decision becomes a markdown file in docs/adr/. Format is short and stable:

docs/adr/0014-postgres-listen-notify-vs-redis-pubsub.md 
# ADR-0014: Use Postgres LISTEN/NOTIFY for in-app real-time, not Redis pub/sub

## Status
Accepted · 2026-04-12 · Author: @conan

## Context
We need real-time notifications in the dashboard. Three options surfaced
in brainstorm:
- WebSocket server with Redis pub/sub fanout
- Postgres LISTEN/NOTIFY with thin SSE bridge
- Pusher / Ably (managed)

Constraints:
- Multi-tenant Postgres already deployed
- 50–500 concurrent users per tenant max (small scale)
- 3-week timeline
- Avoid new infra dependencies if possible

## Decision
Use Postgres LISTEN/NOTIFY with a thin Node SSE bridge.

## Consequences

Positive:
- Zero new infra (Redis would need ops attention)
- Strong ordering guarantees per tenant
- Notifications are atomic with DB transactions

Negative:
- Doesn't scale past ~10k concurrent (we're well under)
- LISTEN/NOTIFY payload size limit 8KB — must reference IDs, not embed data
- One node process holds the LISTEN — needs HA strategy if scaled out

Reversible: yes. Internal API stays same; we can swap to Redis later.

## Alternatives rejected
- Redis pub/sub: rejected for new infra cost vs scale need
- Pusher: rejected for vendor lock-in + cost at our user count

## How AI was used
- Claude Code generated the 3-option brainstorm
- Codex CLI scout reported existing SSE patterns in /server/realtime/
- Both reviewed final ADR for missed gotchas (caught the 8KB limit)

ADRs serve double duty: they document decisions for humans, AND they're high-signal context for future AI sessions. When AI works on related code months later, reading the ADRs gives it the reasoning behind current architecture, not just the current state. This is one of the highest-leverage docs you can keep.

TDD with AI · the right rhythm

Test-driven development pairs unusually well with AI. The reason: tests are specifications the model can verify against. Without them, the model declares "done" based on its own judgment, which is unreliable.

Red — Green — Refactor, AI flavor

  1. Red — describe the behavior. Ask AI to write failing tests that capture it. Run them, confirm they fail for the right reason.
  2. Green — ask AI to implement just enough to make the tests pass. Stop at green. Resist the temptation to over-build.
  3. Refactor — ask AI to clean up the implementation. Tests stay green. Often best done as a separate AI session for fresh perspective.

The single most underused trick: have the model run the tests itself, in the loop. With Claude Code's Bash tool or Codex's sandbox, the model writes code, runs tests, sees output, fixes, re-runs. This is one of the highest-leverage uses of an agent — it tightens the feedback loop from minutes to seconds.

Plan mode discipline

Plan mode in Claude Code is the implementation phase's most underused feature. The pattern:

  1. Toggle plan mode (Shift+Tab in Claude Code).
  2. Describe the change. The agent investigates with read-only tools, generates a plan.
  3. Read the plan critically. Edit it directly or ask the agent to refine.
  4. Approve. The agent executes — and now you mostly review diffs.

When to skip plan mode:

  • Single-file trivial change ("rename this variable")
  • Pure exploration where the destination is unknown
  • Bug fixes where the fix is obvious once the bug is found

When to insist on plan mode:

  • Any change touching 3+ files
  • Architectural changes
  • Anything with a deadline (the plan-then-execute structure prevents the model from going down rabbit holes)
  • Any code area you don't fully know yourself

Codex Cloud for backlog implementation

Codex Cloud earns its keep on the long tail of small, well-scoped implementation tasks. The workflow:

  1. Maintain a "ready for AI" label or list in your tracker — issues with clear acceptance criteria, narrow scope, low blast radius.
  2. Fire 3–5 to Codex Cloud at start of day. Each runs in its own sandbox.
  3. By midday, several PRs are ready for review. You review serially, often closing 5+ tickets in an hour.
  4. The ones that come back broken or wrong-direction get either rejected or refined and re-fired.

Throughput on this kind of work goes up dramatically — but only when you're disciplined about which tickets are appropriate. Anything requiring nuanced product judgment is not for Cloud; anything mechanical with clear specs is.

▸ AI as driver

Review & Test · Cross-Perspective Quality Gates

AI as code reviewer is a force multiplier — when set up right. Eval-driven development is mandatory for AI features themselves. This module is about both.

AI as code reviewer · the structural advantage

An AI reviewer doesn't get tired, doesn't have ego, doesn't skip files. It catches the boring-but-real issues — naming inconsistencies, missing tests, unhandled errors, unsafe SQL, leaked secrets, perf antipatterns — that human reviewers tend to scan past.

The point is not to replace human review. The point is to raise the floor: by the time a PR hits a human reviewer, it has already been screened for the mechanical issues. The human can focus on the judgment calls that need a human.

Pattern 1 · Codex's built-in code review

Codex CLI has a first-class code review feature: get your code reviewed by a separate Codex agent before you commit. Run it as a pre-commit step.

.git/hooks/pre-commit 
#!/usr/bin/env bash
# Pre-commit hook: run Codex review on staged changes

set -e

# Check if there are staged changes
if ! git diff --cached --quiet; then
  echo "Running Codex review on staged changes..."
  
  # Run codex exec with the review prompt, scoped to staged diff
  REVIEW=$(git diff --cached | codex exec \
    --ask-for-approval never \
    --sandbox read-only \
    "Review this diff for: bugs, security issues, perf problems, missing tests, convention violations. Output a concise report. If approved with no major issues, start with 'APPROVED'. Otherwise list issues with severity.")
  
  echo "$REVIEW"
  
  # Block commit if review didn't approve
  if [[ ! "$REVIEW" == APPROVED* ]]; then
    echo ""
    read -p "Codex flagged issues. Commit anyway? (y/N) " -n 1 -r
    echo
    if [[ ! $REPLY =~ ^[Yy]$ ]]; then
      echo "Commit aborted. Address issues or override with --no-verify."
      exit 1
    fi
  fi
fi

Pattern 3 · Builder-validator chain

Anthropic's pattern for parallel agent work generalizes to review: after builders finish, a dedicated validator agent reviews combined output. The validator's only job is to look for integration issues that no individual builder would see.

Practical setup for parallel-built PRs:

  • Each builder agent operates on its own scope (worktree)
  • After all builders complete, a validator runs against the integrated diff
  • Validator's prompt is specifically about cross-component concerns: interface mismatches, duplicated logic, conflicting assumptions
  • Validator's findings either trigger another builder pass or get raised to humans

Test generation · what AI is great at, and what it's not

▸ AI excels at
  • Boilerplate scaffolding (mocks, fixtures, setup)
  • Edge-case enumeration ("what could go wrong here?")
  • Snapshot tests for stable output
  • Property-based test cases
  • Test descriptions and naming
  • Filling in missing coverage on existing code
▸ AI struggles with
  • Tests that require deep domain insight to know what's "correct"
  • Integration tests that require real environment setup
  • Tests that need to verify side effects across systems
  • Performance/load tests with realistic distributions
  • Tests of randomized or stateful behavior without explicit seeds

The right rhythm: humans specify what needs to be tested (which behaviors, which boundaries). AI generates how to test (the scaffolding, the assertions, the edge cases). You then read the tests carefully — generated tests that are wrong are worse than no tests, because they create false confidence.

Eval-driven development for AI features

If you're shipping AI features (which you are), evals are not optional. They are the regression test for prompts, contexts, models, and harness changes. Without them, "this feels better" is your only measure, and that's not engineering.

Minimum viable eval setup

  1. 20–100 representative inputs (real production data, anonymized)
  2. For each input: expected output OR a graded rubric (correct facts, correct format, no hallucination)
  3. Auto-grader: another LLM call comparing actual vs expected, returning pass/fail + reason. Or exact-match for structured outputs.
  4. Runner that produces pass-rate + drift summary on every prompt/model/harness change
typescript · minimal eval harness 
import { z } from 'zod';

interface EvalCase {
  id: string;
  input: unknown;
  expected: {
    schema?: z.ZodSchema;
    mustContain?: string[];
    mustNotContain?: string[];
  };
  rubric?: string;  // for LLM grader on subjective criteria
  tags?: string[];   // for slicing results: "edge_case", "vietnamese", etc.
}

async function runEvals(cases: EvalCase[], system: PromptVersion) {
  const results = [];
  for (const c of cases) {
    const output = await runPrompt(system, c.input);
    
    const checks = {
      schemaOk: c.expected.schema 
        ? c.expected.schema.safeParse(output).success 
        : true,
      containsAll: c.expected.mustContain?.every(s => output.includes(s)) ?? true,
      avoidsAll: c.expected.mustNotContain?.every(s => !output.includes(s)) ?? true,
      rubricPass: c.rubric ? await llmGrade(c.rubric, c.input, output) : true
    };
    
    results.push({
      id: c.id, tags: c.tags ?? [],
      pass: Object.values(checks).every(Boolean),
      checks, output
    });
  }
  
  const passRate = results.filter(r => r.pass).length / results.length;
  const byTag = groupByTag(results);
  
  return { passRate, byTag, results, version: system.version };
}

// Wire into CI: every prompt change runs evals, regression alerts.
// Track pass-rate over time as a first-class quality metric.

Anti-patterns to call out for the team

  • "Just one quick prompt tweak" — without evals, you have no idea if it improves or regresses. Always run evals after prompt changes, even small ones.
  • AI review as rubber stamp — if the team treats AI approval as "good enough to merge," AI's mistakes go straight to prod. Human review is non-negotiable.
  • Generated tests without reading them — tests that pass for the wrong reason are worse than no tests.
  • Same model reviews itself — fresh context (subagent) or different tool entirely. Self-review is biased.
  • Skipping evals because "we don't have data yet" — start with 10 hand-crafted cases. Grow over time. Some evals beat no evals.

  3. Identify one AI feature in your codebase. Build a minimum eval suite (10 hand-crafted cases). Run it on current prompt. Record baseline pass rate.
  4. Make one prompt improvement. Run evals. See if pass rate moved (and which cases changed status).

Deploy & Operate · AI in Production

Shipping AI features changes how you do CI/CD, how you write runbooks, and how you handle incidents. The agents are now part of your production system — and a force multiplier for ops work.

CI/CD with AI in the loop

AI's role in CI/CD splits into two categories:

  1. AI helping CI — generating workflow files, troubleshooting failed builds, writing migration scripts
  2. AI as part of CI — automated review on PRs, eval-suite runs as gates, automated triage

Practical CI integrations worth setting up

  • Eval gate — every PR touching prompts/AI features runs the eval suite. Fail merge if pass rate drops below threshold.
  • AI review comment — Codex Cloud or a Claude Code GitHub Action posts a review comment on every PR. Informational, not blocking.
  • Auto-generated changelog — agent reads merged PRs, drafts changelog entries by category.
  • Migration safety check — for any PR touching DB migrations, an agent runs the migration in a test DB with production-shaped sample data, reports issues.

Runbooks · the perfect AI generation target

Runbooks are structured procedural docs — exactly the shape AI generates well. The pattern:

  1. For each operational scenario (deploy, rollback, common incidents, scaling events), draft the runbook with AI from your existing infra/config files.
  2. Test the runbook by following it step-by-step in a non-prod environment. Note where it's wrong or vague.
  3. Have the AI revise based on your notes.
  4. Store runbooks in docs/runbooks/. They become high-signal context for incident-response prompts later.
prompt · generate runbook from infra

- /server/db/index.ts (connection setup)
- /docs/architecture.md
- /docs/runbooks/_template.md (use this format)

The runbook must include:
1. How to detect (which dashboard panels, log patterns, alert names)
2. Immediate triage (what to check first, in order)
3. Mitigation (commands to run, in order, with expected output)
4. Root cause analysis (what to look at to understand WHY)
5. Long-term fix (architectural changes if pattern repeats)
6. Post-incident: what to update (this runbook, monitoring, code)

Be specific. Use exact commands. Reference real metric names from

Incident response · agents as accelerators

During incidents, an AI agent's value is speed. The right setup:

  1. Pre-prepared incident response skill — a Claude Code skill or Codex skill that knows your stack, your runbooks, your dashboards.
  2. Read-only first — incident agents start in read-only mode. They investigate, summarize, propose. They do not act without explicit human approval.
  3. Connected to logs/metrics — via MCP server or tool integration. Agent reads Datadog/Sentry/CloudWatch, correlates timing.
  4. Generates the timeline — as the incident unfolds, the agent maintains a structured timeline that becomes the incident retro doc.

The discipline: AI never acts on production during an incident without human approval. The model can hallucinate "fixes" that make things worse. Read-only investigation and proposal generation is the safe productive mode.

Debugging production with AI

Production debugging benefits hugely from agents — but the harness around them needs care because the stakes are high.

Setup that works

  • Read-only agent identity — the production-debug agent uses credentials that can read logs, query DBs, hit health endpoints, but cannot write or deploy.
  • Bounded workspace — Codex Cloud sandbox or Claude Code in a fresh worktree on a debug branch.
  • Structured incident brief — the agent gets a brief: "Reports of slow checkout in last 30min. Affected tenant: tenant-XYZ. Latest deploy: deploy-abc123. Suspected: DB or API."
  • Multi-tool investigation — agent queries metrics, reads logs, correlates with recent deploys, generates a hypothesis with evidence.

What you get back

A structured report: probable root cause, evidence, suggested mitigation, suggested permanent fix. You decide what to act on.

Observability for AI features themselves

Your AI features need their own observability layer, distinct from app metrics:

  • Per-call telemetry — model, prompt version, input tokens, output tokens, cache hit ratio, latency, cost
  • Per-feature aggregates — daily cost, p95 latency, error rate, eval pass rate
  • Per-user cost — cost per user per day for billing reconciliation and abuse detection
  • Sampled full-trace logs — for n% of calls, full input/output retained (PII redacted) for debugging
  • Drift detection — periodic eval runs against production samples; alert on regression

Tools that exist for this: Langfuse, Helicone, Braintrust, LangSmith. Even structured JSON logs to your log aggregator beats nothing.

Cost discipline · the production reality

AI features have a unique cost characteristic: cost can grow non-linearly with abuse or unexpected user behavior. Production patterns:

  • Per-user daily quota — hard cap. Excess returns "limit reached" rather than just costing more.
  • Feature-level kill switch — if cost spikes 5x baseline within an hour, auto-disable the feature, alert on-call.
  • Cheaper-model fallback — when premium model is throttled, fall back to a cheaper one for the rest of the request rather than failing.
  • Streaming + early termination — for long generations, allow user to stop. Bills only what was generated.
  • Cache aggressively — both prompt caching and response caching for repeated identical queries.

Team Adoption Playbook

How to roll this out across a team without it dissolving into "everyone has their own setup." The rollout phases, the standards, the cost story, the training path, and when not to use AI at all.

Why team adoption fails (and how to avoid it)

Common failure modes when a team tries to adopt AI tooling:

  • Tool fragmentation — three engineers use Cursor, two use Claude Code, one uses Codex CLI, no shared conventions. Each builds their own habits, none reusable.
  • "Senior uses, juniors don't" — the senior internalizes the patterns, the juniors don't get exposure, the gap widens, juniors can't review senior's AI-assisted output.
  • Quality regression that goes unnoticed — code gets shipped faster but bug rate creeps up. Without metrics, no one notices.
  • Cost surprise — month one is fine, month three is a $20k bill no one expected.
  • Skill atrophy — engineers stop reading code carefully, stop practicing fundamentals. Six months in, the team is faster but weaker.

The playbook below is structured to avoid each.

Phase 1 · Establish standards (week 1–2)

Before scaling, lock down the shared substrate:

  • Pick the toolset — Claude Code + Codex (or pick which surfaces of each). Document the choice. New engineers install both.
  • Canonical conventions doc — docs/CONVENTIONS.md. Both CLAUDE.md and AGENTS.md point to it.
  • Shared skills repo — shared/skills/, symlinked into both .claude/skills/ and .agents/skills/. Start with 3–5 high-value skills (db-migration, code-review, deploy, etc.).
  • Shared subagent definitions — checked into the repo, available to everyone.
  • Pre-commit hooks — at minimum: dangerous-command blocker, post-edit formatter, AI review (informational).
  • Eval suites for any AI features in production — even small. Wire into CI.

Output of this phase: a developer can clone the repo, run one setup script, and have a working AI-augmented dev environment in 15 minutes.

Phase 2 · Pilot team learns by doing (week 3–6)

Pick 2–3 engineers who will go deep first. They run the workflows from Modules 8–12 on real work. Their job:

  • Use the tooling daily on their actual tickets
  • Maintain a shared "lessons learned" doc — what works, what doesn't, what's surprisingly hard
  • Refine the shared skills/subagents/hooks based on real usage
  • Track per-task time and cost (rough is fine)
  • End of phase: present findings to broader team, lead the rollout

Pilot team should include at least one skeptic. Universal enthusiasm is suspicious — you want someone actively looking for the failure modes so you find them now, not in production.

Phase 3 · Broader rollout with paired learning (week 7+)

The remaining team adopts. Structure that prevents fragmentation:

  • Pairing — every engineer paired with someone from the pilot for the first two weeks. Watching is faster than reading docs.
  • Weekly "AI clinic" — 30 minutes, one engineer demos a workflow that worked or one that broke. Group discusses.
  • Shared incident log — when AI ships a bug, document it. What didn't the harness catch? What review missed it? Patterns emerge.
  • Shared skill PRs — anyone can propose a new skill via PR to the shared skills repo. Reviewed like code.

Cost tracking that matters

Three metrics to track per dev/per project, weekly:

  • Inference cost — actual $ spent on Claude/OpenAI APIs. Per dev (for awareness) and per feature (for cost-of-feature analysis).
  • Time-to-merge — from PR open to merge, by complexity bucket. Should trend down with adoption; if not, dig into why.
  • Bug rate post-merge — bugs filed per merged PR. If this trends UP after AI adoption, you're shipping faster but worse — pull back on review automation.

Surface these to the team. Cost transparency prevents the "month-three surprise" and makes engineers cost-aware in their tool choices.

When NOT to use AI

This is the hardest discipline to maintain in an AI-enthusiastic team. Cases where reaching for AI is the wrong call:

  • Areas you don't yet understand yourself — if you can't review AI's output critically, you shouldn't be the one shipping AI's output. Learn the area first.
  • Security-critical paths under time pressure — AI catches a lot but is not a security engineer. High-stakes auth/crypto/payment logic deserves human design and human review primarily, AI as backup.
  • Code as learning exercise — when a junior is meant to learn by doing, AI shortcuts the learning. Use AI for review/feedback, not generation.
  • Genuinely novel problems — if you're inventing something the model has no priors for, AI generates plausible-looking but wrong solutions. Better to think first, then use AI to refine.
  • One-line trivial changes — the overhead of prompting beats just typing it.
  • When you're tired — tired humans don't catch AI mistakes. The risk profile inverts.

The team that uses AI most effectively also has the strongest "no AI here" instinct. It's a meta-skill: knowing when the tool helps vs hurts. Cultivate it explicitly. Talk about it. Make it part of code review feedback when relevant.

Skill atrophy prevention

This is the silent risk. After 6 months of heavy AI use, engineers can lose:

  • Reflexive code reading skill (because AI summarizes for them)
  • Mental model of unfamiliar libraries (because AI bridges the gap)
  • Patience for difficult debugging (because AI does the first pass)
  • Foundational knowledge in areas they delegate to AI

Counter-practices to bake in:

  • "AI off" days or sessions — pick one task per week to do without AI. Notice what's harder, what you've forgotten.
  • Read AI-generated code carefully, not just diff-skim — mandatory practice. If you can't explain a function the AI wrote, you can't ship it.
  • Periodic deep-dives into "why does this work" — for code AI wrote, pick a piece occasionally and trace through it line by line. Maintains the muscle.
  • Code reading book club — read open-source code as a team monthly. Restores reading-without-AI as a normal skill.

The team-level success measure

You'll know the rollout worked when these things are true:

  1. A new engineer is productive in days, not weeks (good CLAUDE.md + AGENTS.md + skills do this)
  2. Code reviews focus on the interesting calls, not the mechanical issues (AI handled those)
  3. Time-to-merge is meaningfully down for routine work; quality stayed flat or improved
  4. Cost per feature is predictable and trending down as the team gets better at it
  5. Engineers can articulate when AI is the wrong tool, not just when it's the right one
  6. The shared skills/subagents/hooks repo has more contributors than it does maintainers

The closing principle

The harness is the product. The model is interchangeable. Your team's leverage compounds with the harness you build — not with the model you use.

Models will keep getting better. The teams that out-pace will not be the ones who switch models fastest — they'll be the ones whose harness is best, whose skills library is richest, whose conventions are tightest, whose review discipline is most evolved. Spend most of your engineering effort there.
